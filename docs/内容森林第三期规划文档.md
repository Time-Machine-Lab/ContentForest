# 内容森林 - 第三期规划文档

## 1. 本期目标与定位

### 1.1 本期愿景陈述

本期结束后，用户进入某个种子工作区时，可以选择先完成一次轻量的工作区初始化：系统通过少量选择题理解用户想法，边问边调研，自动准备种子主简报、首轮营养和首轮生长策略，让用户不需要先学习基因、营养库和枝化管线，也能更快获得第一批达到最低发布标准的候选果实。

### 1.2 与项目总方向的对齐

本期直接服务于内容森林“灵感种子 → 枝化生长 → 物竞天择 → 数据反馈 → 迭代进化”的核心闭环，重点补齐从“创建种子”到“首次可发布候选果实”之间的冷启动体验。

第二期侧重建设种子主简报、营养活化、枝化生长管线、基因经验复用和多媒体果实等能力；第三期则把这些能力收拢到用户进入工作区前后的初始化体验中，让系统主动完成上下文准备，而不是让用户在首次生成失败后再被迫理解一堆概念。

### 1.3 本期核心要解决的问题

- 问题 1：用户首次进入工作区后直接生成，容易因为种子信息不足、平台语境缺失、营养不足而得到不满意结果。  
  期望：用户在首次生成前可以选择工作区初始化，由系统主动补齐首轮生成所需的最低上下文。

- 问题 2：当首次生成不佳时，用户往往不知道该补什么营养、为什么淘汰、该选择哪些基因，容易进入迭代迷茫。  
  期望：系统用少量引导问题和自动调研帮助用户确定初始方向，降低用户理解基因、营养和管线的门槛。

- 问题 3：用户长期卡在生成、淘汰、再生成的内循环中，迟迟无法进入发布验证阶段。  
  期望：首轮初始化要以“尽快产出最低可发布候选”为目标，而不是追求一次性完美理解用户全部想法。

## 2. 功能模块规划

### 2.1 工作区初始化入口

**模块一句话价值**：让用户在进入工作区前选择是否先让系统帮种子完成一次生长准备。

**用户视角描述**：

背景与目的（Why）：新用户创建种子后，最自然的期待是马上看到种子生长出内容。如果直接进入空工作区并要求用户理解生成器、营养、基因和搜索策略，首次体验会变重。工作区初始化入口的目的是在用户开始前给出一个低压力选择：要么直接进入工作区，要么先让系统帮忙准备首轮上下文。

做什么（What）：用户从种子卡片进入工作区时，系统询问是否进行工作区初始化。用户选择“开始初始化”后进入加载与问答阶段；用户选择“跳过”后直接进入原工作区流程，并保留后续手动发起初始化的入口。

预期效果（How it looks）：用户点击种子卡片后，先看到一个轻量确认界面，文案强调“约 1-3 分钟，帮助提升首轮生成质量”。界面提供两个主要动作：“开始初始化”和“跳过，直接进入工作区”。该选择不制造压力，用户可以随时跳过。

**本期不包含**：

- 不强制所有用户初始化
- 不阻塞原有工作区进入流程
- 不要求用户先理解基因和营养
- 不承诺一次初始化得到完美内容

#### 2.1.1 背景与目的

工作区初始化入口用于降低冷启动摩擦，让用户在首次生成前获得系统辅助，而不是在首次失败后才开始补救。

#### 2.1.2 核心功能描述

1. 用户点击种子卡片进入工作区。
2. 用户看到是否进行工作区初始化的提示。
3. 用户选择开始初始化，进入加载与问答阶段。
4. 用户选择跳过，直接进入原工作区。
5. 用户在工作区内仍可重新发起初始化。

#### 2.1.3 预期交付形态

入口呈现为进入工作区前的一张轻量确认面板，说明初始化会帮助系统理解种子、补充首轮营养、准备简报和首轮生长策略。面板只提供明确选择，不展示复杂设置；跳过后用户进入原工作区，不损失任何功能。

#### 2.1.4 本期不做（模块级边界）

- 不做强制 onboarding
- 不做复杂偏好问卷
- 不做账号级长期画像
- 不替用户自动发布内容

### 2.2 引导式种子问答

**模块一句话价值**：用少量高信息量问题理解用户意图，避免用户一开始就写长表单或解释专业概念。

**用户视角描述**：

背景与目的（Why）：用户往往知道自己“想做什么”，但未必知道应该补充哪些信息才能让内容生成更好。如果系统一次性抛出完整问卷，用户会觉得负担很重；如果完全不问，系统又容易误解方向。引导式问答的目标是只问当前最有价值的问题，用较少回答换取更明确的首轮创作方向。

做什么（What）：系统读取种子后，按信息增益选择下一个问题。问题只使用单选或多选题，并始终保留“我不知道”和“输入其他”。每次可以只问一个问题，根据用户回答决定下一问；问题数量设置上限，默认最多 5 个。用户也可以提前结束问答，让系统基于已有答案继续初始化。

预期效果（How it looks）：用户看到的不是一张长问卷，而是一段对话式引导。比如种子是“我想做一个搞笑博主”，系统先问“想做什么方向的搞笑博主？”并给出情景搞笑、段子手、梗图/热梗、我不知道、输入其他。用户回答后，系统再决定是否询问平台、内容身份、表达风格或素材情况。

**本期不包含**：

- 不支持开放式长篇问答为主
- 不一次性展示全部问题
- 不要求用户必须回答所有问题
- 不根据问答生成永久用户画像

#### 2.2.1 背景与目的

引导式问答用于在低负担下缩小方向不确定性，让系统更接近用户想法，并判断首轮需要补哪些上下文。

#### 2.2.2 核心功能描述

1. 用户进入初始化问答阶段。
2. 用户逐题回答单选或多选问题。
3. 用户可选择“我不知道”。
4. 用户可输入其他答案。
5. 用户可提前结束问答。
6. 用户看到系统正在根据回答更新理解。

#### 2.2.3 预期交付形态

问答区以聊天式卡片呈现，每次只突出一个问题，选项按钮清晰可点。问题旁边可以展示“第 2 / 最多 5 问”的轻量进度提示。用户回答后，界面显示系统正在理解和准备资料，而不是让用户等待空白加载。

#### 2.2.4 本期不做（模块级边界）

- 不做复杂心理测评式问卷
- 不做几十题长流程
- 不强制用户精确表达定位
- 不把用户回答当绝对事实

### 2.3 边问边调研

**模块一句话价值**：让初始化阶段不只是问用户，而是同步为种子准备首轮营养和简报方向。

**用户视角描述**：

背景与目的（Why）：用户回答问题后，如果系统只是继续问，仍然会让人觉得耗时。更好的体验是问答和调研并行：用户提供方向，系统立刻开始补充平台语境、爆款样例、用户语言和内容路径。这样初始化阶段就像工作区正在“加载生长条件”，而不是单纯填问卷。

做什么（What）：用户每回答一部分问题，系统就可以开始准备对应的调研任务。调研目标包括生成种子主简报、发现种子缺失营养、搜索相关爆款样例、整理平台表达规律、生成首批临时营养卡片，并准备首轮枝化生长建议。调研结果不会自动写入正式营养库，用户可以在初始化过程中采纳、忽略，或在进入工作区后继续沉淀。

预期效果（How it looks）：初始化界面分为问答区和准备进度区。用户回答问题时，旁边能看到系统正在准备“平台方向”“爆款样例”“首轮营养”“种子主简报”等项目。某些项目完成后会以卡片形式出现，用户可以展开查看，也可以先不处理，直接进入首轮生成。

**本期不包含**：

- 不做脚本化批量采集
- 不自动写入正式营养库
- 不展示复杂来源可信度系统
- 不保证覆盖平台全部规律

#### 2.3.1 背景与目的

边问边调研用于把用户回答立即转化为首轮上下文，让初始化阶段产生可见成果，而不是只收集信息。

#### 2.3.2 核心功能描述

1. 用户回答问题后看到准备进度。
2. 用户看到系统生成种子主简报草案。
3. 用户看到首批临时营养卡片。
4. 用户看到相关爆款样例摘要。
5. 用户选择采纳或忽略初始化结果。
6. 用户进入工作区时带入已采纳内容。

#### 2.3.3 预期交付形态

初始化阶段呈现为“问答 + 准备进度”的双区界面。左侧或上方是当前问题，右侧或下方展示初始化项目进度，例如“理解种子方向”“搜索平台样例”“整理首轮营养”“生成主简报”。完成后的结果以卡片方式展示，用户可以一键采纳。

#### 2.3.4 本期不做（模块级边界）

- 不做独立平台情报后台
- 不做长期研究项目管理
- 不自动沉淀所有调研结果
- 不替用户决定最终平台

### 2.4 首轮生长预配置

**模块一句话价值**：把问答和调研结果自动转化为悬浮输入框中的首轮生长配置。

**用户视角描述**：

背景与目的（Why）：初始化阶段已经创建或准备了种子主简报、种子专属营养库、相关营养卡片和首轮方向判断。如果再额外引入“启动包”页面，会增加一个新的抽象层。首轮生长预配置的目的，是把初始化结果直接落实到工作区悬浮输入框中，让用户进入工作区时看到系统已经帮自己填好了首轮生长配置。

做什么（What）：初始化完成后，系统直接进入工作区，并自动展开悬浮输入框。输入框中预填首轮生长配置，包括推荐生成器、果实数量、引用营养内容、搜索方向、探索强度或突变倾向、输入内容和是否生成保守可发布候选。用户可以直接点击开始生长，也可以在输入框内微调配置。

预期效果（How it looks）：初始化完成后，用户进入工作区画布，悬浮输入框自动打开并显示“已为首轮生长准备好”。输入内容、生成器、果实数量、引用营养和搜索方向都已预填。用户不需要理解“启动包”或额外确认页面，只需要检查是否符合预期，然后点击“开始生长”。

**本期不包含**：

- 不暴露复杂管线参数
- 不要求用户手动选择全部基因
- 不生成永久不可修改配置
- 不承诺所有候选都可发布

#### 2.4.1 背景与目的

首轮生长预配置用于把系统准备好的上下文直接落到用户真正发起生长的操作界面中，减少中间页面和抽象概念。它强调“系统已经帮你填好了”，而不是让用户再理解一组新的准备结果。

#### 2.4.2 核心功能描述

1. 初始化完成后，系统进入工作区。
2. 系统自动展开悬浮输入框。
3. 系统预填首轮输入内容。
4. 系统预选推荐生成器。
5. 系统预设果实数量、搜索方向和探索强度。
6. 系统自动引用初始化阶段准备的临时营养或常驻营养。
7. 用户检查或微调配置。
8. 用户点击开始首轮生长。

#### 2.4.3 预期交付形态

悬浮输入框是首轮预配置的主要承载界面。输入框上方或内部展示一条轻量提示：“初始化完成，已为首轮生长准备好配置”。引用营养、生成器和搜索方向以可见但不喧宾夺主的方式展示，用户可以删除、替换或调整。更详细的种子主简报和营养内容仍然在对应面板中查看，不在输入框里堆满信息。

#### 2.4.4 本期不做（模块级边界）

- 不做复杂配置向导
- 不新增启动包页面或启动包实体
- 不展示全部底层上下文
- 不自动替用户确认正式营养
- 不自动替用户确认正式基因

### 2.5 首轮最低可发布候选

**模块一句话价值**：让第一次枝化生长不只追求发散，而是至少产出一条接近可发布标准的保守候选。

**用户视角描述**：

背景与目的（Why）：内容森林的最终验证来自发布后的真实反馈。如果用户长期停留在生成和淘汰中，就无法进入市场验证。首轮最低可发布候选的目标不是保证爆款，而是让用户更快拿到一条“可以认真考虑发布”的内容，先进入真实反馈闭环。

做什么（What）：工作区初始化后的首轮生长同时包含探索候选和保守候选。探索候选用于尝试不同方向，保守候选用于满足基本质量线，例如表达清晰、广告感可控、平台语感基本正确、结构完整、没有明显违背用户约束。用户可以选择发布、继续生长或淘汰。

预期效果（How it looks）：首轮生成结束后，内容树上出现几颗方向不同的果实，其中至少一颗被标记为“保守可发布候选”。用户点击后能看到为什么它更适合先发布验证，例如“产品露出较晚”“内容更像真实分享”“结构完整”。这不是系统替用户判断成功，只是帮助用户更快进入发布阶段。

**本期不包含**：

- 不承诺候选一定爆款
- 不自动发布到外部平台
- 不替用户做最终选择
- 不用 AI 评估代替真实数据

#### 2.5.1 背景与目的

首轮最低可发布候选用于缩短从创建种子到发布验证的距离，避免用户长期卡在生成内循环。

#### 2.5.2 核心功能描述

1. 用户从预填好的悬浮输入框发起首轮生长。
2. 用户看到探索候选和保守候选。
3. 用户打开保守可发布候选查看理由。
4. 用户选择发布、继续生长或淘汰。
5. 用户保留对所有果实的最终判断权。

#### 2.5.3 预期交付形态

首轮果实在内容树上以普通节点展示，但“保守可发布候选”可以带有轻量标识。果实详情中展示可发布理由，帮助用户判断是否进入发布验证。该标识不影响物竞天择状态，也不代表系统保证效果。

#### 2.5.4 本期不做（模块级边界）

- 不做自动质量评分系统
- 不做平台效果预测器
- 不自动发布内容
- 不隐藏其他探索候选

## 3. 本期整体交付形态

第三期结束后，用户从种子库点击某个种子进入工作区时，会先看到是否进行工作区初始化的轻量提示。选择跳过时，用户进入原有工作区；选择开始初始化时，用户进入一个加载与问答并行的初始化界面。

初始化过程中，系统围绕当前种子逐步提出少量单选或多选问题，每题都保留“我不知道”和“输入其他”。用户回答后，界面同步展示系统正在准备的内容：种子理解、平台方向、爆款样例、首轮营养、主简报草案和首轮探索建议。用户不需要理解营养库、基因库或枝化管线的底层概念，只需要做少量选择。

初始化完成后，用户直接进入工作区，悬浮输入框自动展开，并已经预填首轮生长配置。配置包括输入内容、推荐生成器、果实数量、引用营养、搜索方向、探索强度和是否生成保守可发布候选。用户可以直接点击“开始生长”，也可以微调后再开始。首轮生长完成后，内容树出现多颗果实，其中至少包含一颗“保守可发布候选”，帮助用户更快进入发布验证阶段。

## 4. 本期成功标准

| 验收维度 | 成功条件 | 验证方式 |
| :--- | :--- | :--- |
| 初始化入口清晰 | 用户从种子卡片进入工作区时，可以选择初始化或跳过 | 产品走查 |
| 原流程不受影响 | 用户跳过初始化后，可以进入原工作区并正常发起枝化生长 | 工作区流程测试 |
| 问答负担可控 | 初始化问答默认不超过 5 个问题，且每题包含“我不知道”和“输入其他” | 产品走查 |
| 问答可动态推进 | 用户回答后，下一题能基于已有回答调整方向，而不是固定长问卷 | 样例走查 |
| 调研过程可感知 | 初始化阶段能展示种子理解、营养准备、样例搜索或简报生成等进度 | 交互走查 |
| 首轮预配置可用 | 初始化完成后，悬浮输入框能自动预填输入内容、生成器、果实数量、引用营养和搜索方向 | 产品走查 |
| 一键进入首轮生长 | 用户可以从预填好的悬浮输入框直接发起首轮枝化生长 | 工作区流程测试 |
| 首轮候选更接近发布 | 首轮结果中至少有一颗被标记为保守可发布候选，并说明理由 | 样例评审 |
| 概念暴露减少 | 新用户完成初始化和首轮生成时，不需要主动理解基因、营养库和管线参数 | 可用性测试 |
| 发布路径缩短 | 用户从创建种子到看到可发布候选的路径不需要经历手动汲取基因和手动补营养 | 产品走查 |

## 5. 参考约束与衔接说明

### 5.1 上期回顾

第二期规划已经明确了种子主简报、营养库活化、枝化生长管线、基因经验复用、多媒体果实和生成路径可视化等能力。这些能力为第三期工作区初始化提供了基础：第三期不重新发明这些模块，而是把它们组织成用户进入工作区前后的低门槛启动体验。

### 5.2 下期预告

后续需要继续讨论和处理“内容爆炸”问题，重点包括：

- 基因经验过多时如何推荐、折叠和自动选择。
- 营养卡片过多时如何合并、归档和保持新鲜。
- 用户如何理解当前真正影响生成的少量关键上下文。
- 系统如何避免把每次失败都转化为越来越多的规则负担。

### 5.3 对齐项目总方向

本期规划直接对齐内容森林顶层概念中的以下方向：

- “灵感为种子” → 对应本章 2.1 工作区初始化入口、2.2 引导式种子问答。
- “AI + 经验 内容生成” → 对应本章 2.3 边问边调研、2.4 首轮生长预配置。
- “自我迭代 + 进化” → 对应本章 2.5 首轮最低可发布候选。
- “物竞天择” → 对应本章第 4 节中对首轮候选、发布路径和用户判断权的验收要求。
