# 内容森林 - 第三期规划文档

## 1. 本期目标与定位

### 1.1 本期愿景陈述

本期结束后，用户进入某个种子工作区时，可以选择先完成一次轻量的工作区初始化：系统通过少量选择题理解用户想法，边问边调研，自动准备种子主简报、首轮营养和首轮生长策略，让用户不需要先学习基因、营养库和枝化管线，也能更快获得第一批达到最低发布标准的候选果实。

### 1.2 与项目总方向的对齐

本期直接服务于内容森林“灵感种子 → 枝化生长 → 物竞天择 → 数据反馈 → 迭代进化”的核心闭环，重点补齐从“创建种子”到“首次生成出可考虑发布验证的完整内容”之间的冷启动体验。

第二期侧重建设种子主简报、营养活化、枝化生长管线、基因经验复用和多媒体果实等能力；第三期则把这些能力收拢到用户进入工作区前后的初始化体验中，让系统主动完成上下文准备，而不是让用户在首次生成失败后再被迫理解一堆概念。

### 1.3 本期核心要解决的问题

- 问题 1：用户首次进入工作区后直接生成，容易因为种子信息不足、平台语境缺失、营养不足而得到不满意结果。  
  期望：用户在首次生成前可以选择工作区初始化，由系统主动补齐首轮生成所需的最低上下文。

- 问题 2：当首次生成不佳时，用户往往不知道该补什么营养、为什么淘汰、该选择哪些基因，容易进入迭代迷茫。  
  期望：系统用少量引导问题和自动调研帮助用户确定初始方向，降低用户理解基因、营养和管线的门槛。

- 问题 3：用户长期卡在生成、淘汰、再生成的内循环中，迟迟无法进入发布验证阶段。  
  期望：首轮初始化要把“可发布性目标”写入首轮生长输入或策略中，而不是追求一次性完美理解用户全部想法。

## 2. 功能模块规划

### 2.1 工作区初始化入口

**模块一句话价值**：让用户在进入工作区前选择是否先让系统帮种子完成一次生长准备。

**用户视角描述**：

背景与目的（Why）：新用户创建种子后，最自然的期待是马上看到种子生长出内容。如果直接进入空工作区并要求用户理解生成器、营养、基因和搜索策略，首次体验会变重。工作区初始化入口的目的是在用户开始前给出一个低压力选择：要么直接进入工作区，要么先让系统帮忙准备首轮上下文。

做什么（What）：用户从种子卡片进入工作区时，系统询问是否进行工作区初始化。用户选择“开始初始化”后进入加载与问答阶段；用户选择“跳过”后直接进入原工作区流程，并保留后续手动发起初始化的入口。

预期效果（How it looks）：用户点击种子卡片后，先看到一个轻量确认界面，文案强调“约 1-3 分钟，帮助提升首轮生成质量”。界面提供两个主要动作：“开始初始化”和“跳过，直接进入工作区”。该选择不制造压力，用户可以随时跳过。

**本期不包含**：

- 不强制所有用户初始化
- 不阻塞原有工作区进入流程
- 不要求用户先理解基因和营养
- 不承诺一次初始化得到完美内容

#### 2.1.1 背景与目的

工作区初始化入口用于降低冷启动摩擦，让用户在首次生成前获得系统辅助，而不是在首次失败后才开始补救。

#### 2.1.2 核心功能描述

1. 用户点击种子卡片进入工作区。
2. 用户看到是否进行工作区初始化的提示。
3. 用户选择开始初始化，进入加载与问答阶段。
4. 用户选择跳过，直接进入原工作区。
5. 用户在工作区内仍可重新发起初始化。

#### 2.1.3 预期交付形态

入口呈现为进入工作区前的一张轻量确认面板，说明初始化会帮助系统理解种子、补充首轮营养、准备简报和首轮生长策略。面板只提供明确选择，不展示复杂设置；跳过后用户进入原工作区，不损失任何功能。

#### 2.1.4 本期不做（模块级边界）

- 不做强制 onboarding
- 不做复杂偏好问卷
- 不做账号级长期画像
- 不替用户自动发布内容

### 2.2 引导式种子问答

**模块一句话价值**：用少量高信息量问题理解用户意图，避免用户一开始就写长表单或解释专业概念。

**用户视角描述**：

背景与目的（Why）：用户往往知道自己“想做什么”，但未必知道应该补充哪些信息才能让内容生成更好。如果系统一次性抛出完整问卷，用户会觉得负担很重；如果完全不问，系统又容易误解方向。引导式问答的目标是只问当前最有价值的问题，用较少回答换取更明确的首轮创作方向。

做什么（What）：系统读取种子后，按信息增益选择下一个问题。问题只使用单选或多选题，并始终保留“我不知道”和“输入其他”。每次可以只问一个问题，根据用户回答决定下一问；问题数量设置上限，默认最多 8 个。用户也可以提前结束问答，让系统基于已有答案继续初始化。第一版不支持返回修改上一题，避免问答状态变复杂。

预期效果（How it looks）：用户看到的不是一张长问卷，而是一段对话式引导。比如种子是“我想做一个搞笑博主”，系统先问“想做什么方向的搞笑博主？”并给出情景搞笑、段子手、梗图/热梗、我不知道、输入其他。用户回答后，系统再决定是否询问平台、内容身份、表达风格或素材情况。

**本期不包含**：

- 不支持开放式长篇问答为主
- 不一次性展示全部问题
- 不要求用户必须回答所有问题
- 不根据问答生成永久用户画像

#### 2.2.1 背景与目的

引导式问答用于在低负担下缩小方向不确定性，让系统更接近用户想法，并判断首轮需要补哪些上下文。

#### 2.2.2 核心功能描述

1. 用户进入初始化问答阶段。
2. 用户逐题回答单选或多选问题。
3. 用户可选择“我不知道”。
4. 用户可输入其他答案。
5. 用户可提前结束问答。
6. 用户看到系统正在根据回答更新理解。

#### 2.2.3 预期交付形态

问答区以聊天式卡片呈现，每次只突出一个问题，选项按钮清晰可点。问题旁边可以展示“第 2 / 最多 8 问”的轻量进度提示。用户回答后，界面显示系统正在理解和准备资料，而不是让用户等待空白加载。

#### 2.2.4 本期不做（模块级边界）

- 不做复杂心理测评式问卷
- 不做几十题长流程
- 不支持返回修改上一题
- 不强制用户精确表达定位
- 不把用户回答当绝对事实

### 2.3 边问边调研

**模块一句话价值**：让初始化阶段不只是问用户，而是同步为种子准备首轮营养和简报方向。

**用户视角描述**：

背景与目的（Why）：用户回答问题后，如果系统只是继续问，仍然会让人觉得耗时。更好的体验是问答和调研并行：用户提供方向，系统立刻开始补充平台语境、爆款样例、用户语言和内容路径。这样初始化阶段就像工作区正在“加载生长条件”，而不是单纯填问卷。

做什么（What）：用户每回答一部分问题，系统就可以开始准备对应的调研任务。调研目标包括生成种子主简报、发现种子缺失营养、搜索相关爆款样例、整理平台表达规律、生成首批临时营养卡片，并准备首轮枝化生长建议。调研可以联网，联网失败时降级为基于已有上下文继续初始化。调研结果不会自动写入正式营养库，默认先作为临时营养或初始化结果，用户可以在初始化过程中采纳、忽略，或在进入工作区后继续沉淀。

预期效果（How it looks）：初始化界面分为问答区和准备进度区。用户回答问题时，旁边能看到系统正在准备“平台方向”“爆款样例”“首轮营养”“种子主简报”等项目。界面展示调研进展和结果卡片，不展示完整 Agent 研究日志。某些项目完成后会以卡片形式出现，用户可以展开查看，也可以先不处理，直接进入首轮生成。

**本期不包含**：

- 不做脚本化批量采集
- 不自动写入正式营养库
- 不要求联网成功才能继续初始化
- 不展示完整 Agent 研究过程
- 不展示复杂来源可信度系统
- 不保证覆盖平台全部规律

#### 2.3.1 背景与目的

边问边调研用于把用户回答立即转化为首轮上下文，让初始化阶段产生可见成果，而不是只收集信息。

#### 2.3.2 核心功能描述

1. 用户回答问题后看到准备进度。
2. 用户看到系统生成种子主简报草案。
3. 用户看到首批临时营养卡片。
4. 用户看到相关爆款样例摘要。
5. 用户选择采纳或忽略初始化结果。
6. 用户进入工作区时带入已采纳内容。

#### 2.3.3 预期交付形态

初始化阶段呈现为“问答 + 准备进度”的双区界面。左侧或上方是当前问题，右侧或下方展示初始化项目进度，例如“理解种子方向”“搜索平台样例”“整理首轮营养”“生成主简报”。完成后的结果以卡片方式展示，用户可以一键采纳。

#### 2.3.4 本期不做（模块级边界）

- 不做独立平台情报后台
- 不做长期研究项目管理
- 不自动沉淀所有调研结果
- 不替用户决定最终平台

### 2.4 首轮生长预配置

**模块一句话价值**：把问答和调研结果自动转化为悬浮输入框中的首轮生长配置。

**用户视角描述**：

背景与目的（Why）：初始化阶段已经创建或准备了种子主简报、种子专属营养库、相关营养卡片和首轮方向判断。如果再额外引入“启动包”页面，会增加一个新的抽象层。首轮生长预配置的目的，是把初始化结果直接落实到工作区悬浮输入框中，让用户进入工作区时看到系统已经帮自己填好了首轮生长配置。

做什么（What）：初始化完成后，系统直接进入工作区，并自动展开悬浮输入框。输入框遵循原有设计，只是由初始化结果自动改写或预填其中已有的可选内容和可输入内容。预填内容包括推荐生成器、果实数量、引用营养内容、搜索方向、探索强度或突变倾向、输入内容。首轮生成目标中补充“在探索多个方向的同时，至少生成一条可考虑发布验证的完整内容”，但不新增独立果实类型、独立开关或独立标识。用户可以直接点击开始生长，也可以在输入框内微调配置。

预期效果（How it looks）：初始化完成后，用户进入工作区画布，悬浮输入框自动打开并显示“已为首轮生长准备好”。输入内容、生成器、果实数量、引用营养和搜索方向都已预填。用户不需要理解“启动包”或额外确认页面，只需要检查是否符合预期，然后点击“开始生长”。

**本期不包含**：

- 不暴露复杂管线参数
- 不要求用户手动选择全部基因
- 不生成永久不可修改配置
- 不承诺所有候选都可发布

#### 2.4.1 背景与目的

首轮生长预配置用于把系统准备好的上下文直接落到用户真正发起生长的操作界面中，减少中间页面和抽象概念。它强调“系统已经帮你填好了”，而不是让用户再理解一组新的准备结果。

#### 2.4.2 核心功能描述

1. 初始化完成后，系统进入工作区。
2. 系统自动展开悬浮输入框。
3. 系统预填首轮输入内容。
4. 系统预选推荐生成器。
5. 系统预设果实数量，默认 3 个。
6. 系统预设搜索方向和探索强度，探索强度默认使用已有能力中的“激进”。
7. 系统自动引用初始化阶段准备的临时营养或常驻营养。
8. 系统在输入内容或生长策略中加入首轮可发布性目标，但不新增“保守可发布候选”概念。
9. 用户检查或微调配置。
10. 用户点击开始首轮生长。

#### 2.4.3 预期交付形态

悬浮输入框是首轮预配置的主要承载界面。输入框上方或内部展示一条轻量提示：“初始化完成，已为首轮生长准备好配置”。引用营养、生成器和搜索方向以可见但不喧宾夺主的方式展示，用户可以删除、替换或调整。更详细的种子主简报和营养内容仍然在对应面板中查看，不在输入框里堆满信息。

如果系统无法推荐生成器，悬浮输入框保持需要用户手动选择生成器的状态，用户选择生成器后才能开始生长。预填输入内容必须允许用户编辑。

#### 2.4.4 本期不做（模块级边界）

- 不做复杂配置向导
- 不新增启动包页面或启动包实体
- 不新增首轮最低可发布候选模块
- 不新增可发布候选果实类型或状态
- 不展示全部底层上下文
- 不自动替用户确认正式营养
- 不自动替用户确认正式基因

### 2.5 基因卡片可读性规范

**模块一句话价值**：先解决“每条基因看不懂”的问题，让用户快速理解基因的作用、方向、来源和可信度。

**用户视角描述**：

背景与目的（Why）：随着基因经验持续沉淀，基因数量一定会增长。第三期暂时不引入复杂的基因推荐、自动选择或上下文编排器，但需要先解决一个更直接的问题：用户在基因列表中无法通过标题和摘要快速判断每条基因到底是做什么的、来自哪个果实、是正向还是负向、可信度如何。

做什么（What）：为基因卡片建立轻量展示规范。每条基因需要有独立的正负属性标签、规范化标题、一句话说明、来源果实和证据强度。标题不承担正负语义，正负由独立属性和前端样式表达。来源果实用于帮助用户追溯该基因从哪条内容中提炼而来；证据强度用于帮助用户判断该基因目前只是人工判断，还是已经经过发布反馈或多次验证。

预期效果（How it looks）：用户在基因列表中看到的不再是一堆难以理解的 Agent 摘要，而是结构一致的经验卡片。例如：标题为“方法内容前置降低广告感”，左侧或顶部显示“正向”标签，下方显示“来自果实：AI 工具效率复盘”“证据强度：人工选择”，再用一句话解释该基因为什么有用。

**本期不包含**：

- 不做复杂基因推荐系统
- 不做自动基因选择
- 不做基因合并算法
- 不做基因评分排序系统
- 不解决完整的基因爆炸问题

#### 2.5.1 背景与目的

基因爆炸的完整解决方案会涉及推荐、折叠、合并、归档和上下文编排，范围较大。第三期只做一个低成本但高收益的改进：让单条基因卡片变得可读、可扫读、可追溯。

#### 2.5.2 核心功能描述

1. 基因拥有独立的极性属性，表示正向或负向。
2. 前端根据极性属性展示不同标签、颜色或图标。
3. 基因标题采用统一格式，帮助用户理解其作用。
4. 基因卡片展示一句话说明，解释该基因为什么有用或为什么要规避。
5. 基因卡片展示来源果实，支持用户理解该基因从哪里来。
6. 基因卡片展示证据强度，帮助用户判断可信度。

#### 2.5.3 标题与字段规范

基因标题推荐结构为：

> 表达机制 + 影响对象 + 预期作用

标题要求：

- 建议 8-20 字。
- 不包含“正向”或“负向”。
- 不写成长句。
- 避免抽象空话。
- 尽量说明它影响内容的哪个部分，以及会产生什么效果。

标题示例：

- 方法内容前置降低广告感
- 真实使用场景增强信任
- 具体结果承诺提升点击
- 功能堆叠削弱阅读兴趣
- 产品名称后置减少销售感
- 评论区语言增强平台语感
- 对比结构强化问题感知

基因卡片最小展示字段：

- 极性标签：正向 / 负向。
- 规范标题：表达机制 + 影响对象 + 预期作用。
- 来源果实：展示来源果实标题，并支持跳转或打开详情。
- 证据强度：人工选择 / 人工淘汰 / 发布反馈 / 多次验证。
- 一句话说明：用人能看懂的话解释该基因的作用。

#### 2.5.4 预期交付形态

基因卡片以统一结构展示：

```text
[正向] 方法内容前置降低广告感
来自果实：AI 工具效率复盘
证据强度：人工选择
说明：先提供可复用方法，再自然露出产品，适合降低工具推广内容的销售感。
```

```text
[负向] 功能堆叠削弱阅读兴趣
来自果实：MM Agent 功能介绍
证据强度：人工淘汰
说明：连续罗列功能点会让内容更像产品说明书，用户缺少继续阅读的动机。
```

#### 2.5.5 本期不做（模块级边界）

- 不改变基因汲取主流程
- 不要求用户手动整理全部基因
- 不为基因建立复杂权重模型
- 不让卡片展示承载完整基因正文
- 不把证据强度解释为平台效果评分

## 3. 本期整体交付形态

第三期结束后，用户从种子库点击某个种子进入工作区时，会先看到是否进行工作区初始化的轻量提示。选择跳过时，用户进入原有工作区；选择开始初始化时，用户进入一个加载与问答并行的初始化界面。

初始化过程中，系统围绕当前种子逐步提出少量单选或多选问题，每题都保留“我不知道”和“输入其他”。用户回答后，界面同步展示系统正在准备的内容：种子理解、平台方向、爆款样例、首轮营养、主简报草案和首轮探索建议。用户不需要理解营养库、基因库或枝化管线的底层概念，只需要做少量选择。

初始化完成后，用户直接进入工作区，悬浮输入框自动展开，并已经预填首轮生长配置。配置包括输入内容、推荐生成器、果实数量、引用营养、搜索方向和探索强度。系统会在首轮输入内容或生长策略中加入“可发布性目标”，引导首轮生成在发散探索的同时产出至少一条完整、可考虑发布验证的内容。用户可以直接点击“开始生长”，也可以微调后再开始。

在基因列表或工作区基因相关入口中，基因卡片会按统一规范展示正负标签、规范标题、来源果实、证据强度和一句话说明。第三期不尝试完整解决基因爆炸，只先降低用户理解单条基因的成本。

## 5. 参考约束与衔接说明

### 5.1 上期回顾

第二期规划已经明确了种子主简报、营养库活化、枝化生长管线、基因经验复用、多媒体果实和生成路径可视化等能力。这些能力为第三期工作区初始化提供了基础：第三期不重新发明这些模块，而是把它们组织成用户进入工作区前后的低门槛启动体验。

### 5.2 下期预告

后续仍需要继续讨论和处理完整的“内容爆炸”问题，重点包括：

- 基因经验过多时如何推荐、折叠和自动选择。
- 营养卡片过多时如何合并、归档和保持新鲜。
- 用户如何理解当前真正影响生成的少量关键上下文。
- 系统如何避免把每次失败都转化为越来越多的规则负担。

### 5.3 对齐项目总方向

本期规划直接对齐内容森林顶层概念中的以下方向：

- “灵感为种子” → 对应本章 2.1 工作区初始化入口、2.2 引导式种子问答。
- “AI + 经验 内容生成” → 对应本章 2.3 边问边调研、2.4 首轮生长预配置。
- “自我迭代 + 进化” → 对应本章 2.4 首轮生长预配置、2.5 基因卡片可读性规范。
- “物竞天择” → 对应本章第 4 节中对首轮候选、发布路径、用户判断权和基因卡片可读性的验收要求。
