# 内容森林第一期开发规划文档

## 1. 本期目标与定位

### 1.1 本期愿景陈述

本期结束后，用户可以围绕一个灵感种子，在内容树工作区中完成从创建种子、枝化生长、物竞天择、发布验证、数据回流到下一轮迭代进化的最小闭环。

### 1.2 与项目总方向的对齐

本期直接对齐《内容森林概念设计文档》中定义的第一期核心闭环：灵感种子 → 枝化生长 → 人为物竞天择 → 发布验证 → 数据回流 → 迭代进化。本期不追求全自动发布和平台数据采集，而是先让用户可见、可操作、可验证地跑通内容进化流程。

### 1.3 本期核心要解决的问题

- 问题 1：用户目前可以用 AI 生成内容，但难以围绕同一个灵感持续组织多代内容、生长关系和选择结果。
  期望：用户能在一个种子工作区中看到完整内容树，并能从任意种子或果实继续枝化生长。
- 问题 2：内容反馈通常停留在人工记忆或零散表格中，无法反向影响下一轮生成。
  期望：用户能记录发布验证和数据反馈，并把有效表达特征沉淀到种子级基因库，成为下一轮生长可复用的进化经验。

## 2. 功能模块规划

### 2.1 种子库与灵感种子

**模块一句话价值**：让用户把一个想法、产品、项目或宣传内容沉淀为可持续生长的内容源头。

#### 2.1.1 背景与目的

内容森林的一切生长都从种子开始。该模块让用户不只是写一条 prompt，而是创建一个能承载长期内容演化的灵感对象。种子创建后，系统同步为该种子准备种子级基因库，用于承接后续选择、发布和反馈中沉淀出的进化经验。

#### 2.1.2 核心功能描述

1. 用户在种子库页面查看已创建的种子列表。
2. 用户创建种子并填写想法、产品、项目或宣传内容。
3. 用户为种子补充目标受众、目标平台和表达基调。
4. 用户点击种子进入该种子的工作区。
5. 用户查看种子的基础信息和生长状态。

#### 2.1.3 预期交付形态

种子库呈现为一个清晰的列表或卡片页面，用户可以快速创建新种子，也可以进入已有种子的工作区。种子详情不需要复杂配置，重点展示原始灵感、目标方向、最近生长状态和进入工作区的入口。

#### 2.1.4 本期不做（模块级边界）

- 不做复杂种子模板市场
- 不做多人协作编辑
- 不做种子权限管理
- 不做自动选题推荐

### 2.2 生成器管理

**模块一句话价值**：让用户把外部内容创作方法论上传为可复用的枝化生长能力。

#### 2.2.1 背景与目的

生成器本质上是独立的内容创作 Skill，用来告诉 Agent 如何生成某类平台或场景下更容易传播的内容。本期需要让用户能引入生成器，但不建设完整生态。

#### 2.2.2 核心功能描述

1. 用户在生成器页面查看已上传的生成器。
2. 用户上传一个生成器文件或生成器内容。
3. 用户为生成器填写名称、适用平台和内容类型。
4. 用户在枝化生长前选择要使用的生成器。
5. 用户查看生成器的基础说明和适用范围。

#### 2.2.3 预期交付形态

生成器页面是一个轻量管理页，用户可以上传并管理自己的生成器。每个生成器以卡片形式展示名称、适用平台、内容类型和简介。在种子工作区的生长输入框中，用户可以从已上传生成器中选择一个作为本次生长方法。

#### 2.2.4 本期不做（模块级边界）

- 不做生成器市场
- 不做在线复杂编辑器
- 不做生成器付费分发
- 不做生成器评分排行

### 2.3 营养库

**模块一句话价值**：让用户把爆款案例、平台指南、产品资料和专属知识整理为 Agent 可引用的创作素材。

#### 2.3.1 背景与目的

枝化生长不能只依赖模型自身知识。营养库提供可被引用的资料来源，让生成内容更贴近平台经验、领域背景和种子专属语境。

#### 2.3.2 核心功能描述

1. 用户在营养库页面切换公共营养库和种子专属营养库。
2. 用户创建公共营养库并上传营养内容。
3. 用户进入某个种子后查看该种子的专属营养内容。
4. 用户为营养内容填写标题、正文、标签和说明。
5. 用户在枝化生长前选择要参考的营养库或具体资料。

#### 2.3.3 预期交付形态

营养库像一个轻量资料库。公共营养库展示可被所有种子引用的资料，种子专属营养库展示只服务于当前种子的资料。在工作区发起生长时，用户可以通过引用方式选择某个营养库或资料，让本次生成带上明确参考。

#### 2.3.4 本期不做（模块级边界）

- 不做复杂知识库问答
- 不做自动网页采集
- 不做向量化知识管理
- 不做跨用户共享市场

### 2.4 种子工作区与内容树画布

**模块一句话价值**：让用户在一张可浏览的内容树中理解一个灵感的全部生长过程。

#### 2.4.1 背景与目的

内容森林的核心体验不在普通表单，而在能承载种子、果实、代际关系和状态变化的工作区。用户需要看见内容如何从一个灵感长成一棵树。

#### 2.4.2 核心功能描述

1. 用户进入种子工作区后看到该种子的内容树。
2. 用户拖动画布浏览种子和果实节点。
3. 用户点击节点查看种子或果实详情。
4. 用户从种子节点或果实节点发起枝化生长。
5. 用户看到新生成果实作为子节点挂载到来源节点下。

#### 2.4.3 预期交付形态

工作区是一个大画布，中心或起点展示种子节点，后续果实以树状结构向外生长。用户可以拖动浏览、点击节点查看详情，并从选中节点下方唤起悬浮输入框。节点状态通过颜色、标签或图标区分候选、已选择、已淘汰、已发布和已反馈。

#### 2.4.4 本期不做（模块级边界）

- 不做自由白板绘图
- 不做多人实时协作
- 不做复杂自定义布局
- 不做跨种子树合并

### 2.5 枝化生长输入与生成过程

**模块一句话价值**：让用户从任意种子或果实出发，补充本次想法并生成下一批内容果实。

#### 2.5.1 背景与目的

用户需要在自然创作语境中发起生长，而不是被迫填写复杂任务表单。该模块把用户的临时想法、生成器、营养引用和生长参数汇聚到一次可理解的生长操作中。

#### 2.5.2 核心功能描述

1. 用户点击种子或果实后看到悬浮输入框。
2. 用户输入本次生长的补充想法，也可以留空。
3. 用户引用营养库或具体营养内容。
4. 用户选择生成器、果实数量和变异率。
5. 用户回车后看到果实生成中的加载状态。
6. 用户可在等待时浏览其他未生成节点。

#### 2.5.3 预期交付形态

悬浮输入框位于工作区底部，像创作指令台。主输入区承载本次想法，旁边或下方展示生长信息栏，包括生成器、果实数量、变异率和参考营养。提交后，来源节点附近出现生成中动效，完成后新果实节点自动挂载到内容树上。

#### 2.5.4 本期不做（模块级边界）

- 不做批量并发复杂调度
- 不做高级参数系统
- 不做自动选择最佳生成器
- 不做生成中结果编辑

### 2.6 果实详情与人为物竞天择

**模块一句话价值**：让用户对生成出的内容果实进行浏览、选择、保留或淘汰。

#### 2.6.1 背景与目的

枝化生长会产生多个候选果实，但不是所有果实都值得发布验证。该模块让第一期的选择压力由用户显式施加，并把选择结果留在内容树中。

#### 2.6.2 核心功能描述

1. 用户点击果实节点查看内容详情。
2. 用户查看果实正文、摘要、平台类型和基因标签。
3. 用户将果实标记为选择、保留或淘汰。
4. 用户填写选择或淘汰原因。
5. 用户从被选择的果实进入发布验证记录。

#### 2.6.3 预期交付形态

果实详情以侧栏或弹层呈现，重点展示内容正文、核心摘要、生成来源、使用生成器、参考营养和基因标签。详情区提供清晰的物竞天择操作按钮：选择、保留、淘汰。被淘汰果实在树上弱化显示，但不从树中删除。

#### 2.6.4 本期不做（模块级边界）

- 不做 AI 自动决策
- 不做复杂评分模型
- 不做果实版本合并
- 不做批量选择淘汰

### 2.7 发布验证与数据回流

**模块一句话价值**：让用户把真实平台发布和表现数据挂回对应果实，形成可追溯反馈。

#### 2.7.1 背景与目的

第一期不做自动发布和自动监视器，但内容进化必须知道哪个果实被发布到哪里，以及真实表现如何。该模块负责把外部平台验证结果回流到内容树。

#### 2.7.2 核心功能描述

1. 用户将被选择果实标记为已发布。
2. 用户记录发布平台、发布时间和发布链接。
3. 用户记录是否对果实做过人工修改。
4. 用户手动录入平台数据快照。
5. 用户补充评论摘录和主观观察。
6. 用户查看果实的历史数据记录。

#### 2.7.3 预期交付形态

发布验证以果实详情中的记录面板呈现。用户先填写发布信息，再按时间追加数据快照。每条快照包含基础数据和备注，果实节点在画布上显示已发布或已反馈状态。用户可以从果实详情回看该果实的验证过程。

#### 2.7.4 本期不做（模块级边界）

- 不做平台自动发布
- 不做自动数据抓取
- 不做定时监视器
- 不做跨平台数据分析报表

### 2.8 基因汲取与迭代进化

**模块一句话价值**：让用户把果实表现转化为该种子基因库中可复用的成功基因和失败教训。

#### 2.8.1 背景与目的

进化不是多生成几次，而是让上一轮选择和反馈影响下一轮生长。该模块把果实的基因、选择结果和发布数据转化为可沉淀、可确认、可引用的经验。

#### 2.8.2 核心功能描述

1. 用户查看果实自带的基因标签。
2. 用户在发布反馈后触发基因汲取建议。
3. 用户查看成功基因和失败教训建议。
4. 用户确认将建议写入该种子的基因库。
5. 用户从该果实继续枝化生长下一代。
6. 用户看到下一代果实与来源果实的关系。

#### 2.8.3 预期交付形态

果实详情中包含基因区和汲取区。基因区展示内容角度、标题结构、情绪钩子、平台格式、受众切口等标签；汲取区在有选择结果或发布数据后给出可确认的成功基因和失败教训。确认后，这些内容进入该种子的基因库，并在下一次生长时作为高价值上下文之一。

#### 2.8.4 本期不做（模块级边界）

- 不做复杂遗传算法
- 不做自动改写生成器
- 不做全自动基因沉淀
- 不做多目标适应度模型

## 3. 本期整体交付形态

第一期完成后，用户打开产品会先看到种子库。用户创建一个种子，写下自己的灵感、产品或宣传内容后，进入该种子的工作区。工作区是一张内容树画布，起点是种子节点，用户点击种子后在底部悬浮输入框中补充本次生长想法，选择生成器、果实数量、变异率和参考营养库，然后提交枝化生长。生成过程中，来源节点附近出现加载动效；生成完成后，新果实作为子节点出现在内容树上。

用户点击果实可以查看正文、摘要、生成来源、基因标签和参考营养，并对果实执行选择、保留或淘汰。被选择的果实可以记录人工发布信息，用户手动填写平台、链接、发布时间和数据快照。数据回流后，用户可以触发基因汲取建议，将成功基因或失败教训确认写入该种子的基因库。之后，用户可以继续点击该果实或原始种子发起下一轮枝化生长，让内容树逐步形成可追溯的进化路径。

## 5. 参考约束与衔接说明

### 5.1 上期回顾

本期为首期启动，无上期交付内容。本期以前的主要成果是《内容森林概念设计文档》，它定义了项目的核心隐喻、第一期闭环和内容进化方向。

### 5.2 下期预告

下期可以在第一期闭环跑通后探索以下方向：

- 引入 AI 或规则辅助的物竞天择，让选择不再只依赖人工判断。
- 将手动数据回流升级为半自动或自动监视器。
- 强化生成器生态，支持生成器版本、模板复用和外部导入体验。
- 将营养库升级为更完整的知识库能力，提升检索和引用质量。
- 引入更明确的适应度评分，辅助用户理解内容进化方向。

### 5.3 对齐项目总方向

本期规划直接对齐《内容森林概念设计文档》中的以下章节：

- 2.1 核心价值主张：本期围绕“枝化生长、物竞天择、发布验证、数据回流、迭代进化”构建最小闭环。
- 2.2 关键解决方案拆解：本期对应灵感即种子、枝化生长、生成器与营养库、物竞天择、基因汲取和自我迭代。
- 3.2 典型使用场景：本期覆盖从一个灵感生成多平台内容、寻找最佳内容模板、基于反馈继续扩散等核心场景。
- 4.1 第一期核心交付物：本期将种子管理、生成器上传、营养库、内容树工作区、发布验证、数据回流和迭代进化落到用户可操作的产品形态中。
