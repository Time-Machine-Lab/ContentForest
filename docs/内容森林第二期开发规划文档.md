# 内容森林 - 第二期规划

## 1. 本期目标与定位

### 1.1 本期愿景陈述

本期结束后，用户只需要用一小段话描述自己的灵感、产品或想法，内容森林就能主动理解传播目标，补全平台语境，发散出多个差异化内容方向，并通过用户选择、真实发布反馈和基因沉淀，帮助用户更快找到接近爆款表达的内容路线。

### 1.2 与项目总方向的对齐

本期直接服务于内容森林“灵感种子 → 枝化生长 → 物竞天择 → 数据反馈 → 迭代进化”的核心闭环，将第一期的内容生成流程升级为更接近“内容进化搜索”的产品能力。

第一期重点让用户能够完成枝化生长与人工选择；第二期重点让系统具备更强的营养补充、创作发散、真实反馈回流和负反馈调节能力，使果实不只是被生成出来，而是持续向更适合平台传播的方向进化。

### 1.3 本期核心要解决的问题

- 问题 1：用户只提供 100 字左右灵感时，系统缺少足够的平台特征、用户语境和传播素材，容易生成结构完整但广告感强、真实感不足的内容。  
  期望：系统能在低输入成本下自动形成创作简报，补全目标人群、平台语感、内容方向和证据缺口，让用户无需先准备大量资料也能开始探索。

- 问题 2：当前枝化生长更像在已有上下文里生成多个文案变体，探索步长偏小，缺少明确的解空间搜索、真实反馈回流和负反馈调节。  
  期望：系统能根据阶段自动决定大步探索或小步收敛，让每轮生成都更像一次有目的的内容实验。

- 问题 3：当前果实更偏文本形态，且生成过程对用户来说仍然偏黑盒。  
  期望：果实能够承载图片、视频等多媒体内容，多媒体资源也能作为枝化生长输入；同时用户可以在生成中看到临时生成路径图，知道某颗果实当前走到哪一步。

## 2. 功能模块规划

### 2.1 营养库活化与营养汲取任务

**模块一句话价值**：让营养库从“静态资料存放处”升级为“可被补充、刷新、引用和验证的活体知识来源”。

**用户视角描述**：

背景与目的（Why）：用户想在小红书、抖音、推特等平台宣传同一个想法时，常常会临时发现“我缺少这个方向的平台资料”。例如用户想知道“AI 设计工具在小红书是怎么宣传的，有哪些爆款表达”。当前公共营养库虽然可以承载这些资料，但它主要依赖人工提前整理，容易变成死资料：不知道什么时候过时，不知道是否被使用，不知道缺口出现时如何快速补充。

做什么（What）：保留现有“用户手动创建营养内容”的投喂能力，同时新增 Agent 研究入口。当用户觉得缺少某类平台资料时，可以发起营养汲取任务。Agent 根据用户的问题进行研究、整理和拆解，输出一份营养候选。用户可以选择本次临时引用，也可以确认后保存到公共营养库或种子专属营养库。

预期效果（How it looks）：用户仍然主要在营养库和种子工作区中操作，不需要维护新的平台情报工作区。在生长前，如果用户觉得资料不足，可以点击“让 Agent 研究一下”，输入类似“找一下 AI 设计工具在小红书上的宣传方式和爆款表达”。系统生成营养候选后，用户可以直接引用到本次枝化生长，也可以保存为长期营养。

**本期不包含**：

- 不做独立的平台情报工作区
- 不做脚本化批量采集
- 不做复杂推荐算法反推
- 不做平台账号运营自动化
- 不把 Agent 研究结果自动写入正式营养库

#### 2.1.1 背景与目的

平台内容创作依赖活的资料。公共营养库已经可以承载平台规则、爆款样本和内容拆解，但如果资料全靠提前人工维护，营养库就会逐渐变成死数据。该模块的重点不是重新设计知识容器，而是为现有营养库补上“发现缺口 → 研究补充 → 候选确认 → 引用验证 → 过期刷新”的生命循环。

#### 2.1.2 核心功能描述

1. 用户继续通过营养库手动创建营养内容，这是最基础的用户投喂入口。
2. 用户在营养库或枝化生长前发起 Agent 研究任务，描述自己缺少的资料方向。
3. Agent 输出营养候选，内容可以包含平台宣传方式、爆款表达、标题结构、封面文案、用户痛点、广告感边界和可复用样本。
4. 用户选择将营养候选临时用于本次生长，或确认保存到公共营养库、种子专属营养库。
5. 营养内容记录来源、生成时间、适用平台、适用主题、可信度、新鲜度和使用情况。
6. 当营养内容长期未更新或多次引用效果不佳时，系统提示用户重新研究或归档。

#### 2.1.3 预期交付形态

本期不新增独立平台情报页面，而是在两个现有位置增强体验：

- **营养库页面**：新增“Agent 研究生成营养”入口。用户输入研究问题后，系统生成营养候选卡片，用户确认后保存。
- **种子工作区悬浮输入框**：当用户发起枝化生长时，可以临时请求 Agent 补充某类营养，例如“补充小红书 AI 设计工具爆款表达”。生成的营养候选可以本次临时引用，也可以保存。

营养候选以 Markdown 正文承载核心内容，以系统 Meta 记录来源、平台、主题、新鲜度、可信度和使用情况。用户感知上仍然是在使用营养库，而不是维护一个额外系统。

#### 2.1.4 本期不做（模块级边界）

- 不新增平台情报工作区
- 不实现脚本爬虫或批量采集
- 不让 Agent 自动污染正式营养库
- 不自动判断某条营养一定是真理
- 不把营养汲取做成独立大型研究产品

### 2.2 种子主简报与本轮生长简报

**模块一句话价值**：让用户用低成本输入启动高质量内容探索，同时让 Agent 对种子拥有一份可持续修正的创作地图。

**用户视角描述**：

背景与目的（Why）：内容森林希望用户只用一小段话就能开始创作，但一个 100 字想法通常缺少内容对象、受众假设、场景动机、表达边界、证据素材和平台语境。如果直接生成内容，容易变成空泛介绍、硬广或缺乏真实感的套话。种子主简报的目的不是替用户决定最终方向，而是在不增加填写负担的前提下，帮助系统先理解种子、发现探索机会和上下文缺口。

做什么（What）：用户创建种子后，可以选择生成一份长期存在的种子主简报。主简报包含种子理解、内容对象、受众假设、动机与场景、候选探索方向、证据缺口、风险边界和可补充素材建议。用户可以直接使用，也可以编辑、刷新或扩展。每次枝化生长时，系统会基于种子文档、种子主简报、选中节点、用户输入、引用营养和引用基因临时组装本轮生长简报，用作本次 Agent 执行的任务上下文。

预期效果（How it looks）：用户进入种子工作区后，除了看到内容树，还能看到一个“种子主简报”面板。面板用简短卡片展示“这个种子是什么”“可能面向哪些人”“有哪些表达机会”“当前缺少哪些证据”“可以先探索哪些方向”。用户可以一键把某个方向放入悬浮输入框，也可以完全跳过简报直接发起生长。

**本期不包含**：

- 不要求用户填写复杂问卷
- 不做商业定位咨询报告或账号策划报告
- 不强制用户确认后才能生成
- 不保证自动推断完全正确
- 不把简报作为种子文档的替代品

#### 2.2.1 背景与目的

低输入成本是内容森林的重要体验目标，但低输入不代表系统可以忽略上下文缺口。该模块用于把短灵感转化为可执行、可编辑、可复用的创作地图。种子文档仍然是原始事实来源，种子主简报是对种子文档的理解、扩写和探索建议；本轮生长简报则是枝化生长执行前临时组装的任务书。

种子主简报应保持泛用性，不只服务于产品宣传。它需要同时适配产品推广、美食博主、AI 教学、搞笑内容、个人 IP、项目介绍、知识科普等不同类型的种子。因此简报不应固定为“产品、人群、卖点”的单一路径，而应采用“通用理解层 + 可变创作层”的结构。

#### 2.2.2 核心功能描述

1. 用户创建种子后，可以选择生成种子主简报。
2. 用户查看系统对种子的理解、受众假设、场景动机、探索方向和证据缺口。
3. 用户可以编辑、刷新或扩展种子主简报。
4. 用户可以从候选探索方向中选择一个方向发起枝化生长。
5. 用户发起枝化生长时，系统临时组装本轮生长简报。
6. 用户未创建种子主简报时，枝化生长仍可正常执行，只是上下文补全能力较弱。
7. 用户选择果实继续生长时，系统会同时参考种子主简报和果实生长摘要，但不会为每颗果实创建完整主简报。

#### 2.2.3 预期交付形态

种子主简报作为工作区内的轻量侧栏出现，不打断用户生成流程。它由多张小卡组成：种子理解、内容对象、受众假设、创作机会、证据缺口、风险边界、候选方向地图。每张卡都可以被用户折叠或编辑，候选方向可以直接变成悬浮输入框中的生长意图。

本轮生长简报不一定作为独立页面出现。它更像一次枝化生长的内部任务上下文，由系统在提交生长时组装，并在日志或调试视图中可追溯。果实作为生长源时，只提供果实生长摘要、正文、基因标签和历史反馈，不需要维护完整简报。

#### 2.2.4 本期不做（模块级边界）

- 不做长篇市场分析
- 不要求完整品牌资料
- 不阻塞枝化生长
- 不替用户决定最终定位
- 不为每颗果实维护完整主简报
- 不让简报覆盖或改写种子原文事实

### 2.3 枝化生长管线架构化

**模块一句话价值**：让枝化生长从一次 Agent 调用升级为可编排、可观测、可替换的内容进化流水线。

**用户视角描述**：

背景与目的（Why）：内容森林的目标不是一次生成一篇看起来不错的文案，而是为某个灵感寻找最适合平台传播的表达路线。当前枝化生长如果只依赖一个大 Prompt，很容易变成“同一方向的多篇改写”，缺少明确的上下文补全、方向搜索、基因继承、负反馈规避和结果封装。第二期需要把枝化生长拆成一条清晰的管线，让每一层负责一类确定职责。

做什么（What）：用户发起枝化生长时，系统依次完成输入接收、上下文补全、创作搜索、注意力编排、生成执行和结果封装。用户不需要理解复杂算法，但可以看到本轮搜索模式、果实方向、引用资源和生成来源。系统会为不同果实安排更有差异的创作方向，使每颗果实代表一个可被验证的内容假设。

预期效果（How it looks）：在工作区悬浮输入框的高级设置中，用户能看到本轮搜索模式和探索强度。生成后的果实在内容树上不仅有标题，还带有方向标识，例如“真实复盘型”“清单模板型”“反常识观点型”“对比测评型”。用户可以直观看出这一轮不是重复写三篇，而是在试不同路线。开发和调试侧则可以追溯每颗果实经过了哪些上下文、方向、基因和生成器。

**本期不包含**：

- 不做复杂数学优化面板
- 不要求用户理解算法术语
- 不做全自动多轮无人实验
- 不保证一次找到最优方向
- 不做可视化 Agent 工作流编排器
- 不让 Agent 直接越过领域服务写入系统事实

#### 2.3.1 背景与目的

内容进化需要控制探索步长。冷启动时应大步探索多个内容方向；有反馈后应逐步收敛，围绕有效方向变异和强化。为了让这套算法模型可以持续迭代，枝化生长不能被写死成一次固定 Prompt，而应该被设计为可替换策略的管线。

该设计借鉴企业级 Agent 开发中的常见做法：工作流编排、上下文组装、规划与路由、上下文排序、工具执行、输出解析、追踪审计和人工决策节点。内容森林不需要在第二期实现完整企业级平台，但应吸收其中的工程原则，让不确定性交给 Agent，让确定性交给系统。

#### 2.3.2 核心功能描述

1. 输入层：接收来源节点、用户输入、生成器、果实数量、引用资源、多媒体资源和生长参数。
2. 上下文补全层：读取种子文档、种子主简报、来源节点、引用营养、引用基因、多媒体资源和必要的系统上下文。
3. 创作搜索层：决定本轮探索路线，例如广泛探索、方向强化、局部变异、负反馈规避。
4. 注意力编排层：决定哪些上下文影响本次生成、影响什么维度、占据多大权重。
5. 生成执行层：调用生成器 Skill 生成内容 payload，生成器可在自身能力范围内产出文本、图片、视频等多媒体内容。
6. 结果封装层：把生成器输出封装为候选果实，补全标题、生成器来源、方向标签、基因标签、多媒体挂载关系和可追溯信息。
7. 用户基于果实方向、内容质量和真实反馈进行选择、淘汰或继续生长。

#### 2.3.3 预期交付形态

工作区生成区域增加“探索模式”信息。默认由系统推荐，用户可以简单切换。生成结果在内容树上呈现为不同方向的果实簇，每颗果实都能打开查看“本果实正在验证什么内容假设”。整体体验像是在地图上探索不同路线，而不是在同一个点附近反复改字。

工程侧需要保留每颗果实的生长追踪信息，包括输入来源、引用资源、多媒体资源、使用的搜索策略、注意力分配摘要、生成器来源和结果封装过程。该追踪信息不一定全部暴露给普通用户，但需要支持后续排查、回放、评估和算法迭代。

方向与基因需要保持清晰边界：方向表示本轮在内容解空间里往哪里探索，例如真实复盘、避坑教程、搞笑反差、对比测评；基因表示可复用、可继承、可变异或可规避的表达特征，例如产品名后置、反常识开头、清单化结构、直接外链规避。方向可以选择基因，基因也可以影响方向，但二者不互相替代。

#### 2.3.4 本期不做（模块级边界）

- 不展示复杂算法参数
- 不做自动无限迭代
- 不做真实流量自动分配
- 不替代用户选择判断
- 不把搜索策略写成不可替换的固定逻辑
- 不把所有上下文无差别塞给生成器
- 不要求所有生成器都支持多媒体生成

### 2.4 平台基因与负反馈沉淀

**模块一句话价值**：让成功经验和失败教训都能变成下一轮生长可用的内容基因。

**用户视角描述**：

背景与目的（Why）：内容进化不只依赖成功样本，也依赖失败反馈。当前基因汲取已经能沉淀表达特征，但第二期需要让用户选择、用户淘汰、真实发布数据和失败原因共同参与基因生成，使系统逐步知道什么该继承、什么该规避、什么只适合特定场景。

做什么（What）：用户选择、淘汰或回填发布反馈后，系统可以生成待确认的基因建议。基因建议区分成功因子、失败因子、适用人群、平台生态位和下一轮使用方式。用户确认后，它进入种子级或平台级基因经验。

预期效果（How it looks）：用户在基因库中看到的不再只是标签，而是一条条“可复用经验”。例如“产品名后置：前 70% 先讲方法，最后再露出工具，可降低广告感”。每条基因都带有来源证据和下一轮使用建议，用户可以在生长时引用。

**本期不包含**：

- 不自动确认正式基因
- 不删除其他谱系基因
- 不自动改写生成器
- 不把弱证据当成强结论

#### 2.4.1 背景与目的

内容森林需要把每次选择、淘汰和发布反馈转化为可复用经验。该模块用于让正反馈和负反馈都能指导下一轮搜索。

#### 2.4.2 核心功能描述

1. 用户从果实选择或淘汰触发基因汲取。
2. 用户从真实发布反馈触发负反馈沉淀。
3. 用户查看待确认的成功或失败基因建议。
4. 用户确认基因后在生长时引用。
5. 用户查看某条基因适用的方向和边界。

#### 2.4.3 预期交付形态

基因库页面保留正式经验浏览，同时新增“待确认经验”队列。每条建议以卡片展示：基因标题、正/负方向、证据来源、适用生态位和下一轮用法。用户可以确认、忽略或编辑后确认。工作区中引用基因时，用户能看到它是成功因子还是规避因子。

#### 2.4.4 本期不做（模块级边界）

- 不自动确认基因
- 不自动删除旧经验
- 不自动更新生成器
- 不把 AI 评估当真实数据

### 2.5 多媒体果实与生成路径可视化

**模块一句话价值**：让内容森林从纯文本果实扩展到图文、视频等多媒体果实，并让用户在生成中看见果实正在经历的生长路径。

**用户视角描述**：

背景与目的（Why）：平台内容并不总是纯文本。小红书、抖音、B 站、视频号等平台往往需要封面图、配图、短视频、脚本、分镜、参考图风格或素材附件。如果内容森林只把果实理解为 Markdown 文本，就会限制生成器能力，也会限制 Agent 对图片、视频等素材的理解和复用。同时，第二期枝化生长管线变长后，用户需要知道某颗生成中的果实当前走到哪一步，而不是只看到一个模糊的 loading。

做什么（What）：果实支持挂载图片、视频等多媒体内容。生成器可以在能力范围内产出多媒体内容，果实详情页可以展示这些内容。枝化生长输入层支持把多媒体内容作为本轮输入上下文，并明确用途，例如理解图片意思、参考图片风格、基于图片生成文案、模拟风格生成新图。生成中果实支持展示临时生成路径图，让用户点击后看到本次生成大致有哪些步骤、当前执行到哪一步、哪些步骤已完成或新增。

预期效果（How it looks）：用户在果实详情中不仅能看到 Markdown 正文，也能看到关联图片、视频、封面草图、分镜素材等多媒体内容。用户发起枝化生长时，可以选择某个图片或视频作为参考，并说明“理解这张图的风格”或“模仿这个封面方向”。当果实还在生成中时，用户点击生成中节点，可以看到类似 Agent plan 的路径图：任务被拆成若干步骤，执行过程中逐步打勾或补充步骤。生成结束后，该路径图从普通用户界面中消失，不作为果实正式内容保留。

**本期不包含**：

- 不要求所有生成器都支持图片或视频生成
- 不做复杂素材编辑器
- 不做视频剪辑时间线
- 不把生成路径图做成正式任务管理系统
- 不把生成中路径图长期展示在果实详情中

#### 2.5.1 多媒体果实

果实正文仍以 Markdown 作为主要展示载体，但果实可以挂载多媒体资源。多媒体资源可以来自用户上传、营养引用、生成器产出或 Agent 工具产出。前端展示时，Markdown 正文负责表达内容主体，多媒体挂载负责展示图片、视频、封面、分镜、参考素材等内容资产。

生成器不需要被强制统一为某种媒体格式。它只需要在自身能力范围内产出内容：纯文本生成器继续产出 Markdown，图片生成器可以产出图片和说明，视频生成器可以产出视频、脚本、分镜或视频生成结果。内容森林负责把这些输出挂载到果实上，并让用户能在果实详情里查看。

#### 2.5.2 多媒体作为枝化输入

枝化生长输入层需要支持多媒体资源引用。用户可以在悬浮输入框或高级设置中引用图片、视频等资源，并为资源指定用途。用途不是复杂参数，而是帮助 Agent 明确本轮要如何使用该资源，例如：

1. 理解图片或视频内容。
2. 提取图片风格或画面元素。
3. 参考封面结构生成新封面。
4. 基于视频或图片生成文案、标题、脚本。
5. 将多媒体资源作为某个平台内容形态的样例。

Agent 是否真的能处理某类多媒体，取决于当前 Agent 工具和生成器能力。系统需要在能力不支持时给出可理解的提示，而不是让用户误以为所有多媒体能力都天然可用。

#### 2.5.3 果实生成路径图

生成路径图只服务于生成中状态的用户感知。它参考 Agent plan 的交互方式：生成开始时，Agent 或管线根据本轮任务总结出若干步骤；生成过程中，系统可以标记步骤完成、更新当前步骤，也可以根据实际执行新增步骤。用户点击生成中的果实节点时，可以看到该果实当前的总路径和进度。

生成路径图不参与领域决策，也不作为果实正式内容保存。生成完成后，普通用户界面不再展示该路径图；工程侧仍可在日志或追踪记录中保留必要执行信息，用于排查和算法迭代。

#### 2.5.4 本期不做（模块级边界）

- 不实现专业图片编辑、视频剪辑或素材管理软件能力
- 不保证所有多媒体输入都能被 Agent 理解
- 不强制生成器输出统一多媒体结构
- 不把路径图做成可编辑计划
- 不让路径图影响物竞天择状态

## 3. 本期整体交付形态

第二期结束后，用户从种子库进入某个种子工作区时，首先看到的不只是内容树，还能看到系统为该种子生成的种子主简报。用户可以从主简报中选择一个候选探索方向，也可以完全跳过简报，直接在悬浮输入框中发起枝化生长。

生长前，系统会基于种子文档、种子主简报、来源节点、用户输入、引用营养、引用基因和多媒体资源组装本轮生长简报，并显示本轮探索模式，例如“广泛探索”或“方向强化”。生长完成后，内容树上会出现多个带有方向标识的果实。用户点击果实时，可以看到果实正文、多媒体内容、生成器来源、基因标签、发布记录、数据反馈和可继续生长的改进建议。

枝化生长内部不再被视为一次黑盒生成，而是一条可追溯的内容进化管线。系统会经过输入层、上下文补全层、创作搜索层、注意力编排层、生成执行层和结果封装层，把不确定的生成器输出转化为系统可用的候选果实。开发和调试侧可以追溯每颗果实使用了哪些上下文、方向、基因和生成器。

当果实处于生成中状态时，用户点击生成中节点可以看到临时生成路径图，了解当前果实正在经历哪些步骤、哪些步骤已完成、当前执行到哪里。生成完成后，路径图不作为果实正式内容展示。

如果用户认为某个果实像广告、缺少真实场景或没有收藏价值，可以通过选择、淘汰或发布后的真实反馈表达判断。用户也可以把选择、淘汰和真实反馈沉淀为待确认基因。当用户觉得缺少某类平台资料时，可以在营养库或生长输入框中发起 Agent 研究任务，生成可临时引用或确认保存的营养候选。

整体体验从“生成内容”升级为“围绕平台传播目标搜索内容表达路线”。

## 5. 参考约束与衔接说明

### 5.1 上期回顾

第一期已完成内容森林的基础闭环：用户可以创建种子，上传生成器，维护营养库，在工作区对种子或果实发起枝化生长，并通过人工物竞天择选择或淘汰果实。系统已经具备基因库、基因汲取、发布验证和数据回流的领域基础。

第二期在此基础上进一步解决“生成质量为什么不够像平台内容”的问题，重点提升营养活化、创作发散、多媒体表达、生成过程可感知、真实反馈回流和基因沉淀。

### 5.2 下期预告

第三期可进一步探索：

- 更强的营养刷新机制与 Agent 研究质量评估。
- 平台观察 Agent 在用户授权下辅助搜集和拆解真实案例。
- 接入真实发布数据后的自适应搜索资源分配。
- 多平台横向迁移同一个灵感的传播形式。
- 让营养库、基因经验和生成器形成更强的协同演化。

### 5.3 对齐项目总方向

本期规划直接对齐内容森林顶层概念中的以下方向：

- “灵感为种子” → 对应本章 2.2 种子主简报与本轮生长简报。
- “AI + 经验 内容生成” → 对应本章 2.1 营养库活化与营养汲取任务、2.5 多媒体果实与生成路径可视化。
- “自我迭代 + 进化” → 对应本章 2.3 枝化生长管线架构化、2.4 平台基因与负反馈沉淀。
- “物竞天择” → 对应本章第 4 节成功标准中对选择、淘汰、评估和基因沉淀的验证要求。
