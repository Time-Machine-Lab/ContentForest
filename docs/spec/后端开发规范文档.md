# 内容森林后端开发规范

## 1. 核心原则 (Core Principles)

后端代码必须服务于“轻量模块化单体 + 可替换基础设施 + Agent 解耦”的架构方向。任何开发都必须优先保护三条边界：内容本体在文件中，系统事实在数据库中，Agent 能力通过端口调用。

### 1.1 开发技术栈 (Tech Stack)

- **后端运行时**：Node.js + TypeScript。
- **应用形态**：模块化单体，不按微服务拆分。
- **通信协议**：前后端通过 HTTP 接口通信；接口只承载应用能力，不暴露内部存储细节。
- **数据存储**：SQLite 作为第一期数据库，用于维护系统事实；后续允许通过适配层替换为 MySQL。
- **内容存储**：本地文件系统作为第一期内容本体存储，用于保存生成器 Skill、果实 Markdown、营养 Markdown 和附件。
- **Agent 能力**：后端只依赖 `AgentPort` 抽象；具体 Agent Runtime 可内置实现，也可后续替换为第三方 Agent。
- **LLM 接入**：LLM 调用只能存在于 Agent Runtime 或其基础设施适配中，业务模块禁止直接调用模型 SDK。

### 1.2 项目结构与目录规范 (Project Structure)

```text
/content-forest-backend/
├── src/                         /**/ 后端源码根目录，只存放应用代码，不存放运行时数据文件。
│   ├── app/                     /**/ 应用启动与依赖装配层，只负责组装模块、注册中间件和启动服务，不写业务规则。
│   │   ├── bootstrap/           /**/ 启动流程与运行环境装配，禁止出现具体领域逻辑。
│   │   └── config/              /**/ 配置读取与校验，禁止由业务模块直接读取环境变量。
│   │
│   ├── modules/                 /**/ 业务模块层，按内容森林领域模块划分。模块之间只能通过公开应用服务或端口交互。
│   │   ├── seed/                /**/ 种子模块：负责灵感种子相关应用能力，不直接操作文件系统。
│   │   ├── generator/           /**/ 生成器模块：负责生成器上传、导入和选择，不要求生成器遵守果实契约。
│   │   ├── nutrient/            /**/ 营养库模块：负责公共营养和种子专属营养的组织与引用。
│   │   ├── workspace/           /**/ 工作区模块：负责内容树视图、节点关系与工作区状态组织。
│   │   ├── growth/              /**/ 枝化生长模块：负责生长任务编排，禁止直接调用底层 LLM。
│   │   ├── fruit/               /**/ 果实模块：负责果实状态、果实展示入口与内容树挂载关系。
│   │   ├── publication/         /**/ 发布验证模块：负责人工发布记录，不集成平台自动发布。
│   │   ├── feedback/            /**/ 数据回流模块：负责人工数据快照与观察记录，不抓取平台数据。
│   │   └── gene/                /**/ 基因汲取模块：负责基因建议、确认与沉淀，不自动改写生成器。
│   │
│   ├── agent/                   /**/ Agent 能力层，只暴露 AgentPort 和内置 Agent Runtime。业务模块禁止绕过端口调用实现。
│   │   ├── ports/               /**/ AgentPort 定义，描述内容森林需要的 Agent 能力边界。
│   │   ├── runtime/             /**/ 第一期开箱内置 Agent Runtime，实现可替换。
│   │   └── skills/              /**/ 内容森林内置 Skill，如枝化生长 Skill。用户上传生成器不放在这里。
│   │
│   ├── content-access/          /**/ 内容访问层，统一读写生成器 Skill、果实 Markdown、营养 Markdown 和附件。
│   │   ├── ports/               /**/ 内容存储端口，业务模块只能依赖端口。
│   │   └── adapters/            /**/ 本地文件系统等具体实现，禁止被 modules 直接 import。
│   │
│   ├── storage/                 /**/ 系统事实存储层，统一访问数据库。
│   │   ├── ports/               /**/ 数据库存储端口，描述模块需要的持久化能力。
│   │   ├── adapters/            /**/ SQLite 等具体实现，禁止被业务模块绕过端口直接使用。
│   │   └── migrations/          /**/ 数据库迁移文件，只描述系统事实结构演进。
│   │
│   ├── interface/               /**/ 输入输出适配层，负责 HTTP 请求解析、响应组装和错误映射。
│   │   └── http/                /**/ HTTP 控制器，不写业务规则，不直接访问数据库、文件系统或 Agent Runtime。
│   │
│   ├── shared/                  /**/ 跨模块共享的纯基础能力，如错误类型、日志、结果封装、通用校验。
│   │   ├── errors/              /**/ 统一错误定义与错误映射。
│   │   ├── logging/             /**/ 日志封装，不依赖业务模块。
│   │   └── utils/               /**/ 纯函数工具，禁止引入存储、HTTP、Agent 或业务模块。
│   │

├── data/                        /**/ 本地开发运行数据目录，保存 SQLite 文件和内容文件，禁止提交真实运行数据。
└── scripts/                     /**/ 构建、检查、迁移和本地辅助脚本，禁止放业务逻辑。
```

### 1.3 Git 与代码流 (Git Workflow)

- **分支模型**：采用轻量 feature 分支开发，主分支始终保持可运行。
- **提交规范**：提交信息使用清晰前缀，如 `feat:`、`fix:`、`docs:`、`refactor:`、`test:`。
- **变更粒度**：一次提交只做一个意图明确的变更，避免把模块重构、功能开发和格式化混在一起。
- **代码评审**：涉及模块边界、存储策略、AgentPort、内容访问层的变更必须重点评审。
- **文档同步**：修改架构边界或开发规范时，必须同步更新对应文档，不允许只改代码。

### 1.4 命名与代码风格 (Naming & Code Style)

- **语言规范**：后端统一使用 TypeScript，禁止新增无类型 JavaScript 业务代码。
- **模块命名**：业务模块使用领域名，不使用 `controller`、`service`、`model` 作为一级模块目录。
- **端口命名**：对外部能力的依赖必须通过 `Port` 表达，如 Agent、Storage、Content Access 等。
- **适配器命名**：具体基础设施实现必须通过 `Adapter` 表达，禁止业务模块依赖适配器。
- **边界命名**：目录名必须表达 import 边界，不能用含义模糊的 `common-business`、`misc`、`manager`。
- **Lint 工具**：必须使用统一 ESLint 与格式化规则，禁止提交未通过检查的代码。
- **注释原则**：只解释复杂决策和边界原因，不为显而易见的代码写重复注释。

### 1.5 模块实现规则 (Module Implementation Rules)

- **契约先行**：新增或修改领域能力前，必须先补齐对应 `docs/sql/*.sql` 与 `docs/api/*.yaml`；实现中的表结构、接口能力、状态枚举必须能回溯到顶层契约。
- **应用服务入口**：领域模块以 Application Service 作为唯一业务入口；Controller 只负责请求解析、响应包装和错误映射，禁止直接访问 Storage、Content Access、Agent、数据库或文件系统。
- **端口放置**：系统事实存储端口放在 `src/storage/ports`，内容本体访问端口放在 `src/content-access/ports`；业务模块只能依赖 Port，具体 Adapter 可替换。
- **测试替身优先**：领域服务测试优先使用内存 Port Adapter，避免依赖真实数据库或文件系统；基础设施 Adapter 可单独测试。
- **归档语义**：选择归档而非删除的领域对象，不得额外实现硬删除能力；归档不得破坏身份、内容和历史关系。
- **内容与 Meta 分离**：Markdown 只保存用户可见正文；状态、关系、索引、审计时间、内容位置等系统事实必须由数据库维护。
- **错误类型**：业务错误应使用统一应用错误表达，至少区分参数校验失败、资源不存在和内容访问失败；禁止把底层异常原样暴露给前端。

### 1.6 其他开发事项 (Other Conventions)

- Markdown 只保存内容本体，禁止写入由数据库维护的 meta、状态、关系、索引或统计信息。
- 业务模块禁止直接读写本地文件，必须通过内容访问层。
- 业务模块禁止直接使用数据库客户端，必须通过存储端口。
- 业务模块禁止直接调用 LLM SDK，必须通过 AgentPort。
- Agent 不拥有数据库写权限，Agent 输出必须由后端应用服务校验后再落地。
- 生成器 Skill 不强制包含内容森林专用 manifest，也不强制输出果实结构。
- 枝化生长 Skill 是内部 Agent 系统能力，负责把生成器 payload 封装为内容森林可管理的果实。
- 系统事实以数据库为准；文件缺失或损坏应被检测和报告，不允许静默伪造系统状态。
- 任何“临时绕过端口”的写法都必须拒绝，即使能更快完成当前功能。

## 2. 治理与维护 (Governance & Maintenance)

### 2.1 测试规范 (Testing Strategy)

- **模块测试**：种子、生成器、营养库、工作区、枝化生长、果实、发布验证、数据回流、基因汲取模块都应覆盖核心用例。
- **端口测试**：Storage Port、Content Access Port、AgentPort 必须有替身实现或测试替身，确保业务模块可脱离具体基础设施测试。
- **边界测试**：重点验证业务模块不会绕过端口访问数据库、文件系统或 Agent Runtime。
- **端到端流程**：至少覆盖一次完整闭环：创建种子 → 枝化生长 → 生成果实 → 人为物竞天择 → 发布验证 → 数据回流 → 基因汲取。
- **测试策略**：第一期不追求覆盖率数字优先，优先保护核心闭环和架构边界。

### 2.2 部署与 CI/CD (Deployment & CI/CD)

- **环境划分**：第一期至少区分本地开发环境和可演示环境。
- **构建检查**：每次合入主分支前必须通过类型检查、Lint、测试和构建。
- **数据迁移**：数据库结构变化必须通过迁移文件管理，不允许手工修改运行库后不留记录。
- **内容目录**：运行时内容目录必须可配置，禁止在代码中写死开发者本机绝对路径。
- **密钥管理**：LLM Key、数据库连接、内容目录路径等运行配置必须从环境配置读取，禁止提交到仓库。
- **发布纪律**：发布前必须能完成核心闭环走查，不能只以服务启动成功作为发布标准。

### 2.3 监控与告警 (Monitoring & Alerting)

- **日志规范**：关键用户动作、生长任务、Agent 调用、文件写入、数据库写入和异常都必须有结构化日志。
- **任务追踪**：枝化生长和基因汲取属于长耗时任务，必须能追踪开始、完成、失败和取消状态。
- **错误处理**：文件缺失、数据库记录缺失、Agent 输出不可用、LLM 调用失败必须形成明确错误，不允许吞掉。
- **一致性检查**：需要保留文件与数据库关系检查能力，用于发现内容文件缺失、孤儿文件或数据库悬挂引用。
- **用户可感知错误**：后端错误必须被映射为前端可理解的信息，禁止把底层异常原样暴露给用户。
