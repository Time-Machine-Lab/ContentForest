# 内容森林前端开发规范

## 1. 核心原则 (Core Principles)

前端代码必须服务于“内容进化工作台”的产品形态。第一期前端不是营销站，也不是传统后台表格系统，而是围绕种子工作区、内容树画布、枝化生长输入和果实详情构建的高交互应用。

所有前端开发开始前，必须先阅读仓库根目录的 [DESIGN.md](../../DESIGN.md)，确认当前模块符合内容森林设计系统。若设计实现需要偏离 DESIGN.md，必须先在任务说明或评审记录中写明原因。

### 1.1 开发技术栈 (Tech Stack)

- **前端框架**：Nuxt + Vue 3 + TypeScript。
- **渲染策略**：第一期工作台页面以客户端交互为优先，不以 SSR 为核心要求。
- **页面形态**：种子库、生成器、营养库和种子工作区组成主应用。
- **通信协议**：通过后端应用接口读取工作区状态、提交生长任务、更新用户操作结果。
- **内容展示**：果实正文、营养内容和生成器说明以 Markdown 渲染为主。
- **状态来源**：系统事实以后端返回为准，前端只维护交互状态和临时 UI 状态。
- **画布能力**：内容树画布可以使用专门图形/节点库，但必须通过本项目组件层封装，禁止业务页面直接散落依赖库 API。

### 1.2 项目结构与目录规范 (Project Structure)

```text
content-forest-front/
├── app/                         /**/ Nuxt 应用根目录，只负责应用级入口、布局和插件装配。
│   ├── pages/                   /**/ 页面路由层，只编排页面级组件，不写复杂业务逻辑。
│   │   ├── seeds/               /**/ 种子库与种子工作区页面入口。
│   │   ├── generators/          /**/ 生成器管理页面入口。
│   │   └── nutrients/           /**/ 营养库页面入口。
│   │
│   ├── layouts/                 /**/ 应用布局，如主工作台布局和基础导航，不承载业务状态。
│   ├── plugins/                 /**/ Nuxt 插件注册，只放框架级扩展，不放领域逻辑。
│   └── middleware/              /**/ 路由守卫或访问控制，禁止调用领域业务写操作。
│
├── src/                         /**/ 前端业务源码根目录，按领域和能力分层组织。
│   ├── modules/                 /**/ 前端领域模块，按用户可感知模块划分。
│   │   ├── seed/                /**/ 种子库与种子详情相关 UI、状态和数据访问封装。
│   │   ├── generator/           /**/ 生成器上传、列表和选择相关 UI。
│   │   ├── nutrient/            /**/ 营养库列表、资料展示和引用选择 UI。
│   │   ├── workspace/           /**/ 种子工作区页面编排、内容树视图和节点详情入口。
│   │   ├── growth/              /**/ 枝化生长输入框、参数面板、生成中状态和任务反馈。
│   │   ├── fruit/               /**/ 果实卡片、果实详情、Markdown 展示和状态操作。
│   │   ├── publication/         /**/ 发布验证记录 UI。
│   │   ├── feedback/            /**/ 数据回流与数据快照 UI。
│   │   └── gene/                /**/ 基因标签展示、基因汲取建议和确认 UI。
│   │
│   ├── components/              /**/ 跨模块通用 UI 组件，只允许无业务语义或弱业务语义组件。
│   │   ├── base/                /**/ 按钮、输入框、弹层、菜单等基础组件。
│   │   ├── markdown/            /**/ Markdown 渲染封装，禁止各模块自行接入不同渲染方式。
│   │   └── canvas/              /**/ 画布基础封装，隔离具体画布库 API。
│   │
│   ├── composables/             /**/ 可复用组合逻辑。跨模块 composable 必须保持通用，不夹带业务写操作。
│   ├── stores/                  /**/ 前端状态管理，只存交互状态、缓存和任务状态，不作为系统事实来源。
│   ├── api/                     /**/ 后端接口访问封装，页面和组件禁止直接拼接请求。
│   ├── types/                   /**/ 前端类型定义，服务于 UI 和接口响应，不定义后端领域模型真相。
│   ├── styles/                  /**/ 全局样式、主题变量和布局基础样式。
│   └── utils/                   /**/ 纯函数工具，禁止依赖 Vue 状态、后端 API 或业务模块。
│
├── public/                      /**/ 静态资源目录，不存用户生成内容或运行时数据。
├── tests/                       /**/ 前端测试目录，按模块和用户路径组织。
└── docs/                        /**/ 前端补充说明，禁止替代正式规范文档。
```

### 1.3 Git 与代码流 (Git Workflow)

- **分支模型**：采用轻量 feature 分支开发，主分支保持可运行。
- **提交规范**：提交信息使用清晰前缀，如 `feat:`、`fix:`、`docs:`、`refactor:`、`test:`。
- **原型同步**：涉及工作区、内容树、悬浮输入框和果实详情的交互调整，必须同步说明用户路径变化。
- **评审重点**：画布交互、状态管理、Markdown 渲染和接口边界变更必须重点评审。
- **避免混改**：UI 视觉调整、接口接入和状态重构应拆分提交，降低回滚成本。

### 1.4 命名与代码风格 (Naming & Code Style)

- **语言规范**：前端统一使用 TypeScript 和 Vue 3 Composition API。
- **组件命名**：组件名表达用户语义或 UI 语义，禁止使用 `NewComponent`、`CommonPanel` 等模糊命名。
- **模块命名**：前端模块与用户可感知模块保持一致，如 seed、generator、nutrient、workspace、growth、fruit。
- **状态命名**：本地交互状态必须明确是 UI 状态，不得命名成后端系统事实。
- **API 命名**：接口访问函数必须放在 `api/` 或模块内部的数据访问层，禁止组件里散落请求。
- **样式规范**：优先使用组件内聚样式和主题变量，禁止全局样式随意覆盖业务组件。
- **Lint 工具**：必须使用统一 ESLint 与格式化规则，禁止提交未通过检查的代码。

### 1.5 其他开发事项 (Other Conventions)

- 开发任何新页面、组件或交互前，必须先对照 [DESIGN.md](../../DESIGN.md) 确认设计风格、布局原则、组件行为和文案语气。
- 前端不直接读取本地文件，不直接访问 SQLite，不直接调用 LLM。
- 前端不拼装系统事实，种子、果实、节点、发布记录、反馈和基因状态以后端返回为准。
- Markdown 渲染必须通过统一 Markdown 组件，避免 XSS、样式不一致和渲染差异。
- 内容树画布必须通过工作区模块和 canvas 基础封装使用，禁止业务组件直接绑定第三方画布库。
- 生成中状态属于任务视图，不得伪造成已生成果实节点。
- 被淘汰果实不得从前端树中直接删除，只能按后端状态弱化显示。
- 枝化生长输入框必须围绕当前选中节点工作，不允许产生“无来源节点”的生长任务。
- 所有涉及用户确认的操作，如淘汰、发布记录、基因沉淀，都必须有清晰反馈。
- 第一期开工作台体验，不以 SEO 为目标；公开页面如后续需要 SEO，应独立规划。

## 2. 治理与维护 (Governance & Maintenance)

### 2.1 测试规范 (Testing Strategy)

- **组件测试**：基础组件、Markdown 渲染、果实详情、枝化生长输入框和状态按钮应覆盖主要交互。
- **模块测试**：种子库、生成器页面、营养库页面、工作区画布、发布验证和基因汲取模块应覆盖核心用户路径。
- **端到端测试**：至少覆盖一次完整闭环：创建种子 → 进入工作区 → 发起枝化生长 → 查看果实 → 执行物竞天择 → 记录发布 → 回填数据 → 确认基因沉淀。
- **视觉回归**：内容树节点状态、悬浮输入框和果实详情属于核心体验，重要改动应人工截图走查。
- **测试策略**：第一期优先验证核心工作流可完成，不追求组件覆盖率数字优先。

### 2.2 部署与 CI/CD (Deployment & CI/CD)

- **环境划分**：第一期至少区分本地开发环境和可演示环境。
- **构建检查**：每次合入主分支前必须通过类型检查、Lint、测试和构建。
- **配置管理**：后端地址、运行模式和公开配置必须通过环境配置管理，禁止写死本机地址。
- **资源处理**：公开静态资源放入前端静态目录，用户生成内容和运行时文件不得打包进前端。
- **发布走查**：发布前必须完成核心工作台路径走查，尤其是画布、Markdown 渲染和异步生成状态。

### 2.3 监控与告警 (Monitoring & Alerting)

- **前端错误**：运行时错误、接口错误和 Markdown 渲染错误必须可被捕获并向用户展示可理解提示。
- **任务反馈**：枝化生长和基因汲取等耗时操作必须展示明确状态，不能让用户误以为操作无响应。
- **空状态规范**：种子库为空、生成器为空、营养库为空、内容树无果实时必须提供清晰下一步入口。
- **失败恢复**：生成失败、保存失败、发布记录失败和数据回流失败时，前端必须保留用户输入并允许重试。
- **交互一致性**：所有确认、取消、加载、失败、成功反馈在视觉和文案上必须保持一致。
