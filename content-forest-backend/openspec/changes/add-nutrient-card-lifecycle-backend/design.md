## Context

现有营养库领域已经支持公共营养库、种子专属营养库、营养内容 Markdown、归档/回档和按种子查询可引用营养。第二期要求营养研究结果先以“未沉淀营养卡片”存在，允许临时引用和继续对话优化，用户确认后才沉淀为当前种子专属营养。

## Goals / Non-Goals

**Goals:**
- 在不破坏现有营养库边界的前提下，引入种子级营养卡片生命周期。
- 支持未沉淀、已沉淀、已归档三种状态。
- 支持已沉淀营养设置为“常驻营养”，每次枝化生长默认带入。
- 支持未沉淀营养作为本次生长的临时引用。
- 保证已归档营养不可引用。

**Non-Goals:**
- 不保存营养卡片到公共营养库。
- 不实现联网研究 Agent。
- 不实现缺口建议生成。
- 不实现复杂可信度评分。

## Decisions

### Decision 1: 复用营养库领域，不新增独立资料领域

营养卡片仍属于营养库领域，因为它的最终落点是种子专属营养内容。未沉淀卡片是正式营养内容之前的候选状态，不进入公共营养库。

### Decision 2: 用户展示名使用“常驻营养”，技术字段使用 `defaultForGrowth`

“常驻营养”便于用户理解：这条营养会在每次生长时默认带上，但可以在输入框中叉掉。技术字段建议为 `defaultForGrowth`，只表示默认预选，不表示后端强制引用。

### Decision 3: 未沉淀卡片通过临时引用进入枝化生长

正式营养继续使用现有 `nutrientRefs`。未沉淀卡片使用独立引用集合，避免把未确认资料伪装成正式营养，同时满足快速试验。

### Decision 4: 卡片会话只保存绑定关系

本变更只为卡片提供会话标识和加载入口，不实现 Agent 对话能力；具体对话由营养研究 Agent 变更负责。

## Risks / Trade-offs

- [Risk] 未沉淀卡片可引用会让用户误以为它是正式营养 → 在 Agent 输入和 API 响应中明确标记为临时候选。
- [Risk] 常驻营养过多导致每次生长上下文膨胀 → 前端默认带入但允许移除，后续可加数量提醒。
- [Risk] 与现有归档语义冲突 → 已归档卡片和已归档正式营养均不可引用。
