## Context

内容森林已经确立“Markdown 承载内容本体，数据库维护系统事实”的边界。种子正文是用户原始灵感的事实源，不应被简报覆盖；种子主简报是系统对种子的理解、扩写和探索地图，用来帮助用户和 Agent 更快建立上下文。

第二期规划中，种子主简报长期存在、可编辑、可刷新；本轮生长简报由枝化生长管线临时组装。本 change 只负责种子主简报的后端能力和工作区摘要，不实现枝化生长管线。

## Goals / Non-Goals

**Goals:**

- 支持用户手动为种子生成主简报。
- 支持查看、编辑、刷新种子主简报。
- 将主简报正文保存为内容本体，数据库只记录简报系统事实和内容位置。
- 工作区快照返回种子主简报摘要。
- 生成失败时返回可理解错误，不影响种子详情和枝化生长。

**Non-Goals:**

- 不在创建种子后自动生成简报。
- 不保留简报历史版本，第一版刷新采用覆盖语义。
- 不为每颗果实维护完整主简报。
- 不实现本轮生长简报组装和突变搜索策略。
- 不让简报覆盖、改写或替代种子原文。

## Decisions

### Decision 1: 主简报由用户手动生成

种子创建后不自动调用 Agent。用户在种子详情或工作区中手动触发生成主简报。

理由：简报生成会消耗 LLM 资源，也可能不是每个种子都需要。手动入口更符合当前轻量 MVP。

### Decision 2: 第一版刷新覆盖当前简报

一个种子第一版只维护一份当前主简报。刷新成功后覆盖正文和更新时间；刷新失败时保留旧简报。

理由：历史版本会引入版本浏览、回滚和差异对比，超出本期原子能力。

### Decision 3: 简报正文走内容访问层

简报内容可以是结构化 Markdown 或系统约定的文档格式，但数据库不保存正文全文，只保存 `content_location`、状态、生成来源和审计信息。

理由：延续现有种子、果实、营养、基因库的文件系统边界，避免 Markdown 和数据库 meta 混杂。

### Decision 4: 简报生成通过 AgentPort 或内置 Agent 能力

后端应用服务只负责编排输入、状态和内容落地；简报正文由 Agent 能力生成。Agent 输出必须经过后端校验后才能写入内容访问层。

理由：生成是不确定环节，系统事实和持久化必须由内容森林掌控。

### Decision 5: 工作区只返回摘要

工作区快照只返回是否存在简报、更新时间、内容位置和少量摘要字段，不把完整简报正文塞入工作区快照。

理由：工作区快照需要保持轻量；完整简报由前端按需调用种子简报详情接口读取。

## Risks / Trade-offs

- [Risk] 简报生成质量不稳定 → Mitigation: 允许用户编辑，刷新失败保留旧版本。
- [Risk] 简报结构过度产品宣传化 → Mitigation: 简报生成提示必须采用通用创作地图，不固定为产品/卖点模型。
- [Risk] 工作区快照膨胀 → Mitigation: 只返回摘要，不返回完整正文。
- [Risk] Agent 输出污染系统事实 → Mitigation: 后端校验并只写内容本体，meta 由系统生成。

## Migration Plan

1. 先更新 `docs/sql/seed.sql` 与 `docs/api/seed.yaml`、`docs/api/workspace.yaml`。
2. 增加种子主简报存储和内容访问层写入能力。
3. 增加简报生成、查看、编辑、刷新应用能力。
4. 将简报摘要并入工作区聚合。
5. 补充测试并运行 OpenSpec 严格校验。

回滚时可停止暴露简报接口，已有简报文件和记录不影响种子、果实、生长任务读取。
