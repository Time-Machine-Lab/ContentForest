## Context

内容森林后端已经具备种子、生成器、果实和 Agent Core 基础能力。果实模块可以提供果实详情、基因标签和选择状态；Agent Core 已支持 `AgentPort.runTask`、`gene_extraction` 任务类型、Skill Runtime、Tool Runtime、输出校验和 Trace。当前 `src/modules/gene` 仍为空壳，尚未把“果实选择/淘汰与后续反馈如何沉淀为基因经验”落到后端能力中。

本设计基于 `docs/design/domain/基因汲取领域模块设计文档.md`、`docs/design/内容森林Agent架构设计文档.md` 和 `docs/内容森林第一期开发规划文档.md`。模块应遵守“Markdown 保存内容本体，数据库保存系统事实”的存储原则。基因建议是待确认草稿，轻量持久化在数据库；基因经验是用户确认后的正式资产，正文写入种子级基因库 Markdown，meta 写入数据库。

## Goals / Non-Goals

**Goals:**

- 实现基因汲取后端模块，覆盖种子级基因库、汲取提醒、汲取任务、基因建议和基因经验。
- 在种子创建后保证对应种子级基因库可用。
- 支持基于果实选择/淘汰创建第一版轻量汲取提醒。
- 支持用户从提醒或证据来源人为触发基因汲取任务。
- 调用 AgentPort 的 `gene_extraction` 任务能力生成基因建议。
- 将 Agent 生成的基因建议持久化为待确认草稿，支持查看、编辑、确认和放弃。
- 确认基因建议后，写入正式基因经验，并将正文保存到 `data/content/genes/seed-scoped/` 下的种子级基因库目录。
- 支持查询某个种子的可引用基因经验，供后续枝化生长作为基因上下文。
- 支持证据来源和证据强度表达，第一版允许选择/淘汰作为弱证据。
- 补齐 `docs/api/gene.yaml` 与 `docs/sql/gene.sql` 顶层契约文档。

**Non-Goals:**

- 不实现复杂遗传算法、自动繁殖策略或统一适应度评分。
- 不实现跨种子共享基因库。
- 不实现全自动汲取或自动确认基因经验。
- 不将基因建议写入 Markdown；只有确认后的基因经验才写 Markdown。
- 不把基因经验写入营养库。
- 不自动改写生成器。
- 不直接实现发布验证和数据回流模块；只预留发布记录与反馈快照作为后续可接入证据来源。
- 不实现复杂任务队列、分布式调度或后台 Worker。

## Decisions

### Decision 1: 基因建议与基因经验分离

Agent 生成结果先落为“基因建议”，用户确认后才转为“基因经验”。基因建议是待确认草稿，只保存到数据库；基因经验是正式进化资产，使用 Markdown 保存正文，并由数据库维护系统事实。

选择该方案的原因是 Agent 输出不能直接成为系统经验。持久化建议可以支持刷新不丢、编辑后确认、放弃建议和避免重复汲取；确认后再写 Markdown 则能保持内容本体目录只保存正式资产。

备选方案是不持久化建议，Agent 结果只返回前端。该方案实现更快，但用户刷新后丢失建议，也无法追踪某次提醒是否已经生成过建议。另一个备选方案是建议也写 Markdown，但会让临时中间态污染基因库文件结构，因此不采用。

### Decision 2: 基因库随种子准备，但内容由基因模块维护

每个种子都拥有种子级基因库。基因模块提供为种子准备基因库的能力，并负责后续基因经验的创建、查询和归档语义。种子模块只需要在创建种子后协同触发该能力，或者由装配层在种子创建流程中调用。

备选方案是让种子模块直接创建基因库系统事实。该方案会让种子领域侵入基因库内容边界，不利于后续基因模块独立演进。

### Decision 3: 轻量汲取提醒作为真实业务对象

第一期实现轻量汲取提醒。提醒由果实选择/淘汰等领域消息或应用服务协作创建，用于告诉用户“这里可能值得汲取”。提醒不会自动执行 Agent，也不会自动生成正式基因经验。

选择该方案是为了保留“人为触发 + 系统提醒”的产品闭环。完全不做提醒会让基因汲取只剩手动入口，难以体现系统对进化机会的捕捉。

备选方案是直接在用户选择果实时自动调用 Agent。该方案会增加成本和不确定性，也违背用户确认边界，因此不采用。

### Decision 4: 第一版允许选择/淘汰作为弱证据

第一版基因汲取允许只基于果实选择或淘汰触发。此类证据属于弱证据，代表用户判断，而不是平台真实反馈。后续发布记录和数据快照接入后，可作为中证据、强证据和趋势证据。

这样可以在发布验证和数据回流模块尚未完全可用时先跑通基因汲取闭环，同时为后续真实数据增强保留语义空间。

### Decision 5: 基因谱系和生态位由 Agent 建议、用户确认

Agent 在生成基因建议时可以给出谱系和生态位建议，用户在确认前可编辑。确认后的谱系和生态位成为基因经验 meta 的一部分。

备选方案是要求用户完全手动填写谱系。该方案更可控但操作负担重，不符合 AI 杠杆的产品定位。另一个方案是 Agent 自动决定且用户不可改，会削弱人的判断力，因此不采用。

### Decision 6: 基因汲取任务保持业务批次语义

汲取任务用于记录一次用户触发的 Agent 分析过程，状态保持最小集合：进行中、已完成、失败。任务失败允许重试，但不建设通用任务系统。

选择该方案可以为前端提供用户可感知的运行状态和失败提示，同时避免把基因汲取做成复杂任务调度平台。

### Decision 7: Agent 只生成建议，后端负责校验与落地

基因模块调用 AgentPort 的 `gene_extraction` 能力。Agent 只返回建议内容、谱系建议、生态位建议和证据理解。后端应用服务负责校验输出、保存建议、确认后写入 Markdown 和数据库。

这延续 Agent 架构文档的边界：Agent 不写文件、不写数据库、不修改果实、发布记录或反馈快照。

### Decision 8: 基因经验支持编辑与归档，不支持删除

确认后的基因经验允许编辑正文和归档。归档后的基因经验不应作为新的枝化生长引用上下文，但仍可查看和追溯。第一期不提供硬删除能力。

该方案与种子、果实、营养库等模块“不轻易删除系统资产”的设计一致，有利于保留内容进化历史。

## Risks / Trade-offs

- [Risk] 只基于选择/淘汰会产生主观偏差 → Mitigation：明确标记为弱证据，后续接入发布验证和数据快照作为更强证据。
- [Risk] 基因建议持久化增加对象和状态复杂度 → Mitigation：第一期只保留待确认、已确认、已放弃三类建议状态，避免复杂审批流。
- [Risk] Agent 输出结构不稳定 → Mitigation：基因模块对 Agent 输出做应用层校验；无法形成可用建议时任务失败并返回可理解错误。
- [Risk] 正式基因经验写文件与数据库可能出现不一致 → Mitigation：遵循现有内容访问层模式，先写内容本体再写系统事实，失败时清理已创建内容文件。
- [Risk] 领域消息基础设施尚未完整 → Mitigation：第一期可先通过应用服务显式调用创建提醒，保持领域语言为“汲取线索/领域消息”，后续再替换为事件监听器。
- [Risk] 发布验证和数据回流模块仍在推进 → Mitigation：第一期以果实选择/淘汰为可用证据，发布和反馈证据通过端口或预留关系后续接入。
- [Risk] 基因谱系过早复杂化 → Mitigation：第一期将谱系作为轻量组织信息，不实现复杂遗传算法和自动谱系优化。

## Migration Plan

本次为新增后端能力，不迁移已有业务数据。实现顺序建议先补齐 `docs/api/gene.yaml` 和 `docs/sql/gene.sql`，再创建基因模块、存储端口、内容访问端口、适配器、控制器和测试。

如果需要回滚，可移除新增 gene 模块、gene 控制器、gene SQL 迁移、基因 Markdown 内容访问适配器和相关 OpenAPI/SQL 契约，不影响现有种子、果实、生成器和 Agent Core 能力。

## Agent Skill Boundary Clarification

基因汲取不依赖用户上传的生成器 Skill。第一期应实现一个 Agent 系统内置的基因汲取 Skill，并在 Agent Runtime 装配阶段注册。基因领域服务只调用通用 `AgentPort.runTask`，传入 `gene_extraction` 任务；不为基因汲取新增专用 AgentPort 方法。

如果当前分支尚未提供 `gene_extraction` 任务类型，实现时需要先补齐任务类型；如果已经存在，则只需要补齐内置 Skill、输入输出契约、运行时注册和必要的只读工具。该 Skill 可以通过工具读取果实正文、种子正文和已确认基因经验，但不得直接写数据库或写文件。Agent 的职责仍然只到“生成基因建议”，建议持久化、确认、写入 Markdown 和状态流转都由基因领域服务完成。
