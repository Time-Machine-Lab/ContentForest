## Context

内容森林后端已经具备种子、生成器、果实与 Agent Core 基础能力。果实模块已经提供内部 `createFruitFromCandidate`、按父节点查询子果实和果实生长源引用能力；Agent Core 已提供 `AgentPort.runTask`、任务类型、Skill Runtime、Tool Runtime、输出校验与运行追踪底座。枝化生长 Skill、Agent 所需 Tool、枝化生长入口和输出契约由 Agent 侧变更提供，本变更只负责对接这些能力。

当前缺口是枝化生长领域模块仍为空目录，工作区无法从种子或果实发起生长，也无法获得生长任务状态、节点生长锁和失败重试输入。该设计基于 `docs/design/domain/枝化生长领域模块设计文档.md`、`docs/design/workbench/内容森林工作区业务设计文档.md` 和 `docs/design/内容森林Agent架构设计文档.md`。

## Goals / Non-Goals

**Goals:**

- 实现枝化生长领域模块，支持从种子或果实发起一次生长批次。
- 创建并维护生长任务，状态限定为生成中、已完成、失败。
- 对来源节点加生长锁，防止同一节点重复生长，同时不影响其他节点。
- 校验来源节点、生成器、果实数量和引用资源的可用性。
- 将枝化生长输入转换为 AgentPort 的 `growth` 任务上下文与授权包。
- 当果实数量为 N 时，创建一个生长任务，并执行 N 次内部果实生成尝试；第一期 N 最大为 6。
- 接收 Agent 候选结果，并调用果实模块内部能力创建候选果实、挂载到来源节点。
- 支持部分成功不回滚；至少一个果实创建成功即任务完成，零成功才失败。
- 支持保存该节点最近一次失败任务的输入，用于前端重试恢复。
- 提供生长任务状态查询能力，供前端轮询。
- 提供节点是否正在生长的查询能力，供工作区视图聚合展示。
- 新增 `docs/api/growth.yaml` 与 `docs/sql/growth.sql` 顶层契约文档。

**Non-Goals:**

- 不实现突变概率参数，第一期不传给 Agent。
- 不实现杂交、高级生长策略、多模型对比或自动选择生成器。
- 不实现复杂任务队列、分布式调度、后台 Worker 或并发批量调度。
- 不实现工作区整棵树查询接口；该能力属于工作区查询服务。
- 不实现枝化生长 Skill、Agent Tool 或 AgentPort 枝化生长入口；本变更只向 AgentPort 传递授权范围并消费其输出。
- 不直接读写 Markdown 文件，不直接调用 LLM SDK。
- 不改变果实领域选择状态、发布验证或数据回流规则。

## Decisions

### Decision 1: 一个用户提交对应一个生长任务批次

用户在工作区点击生成时，后端创建一个生长任务，而不是为每个果实创建用户可见任务。果实数量只影响该任务内部的生成尝试次数。

选择该方案的原因是用户心智中“一次点击生成”就是一次枝化生长批次，前端也只需要轮询一个任务状态。内部尝试用于记录每个候选果实的成功或失败，不作为独立用户操作对象暴露。

备选方案是为数量 N 创建 N 个任务。该方案更细粒度，但会让状态展示、失败恢复和重试语义复杂化，也容易把枝化生长变成通用任务系统。

### Decision 2: 第一期开启串行果实生成尝试

数量为 5 时，生长任务内部按顺序执行 5 次 AgentPort 调用，每次目标是返回 1 个候选果实。

选择串行的原因是第一期更容易控制生长锁、Agent 调用失败、部分成功统计和果实创建顺序。后续如果需要提升速度，可在保持任务语义不变的前提下引入小并发。

备选方案是一次 Agent 调用返回多个候选果实。该方案调用次数少，但更难控制单个候选失败，也更容易出现“全部内容风格过于相似”的结果。第一期用多次尝试更贴近“多颗果实分别生长”的业务隐喻。

### Decision 3: 生长锁归属于枝化生长模块

来源节点是否正在生长由枝化生长模块通过进行中的任务或生长锁判断。果实模块和种子模块不维护该状态。

选择该方案是因为生长锁是枝化生长过程约束，不是果实或种子的固有属性。工作区如果需要显示 `isGrowing`，应通过查询服务从枝化生长状态派生。

备选方案是在果实或种子系统事实上保存是否正在生长。该方案查询简单，但会造成状态归属混乱，也更容易出现锁释放不一致。

### Decision 4: Growth 模块只对接 AgentPort 输入输出契约

枝化生长模块不会把所有 Markdown 全文直接塞给 AgentPort。它向 AgentPort 传递 `growth` 类型任务、来源节点引用、用户输入、生成器引用、营养引用、基因引用、当前种子范围和本次尝试目标。Agent 后续通过受控 Tool 在授权范围内读取内容。

选择该方案是为了对齐 Agent 架构文档中的“Tool 受控读取”和“任务授权优先”。枝化生长模块负责校验与授权，Agent 负责按需读取，Tool 负责受控取内容。

备选方案是后端预先拼接完整 Prompt。该方案实现直接，但会造成上下文不可控，也不利于后续引用资源扩展。

### Decision 5: 引用资源由后端校验，Agent 通过 Tool 读取

前端负责 `@` 引用交互并提交引用标识，枝化生长模块负责校验引用是否属于当前种子可访问范围。AgentPort 只接收授权后的引用范围，不接收真实文件路径。

这样可以保证前端体验灵活，同时后端仍是资源权限的最终校验方。Agent 不应因为前端传入了某个引用就自动获得读取权限。

### Decision 6: 任务完成/失败基于成功创建果实数量判定

Agent 返回候选结果后，枝化生长模块调用果实模块创建果实。只有果实模块成功创建并挂载后，该尝试才算成功。任务最终状态由成功创建的果实数量决定：至少一个成功为已完成，零成功为失败。

选择该方案是因为内容森林真正需要的是“果实进入内容树”，而不是 Agent 产生了文本。Agent 输出未能落成果实，不应算成功。

### Decision 7: 失败输入只恢复该节点最近一次失败任务

当一个来源节点最近一次生长任务失败且没有任何果实成功时，系统保留该任务的输入、生成器、数量和引用，供工作区点击该节点时恢复。

选择该方案可以满足第一期重试体验，同时避免建设完整历史版本恢复。后续如需多次失败历史，可在该语义上扩展。

### Decision 8: 轮询接口属于应用查询能力，不属于领域规则

前端轮询生长状态是交互策略。后端提供任务状态查询能力和节点生长状态查询能力，但领域文档不把“轮询”作为枝化生长规则。

第一期前端提交生长后获得任务标识，轮询任务状态；任务完成后刷新工作区树。工作区树视图由工作区查询服务聚合，枝化生长模块只提供必要状态来源。

## Risks / Trade-offs

- [Risk] 串行执行多个果实生成尝试较慢 → 第一期开工优先稳定闭环；后续可在任务语义不变的情况下引入小并发。
- [Risk] Agent 侧枝化生长能力并行实现，输入输出契约可能变化 → 本变更必须把 Agent 对接契约作为前置任务，并在 Growth 应用层做输出适配和校验，避免直接信任 Agent 内容。
- [Risk] 生长锁释放失败会导致节点永久不可生长 → 任务完成、失败和异常路径都必须释放锁，并通过测试覆盖。
- [Risk] 部分成功时用户可能不知道内部失败 → 第一期只展示成功生成的果实，内部失败进入日志和任务详情；后续可增强提示。
- [Risk] 引用资源校验依赖营养库和基因库能力成熟度 → 第一期开工可先定义端口和测试替身，待相关模块完成后接入真实校验。
- [Risk] 任务状态查询与工作区树查询边界混淆 → 生长模块只暴露生长任务和节点锁状态，整棵树聚合留给工作区查询服务。

## Migration Plan

本次为新增后端能力，不需要迁移已有用户数据。实现时先补齐 `docs/api/growth.yaml` 和 `docs/sql/growth.sql`，再创建模块骨架、存储适配器、应用服务、控制器和测试。

如果需要回滚，可移除新增 growth 模块、growth 控制器、growth SQL 迁移和 OpenAPI 契约，不影响现有种子、生成器、果实和 Agent Core 能力。

## Open Questions

- 营养库与基因库真实后端能力完成前，引用资源校验是否先用端口替身接入。
- 工作区树视图是否在后续单独变更中实现 `isGrowing` 聚合，还是在本变更中仅提供节点生长状态查询供其消费。
