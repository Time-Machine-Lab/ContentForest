## Context

内容实验校准领域已经在 `docs/design/domain/内容实验校准领域模块设计文档.md` 中定义了完整闭环：预测图、预测快照、校准复盘、基因汲取证据和评分画像演进。本次只落地第一片能力：果实评测员生成“当前预测图”。它服务于用户点击任意果实时理解强点、风险、预期区间和核心赌注，不要求果实已被选择或即将发布。

现有后端已经具备果实详情、物竞天择、发布记录、反馈快照、基因汲取和 Agent Runtime 基础能力。预测图应作为新的系统事实保存，不能写入果实 Markdown，也不能改变果实选择、发布或基因汲取状态。Agent 仍只负责生成建议，领域服务负责校验、持久化和对外返回。

`cheat-on-content` 中可借鉴的部分是 `cheat-score` 的结构化评分输出与 `cheat-score-blind` 的盲评隔离边界：评测只读取当前稿件与评分规则，不读取发布后表现、复盘或历史预测污染源。本项目不会照搬其 `.cheat-state.json`、命令行 hooks、文件缓存或 slash command 形态，而是改造成 ContentForest 的领域服务、Agent Skill、Tool 和 SQL/API 契约。

## Goals / Non-Goals

**Goals:**

- 为任意果实生成、刷新和读取当前预测图。
- 保存默认评分画像和画像版本，记录预测图使用的判断规则来源。
- 通过单一内容实验校准 Controller 暴露预测图 API。
- 通过两张 SQL 表保存评分画像与果实预测图。
- 让果实评测 Agent 遵守盲评边界，只读取授权果实正文和评分画像。
- 用本地 Schema 校验 Agent 输出，只有通过校验的结构化结果才能落地为预测图。

**Non-Goals:**

- 不冻结预测快照，不创建发布基线。
- 不做发布后校准复盘，不生成基因汲取证据。
- 不自动修订评分画像。
- 不模拟平台算法，不构造虚拟用户群，不承诺真实平台表现。
- 不在工作区快照中内嵌预测图详情；前端按果实详情需要单独读取。

## Decisions

1. **预测图作为独立领域模块，而不是扩展果实表字段**

   预测图属于内容实验校准领域，承载评测判断、核心赌注、反事实场景、观察指标和盲评状态。将其放入独立 `fruit_prediction_maps` 表可以避免果实领域被实验判断污染，也便于后续扩展预测快照、复盘和校准样本。

   替代方案是在 `fruits` 表增加 JSON 字段。该方案实现更快，但会把可刷新评测结果混入果实系统事实，后续冻结快照和校准复盘会难以分层。

2. **评分画像先落默认版本，暂不提供画像管理 UI/API**

   MVP 需要记录“这次评测是按哪套规则判断的”，但用户暂时不需要管理多套画像。后端提供默认 `evaluation_profiles` 记录与版本字段，预测图保存 `profileId` 与 `profileVersion`，为后续校准样本和画像演进留出稳定锚点。

   替代方案是把评分规则写死在代码里。该方案无法解释版本来源，也不利于后续校准。

3. **生成和刷新使用同一个命令语义**

   `EvaluateFruit` 与 `RefreshPredictionMap` 在 MVP 中都生成一张新的当前预测图，并将其标记为该果实最新预测图。刷新不会修改历史记录，而是新增记录并切换最新视图。

   替代方案是原地覆盖预测图。该方案更简单，但会丢失用户比较不同稿件版本或不同画像版本判断的可能。

4. **Agent Skill 只产出结构化候选，领域服务负责落地**

   果实评测 Agent 返回预测图候选结构；应用服务执行 Schema 校验、盲评状态校验、果实存在性校验和持久化。这与现有枝化生长、基因汲取的 Agent 边界一致。

   替代方案是让 Agent 直接写数据库。该方案破坏现有 Agent 只读 Tool 和建议输出边界。

5. **盲评边界在 Tool 与任务输入双层约束**

   评测任务输入只包含 `fruitId`、目标画像和刷新原因等必要信息；只读 Tool 仅允许读取该果实 Markdown 与评分画像。Tool 明确拒绝读取发布记录、反馈快照、校准复盘和任何真实表现数据。

   替代方案是依赖 prompt 提醒 Agent 不要读取污染数据。该方案不够可靠，不能满足领域文档中的盲评边界。

## Risks / Trade-offs

- [预测图看起来像结果承诺] -> API 字段必须包含 `confidence`、`confidenceReason` 与非承诺边界，Agent 输出不得写成确定性平台预测。
- [历史锚点不足导致评测过度自信] -> MVP 允许 `historicalAnchors` 为空，但必须在盲评状态或置信度中表达锚点不足。
- [Agent 输出结构不稳定] -> 使用本地 Schema 校验与有限自检修复，失败时返回可理解错误，不落地半结构化内容。
- [前后端 change 并行导致契约漂移] -> 后端任务必须先更新 `docs/api/` 与 `docs/sql/`，前端任务只能消费这些文档。
- [后续快照/复盘需要迁移] -> 当前表保留 `profileVersion`、`blindState`、`generatedAt` 和完整结构 JSON，后续可以从最新预测图冻结为快照而不改写历史图。

## Migration Plan

1. 先新增顶层 API/SQL 文档。
2. 新增数据库迁移与默认评分画像种子数据。
3. 实现领域模型、Repository、应用服务与 Controller。
4. 注册 Agent Runtime 的 `fruit_evaluation` 任务类型、Skill 和只读 Tool。
5. 增加单元测试、应用服务测试、Controller 契约测试和 Agent Skill 结构化输出测试。
6. 回滚时可关闭 Controller 与 Skill 注册；已生成预测图数据保留，不影响果实、发布、反馈和基因模块。

## Open Questions

- 默认评分画像的首版维度命名是否直接采用 `cheat-on-content` 的 rubric 缩写，还是用 ContentForest 领域中文维度重命名后再映射到内部字段？
- 预测图是否需要在工作区快照中返回轻量摘要，本次先不做，后续根据前端性能和使用频率再评估。
