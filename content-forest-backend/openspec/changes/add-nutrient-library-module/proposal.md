## Why

内容森林第一期的枝化生长需要能够引用平台指南、爆款案例、产品资料和种子专属知识，但当前后端尚无营养库领域能力来组织这些 Markdown 资料并提供受控引用。该变更依据 `docs/design/domain/营养库领域模块设计文档.md`、`docs/design/内容森林架构设计文档.md` 和 `docs/内容森林第一期开发规划文档.md`，实现营养库作为轻量资料库的后端能力。

## What Changes

- 新增营养库模块后端能力：创建公共营养库、创建种子专属营养库、查看列表和详情、编辑名称与非必填描述、归档和回档营养库。
- 新增营养内容后端能力：在营养库中新增 Markdown 内容、查看详情、编辑 Markdown 正文、归档和回档营养内容。
- 新增营养库作用域规则：公共营养库可被任意种子引用；种子专属营养库必须绑定一个种子，且只允许归属种子引用。
- 新增枝化生长引用支撑能力：提供可引用营养查询，并校验营养库和营养内容均未归档且作用域匹配。
- 新增营养内容文件存储能力：Markdown 正文写入运行时内容目录，数据库只维护系统事实、状态、归属和相对内容位置。
- 新增营养库模块所需的 API 与 SQL 顶层设计文档要求：在实现前必须创建或更新 `docs/api/nutrient.yaml` 与 `docs/sql/nutrient.sql`。
- 不引入向量检索、知识库问答、自动网页采集、跨用户共享市场、Agent 自动判断资料有效性或基因汲取结果写入营养库。

## Capabilities

### New Capabilities

- `nutrient-library-management`: 定义营养库和营养内容的创建、查看、编辑、归档、回档、作用域校验、Markdown 内容存储和枝化生长引用支撑能力。

### Modified Capabilities

无。

## Impact

- 影响后端业务模块：新增 `nutrient` 领域模块及其应用服务能力。
- 影响运行时文件系统：需要将营养内容 Markdown 保存到 `data/content/nutrients/public/` 和 `data/content/nutrients/seed-scoped/` 下，并通过内容访问层读取。
- 影响内容访问层：需要支持营养内容 Markdown 保存、读取和替换，且只对上层暴露相对内容位置。
- 影响存储层：需要维护营养库、营养内容、作用域、归档状态、种子归属和内容本体位置；具体结构必须先落到 `docs/sql/nutrient.sql`。
- 影响 API 契约：需要提供营养库与营养内容的创建、列表、详情、编辑、归档、回档和可引用营养查询接口；具体契约必须先落到 `docs/api/nutrient.yaml`，并遵守“一份 `.yaml` 对应一个 Controller 文件”。
- 影响后续模块：枝化生长模块和 Agent 只读 Tool 可依赖营养库模块获取被授权的可引用营养内容。
