## Context

当前基因汲取模块已经具备种子级基因库、汲取提醒、汲取任务、待确认建议、正式基因经验和可引用经验查询能力。它解决了“把一次选择、淘汰或反馈沉淀为经验”的问题，但基因库仍偏静态：基因经验一旦确认后，系统只能把它作为平铺上下文提供给后续枝化生长，缺少持续跟进、表现回看、评优和迭代机制。

内容森林的核心逻辑是进化论、遗传模拟和复利效应。若基因只保存一次性结论，就无法判断某条基因在后续使用中是否真的有效，也无法比较同一种子下不同谱系的长期表现。因此，本变更把基因库升级为种子级进化资产：它仍保持轻量，但开始记录基因在后续生成和反馈中的使用痕迹与表现汇总。

本变更必须遵守现有边界：

- 基因库仍是种子级能力，不做跨种子自动共享。
- Markdown 仍只保存基因经验正文，不保存数据库维护的 meta 信息。
- Agent 仍只生成建议，不直接写数据库、不直接改写基因经验。
- 第一阶段不引入重型实验平台、向量数据库、消息队列或自动适应度服务。

## Goals / Non-Goals

**Goals:**

- 为正式基因经验增加可持续跟进的使用记录与表现汇总。
- 支持按种子、谱系和基因经验查看长期表现、排序和回看。
- 让枝化生长和基因汲取 Agent 能消费更有权重感的基因上下文。
- 保留多谱系并存，不因单次反馈自动删除、覆盖或禁用基因。
- 更新顶层设计、API、SQL 和后端实现，保证文档契约与代码一致。

**Non-Goals:**

- 不实现自动发布、自动抓取全平台数据或自动判定爆款。
- 不实现独立实验平台、AB 测试平台或复杂多臂老虎机算法。
- 不让系统自动确认、删除、改写正式基因经验。
- 不把基因库混入营养库，也不做跨种子全局基因市场。
- 不要求前端在本变更中完成复杂可视化，只提供后端能力与契约支撑。

## Decisions

### 1. 使用“基因使用记录 + 基因表现汇总”承载持续跟进

基因持续跟进拆成两层：使用记录保存某条基因在一次枝化生长或后续验证中的具体使用与结果，表现汇总保存系统可快速读取的累计统计。

Alternative considered: 只在 Markdown 中追加复盘笔记。该方案可读性好，但会让 Markdown 承担系统事实，破坏 meta 与内容分离原则，也不利于排序和查询。

### 2. 表现评优采用轻量统计，不做统一适应度黑箱

第一期只记录使用次数、正向结果、负向结果、中性结果、最近表现和轻量评分。评分用于排序和提示，不代表绝对真理，也不会自动驱动删除或禁用。

Alternative considered: 引入完整遗传算法适应度函数。该方案过重，且当前人工发布与人工反馈数据不足，容易造成伪精确。

### 3. 谱系是长期并存的演化分支，不是互斥分类

同一种子下允许多条谱系长期并存。系统可以比较和排序谱系表现，但不能因为某个谱系阶段性较弱而自动淘汰整个谱系。

Alternative considered: 每个种子只维护一个最优基因集合。该方案会压扁内容方向，无法表达用户提到的“情绪价值方向”和“小众精品方向”同时有效的情况。

### 4. Agent 输入消费“演化摘要”，不直接承担统计计算

基因汲取和枝化生长 Agent 可以读取基因经验及其表现摘要，用于判断继承、强化、变异或规避。但统计更新、归属维护和排序逻辑由后端领域服务完成。

Alternative considered: 让 Agent 每次自行分析所有历史反馈。该方案成本高、不可控，也容易让 Agent 越过系统事实边界。

### 5. API 以查询和记录为主，避免暴露算法内部

Gene Controller 增加面向种子级基因库演化的查询能力，以及记录基因使用结果的能力。接口表达业务语义，不暴露具体评分公式，便于后续替换算法。

Alternative considered: 直接暴露评分字段写入接口。该方案会让前端或外部调用者污染系统评优事实。

## Risks / Trade-offs

- [Risk] 早期数据量小，评分可能误导用户 → Mitigation：把评分定位为轻量排序提示，并同时展示使用次数和正负向结果。
- [Risk] 增加 SQL 与存储复杂度 → Mitigation：只新增使用记录和表现汇总两类结构，不引入事件溯源或实验平台。
- [Risk] 生成时过度偏向高分基因导致探索不足 → Mitigation：基因表现只作为高价值上下文，不替代用户输入、生成器、营养库和突变探索。
- [Risk] 负向反馈被误解为全局禁用规则 → Mitigation：保留生态位、谱系和证据来源语义，负向表现只在适用范围内参与评优。

## Migration Plan

1. 更新 `docs/sql/gene.sql`，新增基因使用记录和表现汇总结构。
2. 更新 `docs/api/gene.yaml`，新增种子基因演化摘要查询和基因使用结果记录契约。
3. 扩展领域类型、存储端口和存储适配器。
4. 扩展 GeneService，支持记录使用结果、更新表现汇总、查询种子基因库演化摘要。
5. 扩展 Agent 输入中的可引用基因上下文，让 Agent 能读取表现摘要。
6. 补充测试，覆盖内存存储、SQLite 存储、服务行为和 Agent 输入。

Rollback strategy: 若实现出现问题，可回退新增 API 与新表使用逻辑；既有基因库、建议、经验、提醒和任务流程不受影响。

## Open Questions

- 后续“爆款”的判定是否由用户手动标注、数据阈值规则、还是平台监控器共同决定？
- 基因评分后续是否需要支持按平台、内容类型或目标分拆多个视图？
- 当某条基因表现长期变差时，是推荐归档、推荐分叉，还是推荐重写正文？
