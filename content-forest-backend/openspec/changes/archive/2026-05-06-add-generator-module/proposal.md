## Why

内容森林第一期的枝化生长必须选择一个生成器作为内容创作方法论输入，但当前后端尚无生成器领域能力来导入、管理和提供生成器 Skill。该变更依据 `docs/design/domain/生成器领域模块设计文档.md`、`docs/design/内容森林架构设计文档.md` 和 `docs/内容森林第一期开发规划文档.md`，实现生成器作为外部 Skill 资源的导入、查看、启停、重新上传和生长选择支撑能力。

## What Changes

- 新增生成器模块后端能力：导入生成器 zip、校验 Skill 本体、维护生成器系统 meta、查看列表和详情、启用/停用生成器、重新上传生成器 Skill。
- 新增生成器运行时内容存储能力：上传 zip 后解压到运行时文件系统的生成器内容目录，数据库只维护系统事实和相对内容位置。
- 新增生成器与枝化生长的协作能力：仅启用生成器可被新的枝化生长选择；生成器领域提供可选生成器和 Skill 本体位置，不执行生成器。
- 新增生成器模块所需的 API 与 SQL 顶层设计文档要求：在实现前必须创建或更新 `docs/api/` 下对应 Controller 的 OpenAPI 文档，以及 `docs/sql/` 下对应数据库表结构文档。
- 不引入生成器市场、评分、付费、在线编辑、完整版本管理、输出结构校验、`generator.json` 强制契约或 Agent 执行能力。

## Capabilities

### New Capabilities

- `generator-management`: 定义生成器 Skill 的 zip 导入、合法性校验、启用/停用、重新上传、列表详情、内容位置边界和枝化生长选择支撑能力。

### Modified Capabilities

无。

## Impact

- 影响后端业务模块：新增 `generator` 领域模块及其应用服务能力。
- 影响运行时文件系统：需要将生成器 Skill zip 解压保存到 `data/content/generators/` 下，并通过内容访问层读取。
- 影响内容访问层：需要支持生成器 Skill 文件夹保存、替换和读取，且只对上层暴露相对内容位置。
- 影响存储层：需要维护生成器系统事实、启用状态和内容本体位置；具体结构必须先落到 `docs/sql/*.sql`。
- 影响 API 契约：需要提供生成器导入、列表、详情、启用、停用、重新上传和可选生成器查询等应用接口；具体契约必须先落到 `docs/api/*.yaml`，并遵守“一份 `.yaml` 对应一个 Controller 文件”。
- 影响后续模块：枝化生长模块可依赖生成器模块获取启用生成器列表，并校验所选生成器是否可用于新的生长流程。
