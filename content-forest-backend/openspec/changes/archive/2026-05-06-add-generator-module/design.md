## Context

内容森林第一期需要在枝化生长中使用“生成器”作为内容创作方法论输入。生成器本质是外部可复用的 Skill 文件夹，而不是内容森林私有模板；它可以被上传到内容森林，也应当可以脱离内容森林交给其他 Agent 使用。

当前架构已经明确采用“内容本体文件化，系统事实数据库化”的混合存储方式：生成器 Skill 本体保存到运行时文件系统的生成器目录，数据库只维护生成器身份、展示信息、启用状态和相对内容位置等系统事实。内容访问层负责隔离本地文件系统，业务模块不得直接拼接绝对路径或操作文件。

本变更面向后端生成器领域模块，重点支撑生成器导入、查看、启停、重新上传和为枝化生长提供可选生成器。生成器领域不执行生成器，不调用 Agent，不约束生成器输出，也不负责封装果实。

## Goals / Non-Goals

**Goals:**

- 支持用户通过 zip 导入生成器 Skill，并在导入时补充生成器名称和描述。
- 校验生成器 Skill 的最低合法性：解压后的 Skill 本体必须包含可读取的 `SKILL.md`。
- 将生成器 Skill 解压保存到运行时内容根目录下的生成器内容目录，并只向上层暴露相对内容位置。
- 维护生成器在内容森林中的系统事实，包括展示信息、启用状态、内容位置和必要追溯信息。
- 支持生成器列表、详情、启用、停用和重新上传。
- 为枝化生长提供“可选择的启用生成器”和读取 Skill 本体的协作能力。
- 在正式实现前补齐生成器模块对应的 API 契约文档和 SQL 契约文档。

**Non-Goals:**

- 不实现生成器市场、评分、付费、分享分发或远程安装。
- 不实现在线编辑 Skill 文件夹内容。
- 不要求生成器包含 `generator.json` 或内容森林专用 manifest。
- 不约束生成器输出为结构化数据，也不解析生成器输出内容类型。
- 不执行生成器、不调用 Agent、不封装果实、不提取基因标签。
- 不在生成器领域实现“枝化生长必须选择生成器”的业务规则；该规则属于枝化生长领域。
- 第一期不实现完整版本管理和历史版本恢复。

## Decisions

### 决策一：导入形态只支持 zip，运行时保存为 Skill 文件夹

生成器上传入口第一期只接收 zip。后端接收后通过内容访问层完成解压、路径校验和保存，最终在运行时内容目录中形成可读取的 Skill 文件夹。

选择 zip 的原因是：它比直接上传文件夹更容易通过 HTTP 接口表达，也便于保留 Skill 文件夹内的附件、示例和辅助资料。直接上传文件夹、远程仓库导入和 Skill 市场安装都留给后续阶段。

### 决策二：合法 Skill 只强制要求 `SKILL.md`

生成器是外部 Skill，不应强制要求内容森林专用 manifest。导入和重新上传时，系统只验证解压后的 Skill 本体存在可读取的 `SKILL.md`。

这样可以保持生成器与内容森林解耦：生成器专注表达内容创作方法论，内容森林维护自己的系统事实，枝化生长负责把生成器输出封装成果实。

### 决策三：系统事实进入数据库，Skill 本体保持原样文件化

生成器名称、描述、启用状态、内容位置和追溯关系属于内容森林系统事实，必须由数据库维护。Skill 文件夹只保存生成器本体内容，不写入由数据库维护的 meta 信息。

该决策延续架构文档中的混合存储原则，避免 Markdown 或 Skill 文件成为第二个系统事实来源。

### 决策四：生成器模块通过内容访问层处理文件，不直接访问本地路径

生成器模块只依赖内容访问端口保存、替换和读取 Skill 本体。绝对路径解析、zip 解压、路径越界拦截和本地文件系统细节都限制在内容访问适配器内部。

这样可以保证后续从本地文件系统迁移到对象存储或远程内容服务时，生成器业务能力尽量不受影响。

### 决策五：停用替代删除

第一期不提供硬删除生成器能力，只提供停用和重新启用。停用后的生成器仍可查看，但不会出现在新的枝化生长可选生成器列表中。

这能减少历史生长任务和历史果实追溯被破坏的风险，也符合领域文档中“不可删除，停用可见”的约束。

### 决策六：重新上传替代在线编辑，且不引入完整版本系统

第一期若需要修改生成器 Skill 内容，只允许重新上传 zip。重新上传后，生成器仍保持同一个受管理资源身份，新内容用于后续使用；历史记录仍能追溯到该生成器，但第一期不承诺恢复任意旧版本 Skill 本体。

完整版本管理会引入更复杂的存储、回滚和历史执行语义，不适合作为第一期生成器模块的必要范围。

### 决策七：生成器领域只提供可用性，不承担生长规则

生成器模块可以提供启用生成器列表、生成器详情和 Skill 内容位置。枝化生长是否必须选择生成器、如何读取生成器并交给 Agent、如何把输出封装成果实，都由枝化生长领域负责。

该边界可以避免生成器领域反向依赖生长流程，也让生成器保持纯粹的创作方法论资源管理职责。

## Risks / Trade-offs

- zip 中存在路径穿越或恶意文件 → 内容访问适配器必须在解压和保存时阻止越界路径，并拒绝绝对路径。
- zip 过大或包含不适合本地保存的内容 → 实现阶段需要在 API 契约或应用校验中定义合理的上传限制和错误反馈。
- 文件保存成功但数据库写入失败 → 应用服务需要以事务或补偿方式处理文件与系统事实一致性，避免产生不可追溯的孤儿内容。
- 数据库记录存在但 Skill 文件缺失或损坏 → 详情读取和可选生成器查询必须能感知内容访问失败，并返回明确错误而不是伪造正常状态。
- 重新上传覆盖当前 Skill 本体后，历史执行无法还原旧内容 → 第一期接受该取舍，后续如需要强审计或复现实验结果，再引入生成器版本能力。
- 不要求 `generator.json` 会降低机器可读描述能力 → 第一期用用户补充的名称和描述满足展示与选择，未来可选择性支持非强制 manifest。
