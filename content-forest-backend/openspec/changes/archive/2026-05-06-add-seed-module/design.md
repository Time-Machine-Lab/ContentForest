## Context

种子模块是内容森林第一期的第一个后端领域模块。根据 `docs/design/domain/种子领域模块设计文档.md`，种子是内容树的根节点，承载用户输入的标题与 Markdown 灵感正文，并通过归档/回档表达这棵创意树是否仍处于可生长状态。

该模块必须遵守项目架构中的混合存储原则：种子 Markdown 正文是内容本体，存放在内容访问层背后的文件存储；种子身份、归档状态、根节点关系、内容本体位置等系统事实由数据库维护。当前 `docs/api/` 与 `docs/sql/` 还没有种子模块契约，因此实现前必须先补齐顶层 API 与 SQL 文档。

## Goals / Non-Goals

**Goals:**

- 提供种子的创建、查看、编辑、归档、回档和分类列表能力。
- 保证创建种子时标题和 Markdown 正文都不能为空，不存在草稿态种子。
- 保证种子创建成功后成为内容树根节点。
- 保证已归档种子从主列表隐藏，但可在归档列表查看、编辑和回档。
- 保证已归档种子的工作区只读，不允许该种子及其内容树节点继续发起枝化生长。
- 建立种子模块与内容访问层、存储层、后续枝化生长模块之间的边界。

**Non-Goals:**

- 不实现删除种子。
- 不实现草稿种子。
- 不实现标签、分类、平台方向、搜索或高级排序。
- 不实现种子专属营养库维护。
- 不实现果实管理、枝化生长、生长锁、发布验证、数据回流或基因汲取。
- 不在实现中绕过 `docs/api/` 和 `docs/sql/` 顶层契约。

## Decisions

### 决策一：种子正文作为 Markdown 内容本体保存

种子正文属于用户创作内容，应通过内容访问层保存为 Markdown 内容本体。种子领域只通过内容访问层读写正文，不直接操作文件系统。

替代方案是把正文直接保存进数据库。该方案实现更直接，但会破坏架构文档中“内容本体文件化”的原则，也不利于后续内容迁移和外部编辑。

### 决策二：数据库只维护种子系统事实

数据库负责维护种子的系统事实，包括身份、归档状态、根节点语义和内容本体位置等。Markdown 不保存归档状态、根节点关系或其他 meta 信息。

替代方案是在 Markdown 中写 frontmatter 保存 meta。该方案会形成文件和数据库的双事实来源，容易造成状态不一致，明确不采用。

### 决策三：归档是创意树结束信号，不是删除

归档种子表示该创意、想法或项目已经阶段性结束。归档后种子从主列表隐藏，进入已归档列表；用户仍可查看详情和已有工作区，也可编辑标题和正文，但工作区进入只读状态，不能继续发起枝化生长。

替代方案是归档后仅隐藏主列表但仍允许生长。该方案会削弱“归档代表创意树阶段性结束”的业务语义，因此不采用。

### 决策四：列表分类只按归档状态区分

第一期只提供未归档列表和已归档列表，不提供搜索、高级排序、标签或分类。列表展示顺序采用最简单稳定策略即可，具体排序不作为本变更的核心要求。

替代方案是在第一期加入搜索和标签。该方案会引入非必要的筛选语义和额外存储/API 设计，不符合种子模块轻量定位。

### 决策五：种子可生长性由种子归档状态与枝化生长锁共同决定

种子模块只负责判断种子是否因归档而不可作为生长源；节点是否正在生长、是否被加锁由枝化生长模块负责。后续枝化生长模块调用种子能力时，应同时遵守这两个边界。

替代方案是把生长锁放进种子模块。该方案会让种子领域承担枝化生长过程状态，破坏领域边界，因此不采用。

## Risks / Trade-offs

- [Risk] 文件内容与数据库事实不一致 → Mitigation：所有种子正文读写都通过内容访问层，创建种子时必须保证内容本体和系统事实同时建立成功。
- [Risk] 已归档种子仍允许编辑可能让“只读工作区”语义混淆 → Mitigation：明确“只读工作区”指不能新增生长和修改树内节点状态；种子自身标题/正文编辑仍允许。
- [Risk] 未先定义 API/SQL 就开始实现会违反项目宪法 → Mitigation：tasks 中将创建 `docs/api/` 与 `docs/sql/` 顶层文档列为首要任务。
- [Risk] 种子模块提前承担工作区或枝化生长逻辑 → Mitigation：种子模块只提供根节点与可生长性基础语义，工作区展示和生长锁分别留给对应领域。
