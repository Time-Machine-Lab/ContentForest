## Context

内容森林现有设计已经确立“生成器自由，果实确定，系统事实由内容森林维护”的核心边界。枝化生长 Skill 负责读取来源节点、生成器、营养和基因上下文，驱动生成器产出 payload，再封装为候选果实；基因汲取领域负责把选择、淘汰、发布和反馈中的经验沉淀为种子级基因库。

当前问题在于：这些上下文虽然能进入 Agent，但缺少稳定的策略编排层。多源资料被整体交给模型，模型自行决定重点；一次批量生长的多个 attempt 缺少明确差异化目标；基因更多表现为标签或总结，尚未成为可验证、可继承、可变异的内容表达特征假设。

本次设计基于现有文档中的领域语义进行升级：

- 《内容森林概念设计文档》已经定义内容森林是“AI 内容进化引擎”，但需要补充“内容进化算法模型可迭代”的产品原则。
- 《内容森林架构设计文档》已经定义 AgentPort、内容访问层、文件与数据库边界，本次不改变这些架构边界。
- 《枝化生长领域模块设计文档》已经定义生成器 Payload、果实候选、基因标签，本次在其前面补充“生长策略”。
- 《基因汲取领域模块设计文档》已经定义基因库、基因经验、谱系、生态位，本次把基因经验进一步明确为“内容表达特征假设”。
- 《内容森林Agent架构设计文档》已经定义 Tool 受控读取和 Skill 边界，本次在内置 Skill 内增加策略编排与可观测性。

## Goals / Non-Goals

**Goals:**

- 建立轻量、可迭代的内容进化算法模型，避免把某个爆款公式写死。
- 在枝化生长中引入 attempt 级生长策略，让每个果实生成尝试拥有明确探索方向。
- 将多源上下文从“整包塞给模型”升级为“证据卡片 + 策略任务书”的组织方式。
- 让批量生成具备多样性探索能力，而不是同一 Prompt 的重复采样。
- 将基因定义为可复用、可验证、可继承、可变异或规避的内容表达特征假设。
- 让基因汲取输出更适合下一轮生长使用，包括成功因子、失败因子、适用生态位、证据解释和下一轮建议。
- 同步修正顶层文档中的概念表述，使第一期边界从“不做复杂算法”变成“先做轻量可迭代算法框架，不做重型算法基础设施”。

**Non-Goals:**

- 不引入向量数据库、消息队列、独立实验平台或复杂推荐系统。
- 不实现自动 A/B 测试、自动发布、自动数据抓取或全自动物竞天择。
- 不让 Agent 直接写数据库、写 Markdown 文件、创建果实或确认基因。
- 不改变现有公开 API 契约和数据库表结构。
- 不自动改写外部生成器 Skill，不要求生成器遵守内容森林专用输出契约。
- 不宣称当前算法模型是最终最优解；本次只建立可持续迭代的算法骨架。

## Decisions

### Decision 1: 引入“内容进化算法模型版本”

系统在 Agent 运行语义中引入算法模型版本概念，例如第一版可称为 `content-evolution-v1`。版本不是数据库强约束字段，而是任务上下文、Agent trace、exchange log 和策略产物中的可观测元信息。

理由：我们需要承认“如何生成好内容”本身会持续迭代。用版本标识可以让后续对比不同策略模型、排查生成质量变化、回放日志时知道某个果实由哪一版算法思想驱动。

Alternative considered: 直接把当前提示词写死在 Skill 中。该方案实现最快，但后续很难解释不同阶段生成质量变化，也无法比较算法迭代收益。

### Decision 2: 生长策略是内置枝化生长 Skill 的中间产物

生长策略不是新的业务领域，也不需要第一期单独持久化。它由内置枝化生长 Skill 在读取上下文后形成，用于指导本次 attempt 如何使用生成器、证据、基因和用户输入。

策略至少表达：

- 本次 attempt 的平台/内容目标假设。
- 来源节点中应继承或延展的内容要点。
- 本次使用的营养证据和使用理由。
- 本次继承、强化、变异或规避的基因假设。
- 本次 attempt 与同批次其他 attempt 的差异化方向。
- 本次生成应关注的成功判据。

理由：策略编排应属于“把不确定性喂给生成器之前”的确定性组织工作，符合“生成器负责内容 payload，枝化生长负责封装与编排”的既有架构。

Alternative considered: 新增独立 Strategy 领域模块。该方案概念完整，但第一期会引入过多对象、接口和持久化设计，不符合当前轻量边界。

### Decision 3: 证据卡片替代整包上下文

Tool 可以继续读取授权资源，但枝化生长 Skill 不应把所有长文资料无差别塞进生成器 prompt。它应先组织轻量证据卡片，每张卡片包含来源类型、标题/摘要、相关性理由、建议用途和内容摘录。

理由：RAG、MMR、RRF、learning-to-rank 等成熟信息检索思想都说明，上下文质量不只取决于资料数量，还取决于相关性、多样性和排序。第一期不需要实现复杂检索模型，但需要形成可替换的证据选择接口和清晰的证据组织格式。

Alternative considered: 继续整包拼接上下文。该方案简单，但会放大 prompt 噪声，模型对资料的关注点不可控，也更难解释某个果实为什么这样生成。

### Decision 4: 批量生长使用“探索槽位”而非重复采样

一次 GrowthTask 中的多个 attempt 应在策略层被分配不同探索槽位，例如痛点共鸣、工具价值、反常识观点、案例故事、收藏清单、负反馈规避等。槽位可以由固定启发式生成，也可以后续替换为更智能的策略模型。

理由：内容森林要寻找最佳传播方式，就必须同时探索多个可能方向。同一 prompt 重复采样会产生表面不同但策略相近的果实，降低物竞天择的信息价值。

Alternative considered: 仅提高 temperature 生成多样性。该方案能增加文字差异，但无法保证策略差异，也无法把反馈映射回可复用基因。

### Decision 5: 基因定义为“内容表达特征假设”

基因不是普通标签，也不是绝对真理。它是某个种子、平台、受众、目标和证据集合下的内容表达特征假设。它可以是正向的成功因子，也可以是反向的失败因子。

基因建议应至少能够表达：

- 特征本身是什么。
- 它属于哪类表达维度，如选题、受众、标题、情绪、结构、价值、证明、平台适配、CTA。
- 它适用于什么生态位。
- 它由哪些证据支持。
- 它建议下一轮如何继承、强化、变异、组合或规避。

理由：只有这样的基因才能进入下一轮生长策略，成为可操作的生成约束，而不是停留在“痛点共鸣”这类泛化标签。

Alternative considered: 继续只保存 geneTags。该方案轻量，但无法表达证据、方向、适用边界和下一轮使用方式。

### Decision 6: 策略模型先用启发式 + LLM 编排，不引入重型算法服务

第一版可以使用确定性启发式组织上下文，再用 LLM 生成或整理策略任务书。后续可以逐步引入更强的检索、排序、bandit、适应度评分或实验分析能力。

理由：我们需要快速改善生成质量，并为未来迭代留下清晰边界，而不是在第一期直接建设完整推荐/实验系统。

Alternative considered: 一次性引入向量检索和多臂老虎机。该方案更“算法化”，但需要额外基础设施、数据量和实验闭环，不适合当前阶段。

### Decision 7: 顶层文档同步作为本次变更任务

本次优化会改变核心概念表达，因此必须同步 docs，而不是只改代码。重点修正：

- “生成内容”升级为“内容进化搜索”。
- “基因标签”与“基因经验/基因假设”区分。
- “第一期不做复杂算法”改为“不做重型算法基础设施，但建立可迭代策略框架”。
- Agent 架构中补充策略编排、证据卡片、算法版本和可观测性。

理由：`docs/` 是项目唯一事实来源，核心概念变化必须先在顶层文档中可见。

## Risks / Trade-offs

- [Risk] 策略编排增加 LLM 调用成本 → Mitigation: 第一版优先复用现有两阶段调用，必要时将策略编排合并进生成器 payload prompt，避免新增强制 LLM 调用。
- [Risk] 策略产物过复杂导致第一期开发膨胀 → Mitigation: 不新增 API/SQL，不新增独立领域模块，先作为 Agent 内部中间产物和日志可观测内容。
- [Risk] 基因定义变重后用户确认成本上升 → Mitigation: UI 与后端仍允许用户编辑/确认建议；Agent 输出要面向复用，避免长篇分析报告。
- [Risk] 启发式策略不一定比当前 prompt 更优 → Mitigation: 通过算法版本、trace 和 exchange log 保留可比较性，后续允许替换策略模型。
- [Risk] 过度追求爆款导致内容低质或违规 → Mitigation: 策略模型必须保留品牌适应度、合规稳定性和负反馈规避语义，生成器仍可定义平台合规规则。
- [Risk] 资料筛选误删重要上下文 → Mitigation: 第一版证据卡片保留来源摘要和使用理由，日志记录被采用证据；后续可增加用户显式引用优先级。

## Migration Plan

本次变更不需要数据库迁移和 API 迁移。实施顺序建议：

1. 先同步顶层 docs 和领域文档，确立新概念边界。
2. 在 Agent/枝化生长 Skill 中引入策略编排中间产物和算法版本 trace。
3. 调整候选果实结构化提示，减少 `<think>`、中间分析和系统事实污染。
4. 优化基因汲取 Skill 输出，使建议更像可复用假设，而不是普通总结。
5. 补充测试和日志断言，验证策略、证据、基因方向和可观测性。

回滚时可退回旧 Skill prompt 和旧基因建议格式；由于不改 API/SQL，已有数据仍可读取。

## Open Questions

- 是否需要在下一轮变更中把算法版本、策略摘要或使用证据显式展示到前端果实详情中。
- 是否需要为生成器增加可选的“策略感知”约定，例如生成器可以声明自己支持哪些探索槽位。
- 是否需要把用户手动选择的营养/基因设置为最高优先级证据，还是允许策略模型降低其权重。
- 何时引入真实适应度评分，目前仍建议等数据回流样本更稳定后再单独设计。
