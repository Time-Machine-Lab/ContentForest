## Context

内容森林后端当前已有种子与生成器模块实现，并预留了 `src/modules/publication/` 目录，但尚未实现发布验证能力。根据 `docs/design/domain/发布验证领域模块设计文档.md`，发布验证领域负责记录“已选择果实通过发布器进入外部环境验证”的事实，并为后续数据回流领域提供发布记录入口。

当前最大约束是果实模块仍在开发中。发布验证模块需要校验“只有已选择果实才能发布”，但不应该直接管理果实选择状态，也不应该提前依赖果实模块内部结构。因此本变更需要通过窄端口表达“果实是否可发布”的依赖，等待后续果实模块接入真实实现。

按照后端开发规范，本变更涉及 API 与 SQL 契约，必须同步新增 `docs/api/publication.yaml` 和 `docs/sql/publication.sql`；实现侧一份 API 契约对应一个 Publication Controller，一张发布记录表对应一个 SQL 文档。

## Goals / Non-Goals

**Goals:**

- 实现发布验证领域后端模块，支持人工发布记录的创建、编辑、查看和按果实列出。
- 内置人工发布器语义：第一期只记录用户已在外部平台完成发布的事实，不执行自动发布。
- 保护“只有已选择果实才能发布”的领域规则，并通过窄端口依赖果实可发布校验。
- 支持一个果实拥有多条发布记录，用于多平台或多次外部验证。
- 支持编辑发布目标、发布凭证和发布备注。
- 发布时间跟随发布记录创建时间，不作为用户可编辑字段。
- 发布凭证第一期只作为文本或链接记录，不做截图附件上传。
- 为后续数据回流领域提供发布记录存在性校验或读取入口。
- 新增发布验证 API 契约和 SQL 契约文档。

**Non-Goals:**

- 不实现果实模块真实状态管理；果实可发布校验真实适配由后续果实提案完成。
- 不提供发布记录删除或归档能力。
- 不提供平台枚举，发布目标由用户自由描述。
- 不实现发布器管理页面或发布器增删改查。
- 不实现自动发布器，不接入任何外部平台 API。
- 不记录浏览、点赞、评论等表现数据；这些属于数据回流领域。
- 不实现数据快照能力。
- 不实现截图附件上传或附件管理。

## Decisions

### 决策一：发布记录作为发布验证领域聚合根

发布验证领域的核心事实是“某个果实发生了一次外部发布验证”。因此实现上以发布记录为主要聚合，围绕发布目标、发布凭证、发布备注、创建时间和关联果实组织业务规则。

备选方案是把发布状态直接写回果实模块。该方案会让果实承担发布历史与外部验证职责，破坏“发布状态不属于果实选择状态”的领域边界。发布记录独立建模更利于一个果实多次发布，也更适合后续数据回流按发布记录挂载监控器和快照。

### 决策二：人工发布器作为内置固定能力，不做管理模型

第一期只有人工发布器，表示用户已经在外部平台完成发布并回到系统记录事实。实现上可以把人工发布器作为系统内置发布器语义，创建发布记录时固定使用人工发布器。

备选方案是建立完整发布器管理表和管理接口。该方案会提前扩大范围，但第一期没有多个发布器配置需求，也没有自动发布接入，因此不做发布器管理页面和复杂发布器生命周期。

### 决策三：果实可发布校验通过窄端口表达

发布验证模块必须校验果实是否已选择，但不能直接拥有果实状态。设计上引入果实可发布校验端口，发布服务创建记录前调用该端口确认果实可发布。

在真实果实模块完成前，测试可以使用替身实现该端口；后续果实模块实现后，再由果实模块提供真实适配。这样发布验证模块可以先完成自己的边界和规则，同时避免依赖尚未完成的果实内部模型。

### 决策四：发布时间等于发布记录创建时间

用户确认发布时间没必要单独编辑，发布时间跟随发布记录创建时间。因此发布记录的发布时间由系统在创建记录时生成，并且编辑发布记录时不允许修改。

备选方案是允许用户回填外部平台真实发布时间。该方案在人工回填场景中更灵活，但会增加编辑规则和时间语义复杂度。第一期优先保持简单：发布记录代表用户在内容森林中确认该发布事实的时间。

### 决策五：发布凭证第一期只做文本或链接

发布凭证用于证明外部发布事实，第一期以文本、链接或说明形式保存，不做截图附件上传。这样可以满足最小闭环，又避免提前引入附件上传、文件存储、预览和清理规则。

后续如果需要截图凭证，可在附件系统成熟后扩展发布凭证的表现形式，不改变发布记录作为验证事实的核心语义。

### 决策六：发布记录不可删除、不可归档

发布记录是内容验证历史的一部分。第一期不提供硬删除或归档能力，只允许编辑可修正的信息。错误创建的记录如果未来确实需要处理，应另行设计撤销或作废语义，而不是在本次变更中引入删除。

### 决策七：API 与 SQL 契约先行

根据后端开发规范，本变更需要先补齐 `docs/api/publication.yaml` 与 `docs/sql/publication.sql`，再实现代码。API 契约对应 Publication Controller，SQL 契约对应发布记录存储结构。

这能让接口能力、存储事实和领域文档保持一致，并方便后续前端与数据回流模块对接。

## Risks / Trade-offs

- 果实模块尚未完成，发布验证无法接入真实选择状态 → 通过果实可发布校验端口隔离依赖，先用测试替身覆盖发布验证规则，后续由果实模块提供真实适配。
- 发布时间跟随创建时间，无法表达外部平台实际发布时间 → 第一期开发布验证闭环时接受该限制；如后续需要真实外部发布时间，可作为新需求设计。
- 发布凭证只做文本/链接，无法保存截图 → 避免提前拉入附件系统；后续可在附件能力成熟后扩展。
- 不做删除或归档，错误记录会保留 → 允许编辑目标、凭证和备注修正人工录入；若未来需要撤销语义，再单独设计。
- 发布目标不做枚举，数据统计会较松散 → 第一阶段优先适配不同平台和渠道的自由记录；后续如需要平台分析，可引入标准化目标或标签。
- 发布器只内置人工发布器，自动发布器接入时需要扩展 → 当前模型保留发布器抽象，未来可在不改变发布记录语义的前提下增加自动发布器。
