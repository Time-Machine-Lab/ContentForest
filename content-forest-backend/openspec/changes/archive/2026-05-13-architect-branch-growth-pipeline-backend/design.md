## Context

现有枝化生长已经支持生长任务、attempt、并发尝试、生长锁、最近失败输入和 AgentPort 调用。上一轮算法优化已经引入 attempt 级策略上下文和内容进化策略雏形，但它仍偏内部优化，缺少明确的管线边界、用户可理解的搜索模式、突变激进程度和生成中路径图。

第二期的目标不是一次性做复杂遗传算法，而是把枝化生长架构化：每层职责清晰、策略可替换、过程可追踪、失败可定位。

## Goals / Non-Goals

**Goals:**

- 建立枝化生长六层管线结构。
- 在创建生长任务时支持搜索模式和突变激进程度。
- 系统可为搜索模式和突变激进程度提供默认推荐。
- 动态发现突变方向，而不是依赖固定枚举。
- 生成本轮生长简报作为 Agent 输入上下文。
- 记录用户可理解的生成中路径图，供前端轮询展示当前正在执行的实际工作。

**Non-Goals:**

- 不做全自动多轮无人实验。
- 不引入向量数据库、队列或独立实验平台。
- 不暴露数字突变率给普通用户。
- 不把路径图做成正式任务管理系统。
- 不把 Agent Trace、Tool 调用日志、LLM 调用日志直接作为用户可见路径图。
- 不让 Agent 直接创建果实或写数据库。

## Decisions

### Decision 1: 管线是 Growth 应用层编排，不是新领域

输入层、上下文补全层、创作搜索层、注意力编排层、生成执行层和结果封装层属于枝化生长应用服务内部编排。

理由：它们服务于一次枝化生长任务，不应膨胀成独立领域模块。

### Decision 2: 本轮生长简报是临时上下文

本轮生长简报由种子正文、种子主简报摘要、来源节点、用户输入、引用营养、引用基因、生成器和任务参数组装。它可写入 trace 或 attempt 输入摘要，但不作为长期可编辑文档存在。

理由：长期创作地图归种子主简报；每次任务书归生长管线，避免概念重叠。

### Decision 3: 用户选择突变激进程度而非突变率

前端提交保守、均衡、激进等语义化突变程度。后端将其映射为内部搜索半径和变异策略。

理由：数字突变率对内容创作用户不可感知，语义化程度更容易理解和调试。

### Decision 4: 突变方向动态发现

系统不维护固定突变维度枚举。创作搜索层基于种子、主简报、父果实、营养、基因和用户输入生成候选突变方向，再根据突变激进程度筛选。

理由：内容森林需要适配产品推广、美食、教学、搞笑、知识科普等不同种子，固定维度会把系统锁死在某类内容范式里。

### Decision 5: 路径图是用户进度，不是工程 Trace

路径图由人能理解的实际任务组成，例如获取输入、读取资源补全上下文、发现创作方向、使用生成器、生成文案、生成封面、生成视频、封装候选果实。Agent Trace、Tool 调用、LLM 调用、Skill 调用等工程事件必须保留在日志或调试信息中，不得直接进入普通用户可见的 `pathGraph`。

理由：用户需要知道“AI 正在做什么”，而不是看到运行时内部事件。把 trace 直接展示会造成路径冗长、语义噪声和进度不可理解。

### Decision 6: 路径图由管线阶段和用户进度子步骤共同组成

管线固定提供用户可理解的大阶段，例如准备输入、补全上下文、发现方向、使用生成器、封装果实。Agent 或生成器执行阶段可以追加子步骤，并更新步骤状态，但这些子步骤必须是用户可理解动作。

理由：用户需要知道总进度；生成器内部也可能有多阶段创作，必须允许 Agent 补充。

### Decision 7: 路径图只面向生成中展示

任务完成后，普通用户接口不需要继续展示路径图；工程侧可以保留 trace 和日志。

理由：路径图是过程感知，不是果实正式内容，也不参与物竞天择。

## Risks / Trade-offs

- [Risk] 管线抽象过重拖慢开发 → Mitigation: 第一版只实现轻量内部函数/服务边界，不引入工作流引擎。
- [Risk] LLM 生成突变方向不稳定 → Mitigation: 保留种子核心和用户输入作为不可变约束，失败时降级为默认方向策略。
- [Risk] 路径图更新过频繁 → Mitigation: 仅记录阶段级更新和 Agent 明确上报的子步骤。
- [Risk] 工程 Trace 泄漏到用户路径图 → Mitigation: 后端生成 `pathGraph` 时只消费用户进度事件和管线阶段，trace 仅用于日志和调试。
- [Risk] 前端依赖字段过多 → Mitigation: `docs/api/growth.yaml` 只暴露用户可感知的搜索模式、突变程度、方向摘要和路径步骤。

## Migration Plan

1. 先更新 `docs/api/growth.yaml` 和 `docs/sql/growth.sql`，补充搜索模式、突变程度、用户可见路径步骤和 trace 摘要边界。
2. 更新枝化生长领域文档和 Agent 架构文档。
3. 在 Growth 应用服务中引入管线阶段边界。
4. 调整 AgentPort 输入输出契约。
5. 补充测试并验证旧的枝化生长请求在缺省参数下仍可工作。

回滚时可忽略新增搜索和路径字段，保留旧生长任务执行路径。
