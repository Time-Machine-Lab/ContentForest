## Context

当前基因汲取能力已经具备基础闭环，但模型表达仍偏静态：系统能生成候选结论，却还没有把“为什么选择/为什么淘汰”的人为原因、证据权重、正负向基因语义和后续可追溯性统一纳入同一条链路中。

本次变更聚焦于基因汲取的模型修正，而不是重写整个基因汲取业务。目标是让基因汲取从“生成建议”升级为“证据驱动、可解释、可演进”的沉淀机制，并保持与现有 Agent 层、内容访问层和种子级基因库结构兼容。

## Goals / Non-Goals

**Goals:**
- 让基因汲取输入显式携带用户原因与事件上下文。
- 让基因汲取输出以证据驱动的候选建议为中心，而不是抽象结论。
- 清晰区分正向基因、负向基因与后续可复用的谱系信息。
- 保持 Agent 只负责生成建议，确认与沉淀仍由后端应用层完成。
- 为后续基因持续跟进、评优和迭代留出模型扩展空间。

**Non-Goals:**
- 不在本次变更中引入新的 Agent 框架或新的任务编排系统。
- 不在本次变更中重做基因库页面或工作区交互。
- 不在本次变更中定义具体数据库字段、接口入参出参细节或代码实现。
- 不把基因汲取扩展为全自动决策系统，第一阶段仍保留人为确认入口。

## Decisions

### 1. 将“用户原因”纳入基因汲取输入模型

基因汲取不再只依赖客观事件本身，还要接收用户对本次选择、淘汰、发布或反馈的主观理由。这些理由不直接等同于基因结论，而是作为重要证据来源参与后续推断。

**原因：**
- 同一事件在不同种子和不同内容目标下，含义可能不同。
- 用户原因能显著提升候选建议的可解释性。
- 后续的基因持续评优需要知道最初为什么会被采纳。

**替代方案：**
- 只依赖行为事件，不纳入用户原因。该方案实现更简单，但会降低汲取质量，也不利于后续演化。

### 2. 基因汲取输出定位为“候选建议”，不是最终事实

Agent 的职责是基于授权上下文生成一组候选基因建议，并给出对应解释。最终是否进入种子级基因库，由后端应用层在校验后交给用户确认。

**原因：**
- 维持系统事实与模型推断的边界。
- 防止 Agent 直接写入业务事实导致不可控沉淀。
- 便于后续替换 Agent 或算法实现，而不影响业务落地层。

**替代方案：**
- Agent 直接返回可沉淀基因并自动入库。该方案速度快，但会破坏可控性和可审计性。

### 3. 基因语义分为正向、负向与谱系三层

基因汲取结果不只表达“有效特征”，也要表达“无效特征/应避免特征”以及它们所属的演化谱系。

- **正向基因**：对目标内容表现有促进作用，值得保留和复用。
- **负向基因**：与目标偏离、低效或失败相关，后续生成中应降低权重或避免复现。
- **谱系信息**：用于解释某类基因从哪些内容分支中沉淀出来，以及它在不同场景中的演化关系。

**原因：**
- 只记录成功经验会让系统偏乐观，难以避免重复失败。
- 内容生成的价值不只在于复制成功，也在于规避失效模式。
- 谱系信息是后续“持续跟进 + 评优 + 迭代”机制的基础。

### 4. 证据驱动优先于主观归纳

每条基因建议都必须关联至少一种证据来源，并明确其来自哪个事件链路。证据可以来自选择结果、淘汰结果、发布反馈、数据回流等，但不能脱离上下文空谈。

**原因：**
- 降低“看起来合理但无法验证”的建议比例。
- 让用户在确认时知道这条基因为什么值得保留。
- 便于后续做基因评优、淘汰和重新校准。

**替代方案：**
- 仅输出自然语言结论，不保留证据链。该方案表达轻，但无法支撑长期迭代。

### 5. 继续保持“确认后沉淀”模式

基因库中的正式内容仍只接纳用户确认后的结果。建议、草稿、解释和证据都属于汲取阶段产物，不应在确认前直接进入长期沉淀层。

**原因：**
- 避免错误经验污染基因库。
- 保持“模型推断”和“系统事实”分层。
- 与现有内容森林的数据边界一致。

### 6. 为后续基因迭代预留演化信息

本次设计要显式考虑基因的后续跟踪：一个基因在多次后续生成中的表现应能被观察、评估和再标注。设计上不要求现在就完成完整算法，但要让模型天然支持“复利式增强”。

**原因：**
- 基因库不是静态知识库，而是演化库。
- 后续需要支持基因评优、基因衰减、基因重组等能力扩展。

## Risks / Trade-offs

- [增加输入复杂度] → 通过把“用户原因”做成可选但推荐的上下文，避免第一期操作负担过重。
- [证据链过长导致建议阅读成本上升] → 在展示层只突出结论与摘要，完整证据留给详情页或展开区。
- [正负向基因语义混淆] → 在概念层强制区分“促进因素”和“避免因素”，避免把失败经验误当成可用经验。
- [模型推断过强] → 始终把 Agent 输出限制为建议，最终沉淀必须经过后端确认。
- [未来算法升级影响当前模型] → 使用稳定的语义边界与分层结构，尽量不把算法选择写死在领域契约里。

## Migration Plan

1. 先扩展基因汲取的输入语义，让任务上下文能表达事件原因与证据范围。
2. 再调整 Agent 输出语义，让建议包含正负向区分和证据解释。
3. 后端在确认沉淀前增加更明确的校验与归类逻辑。
4. 逐步补齐基因谱系、评优和后续跟踪能力，但不要求与本次改动同步完成。

Rollback 方面，只要保留旧版“候选建议 -> 用户确认 -> 沉淀”的主链路，便可以回退到更简单的表达方式，而不影响已有基因库数据。

## Open Questions

- 正负向基因在后续展示层是否需要采用统一视图，还是拆分为不同标签体系。
- 基因评优的评分依据是否要在本次变更中先预留语义位，还是留待下一次优化。
- 证据权重是更适合由 Agent 解释，还是由后端进行统一再计算。
- 基因谱系在第一期是否只做人工可读的演化描述，还是需要提前为自动化分析预留更强结构。
