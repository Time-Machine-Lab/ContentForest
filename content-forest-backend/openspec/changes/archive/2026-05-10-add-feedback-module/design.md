## Context

发布验证模块已经提供人工发布记录能力，并通过 `docs/api/publication.yaml` 与 `docs/sql/publication.sql` 定义了发布记录的顶层契约。数据回流领域目前只有领域设计文档，后端 `src/modules/feedback/` 尚未实现，因此发布后的表现数据无法沉淀，也无法稳定供基因汲取读取。

本设计依据 `docs/design/domain/数据回流领域模块设计文档.md`、`docs/design/domain/发布验证领域模块设计文档.md` 和现有后端分层方式展开。数据回流必须依附发布记录，不能直接挂在果实上；第一期只支持人为监控器和手动录入快照。

## Goals / Non-Goals

**Goals:**

- 新增数据回流顶层 API 与 SQL 契约，并保证一份 `feedback.yaml` 对应一个 Feedback Controller，一份 `feedback.sql` 描述反馈相关存储结构。
- 实现发布记录上的人为监控器挂载能力，保证同一发布记录只能挂载一个监控器。
- 实现反馈快照创建、编辑和查询能力，支持自由结构表现数据和用户观察。
- 为基因汲取证据读取提供反馈上下文，使 `feedback` 证据可以解析到快照历史。
- 保持与发布验证模块、基因汲取模块、存储适配器和 HTTP 装配方式一致。

**Non-Goals:**

- 不实现自动监控器，不调用外部平台 API。
- 不实现监控器管理页面或监控器增删改查。
- 不删除或归档反馈快照。
- 不固定表现指标，不判断内容成功或失败，不计算适应度。
- 不修改果实状态，不修改发布记录内容，不触发自动基因汲取。

## Decisions

### 决策一：数据回流以发布记录为聚合边界

数据快照必须挂在发布记录下，而不是直接挂在果实下。一个果实可以有多条发布记录，每条发布记录对应不同外部平台、渠道或实验语境；如果快照直接挂果实，会丢失“这次数据来自哪一次发布”的上下文。

备选方案是将反馈直接挂到果实。该方案前期更简单，但会在多平台发布、多次发布和后续自动监控器接入时产生歧义，因此不采用。

### 决策二：人为监控器作为内置固定监控器语义

第一期不建立监控器管理模型，只提供“挂载人为监控器”的系统能力。人为监控器表示用户手动查看外部平台后回填数据，不代表系统自动抓取。

备选方案是提前建立监控器表和管理接口。该方案为未来自动监控器更完整，但会把第一期复杂度扩大到配置、生命周期和凭证管理，不符合当前轻量化目标。

### 决策三：反馈快照使用自由结构表现数据

表现数据不做固定指标枚举，后端只验证其为可保存的结构化数据，并配合用户观察文本形成快照事实。不同平台可能使用浏览、播放、点赞、收藏、转化等不同指标，第一期不强行归一化。

备选方案是设计固定指标字段。该方案有利于统计，但会过早绑定平台语义，后续仍需要扩展；当前以数据回流闭环为先。

### 决策四：通过发布验证服务校验发布记录存在

数据回流模块创建监控器挂载或快照前，需要校验发布记录存在。该依赖通过应用服务端口表达，而不是直接读取发布模块内部模型。

备选方案是反馈模块直接依赖发布存储适配器。该方案实现更快，但会模糊模块边界；使用端口更符合当前架构解耦方向。

### 决策五：反馈上下文接入基因证据读取

现有基因汲取已经支持 `feedback` 证据类型的语义占位。该变更需要让证据读取工具在遇到反馈证据时可以读取对应快照及其发布记录上下文，但不自动触发基因汲取。

备选方案是反馈快照创建后自动创建汲取提醒。用户当前更倾向人为触发 + 系统提醒，本次先只提供可读取上下文，自动提醒可后续独立设计。

## Risks / Trade-offs

- [自由结构数据后续难统计] → 第一阶段优先保留原始平台差异；未来可在基因汲取或分析层增加指标映射。
- [一条发布记录只能挂一个监控器可能限制未来混合来源] → 当前用于避免数据冲突；未来需要多来源时可设计监控器版本或来源合并策略。
- [人为录入容易出错] → 支持编辑快照修正，但不删除，以保留历史。
- [前端可能先于反馈 API 开发] → 前端提案中将反馈接入标记为依赖 `docs/api/feedback.yaml`，发布记录接入可先独立完成。
