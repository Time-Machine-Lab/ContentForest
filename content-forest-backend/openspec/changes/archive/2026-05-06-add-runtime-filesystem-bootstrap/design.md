## Context

内容森林第一期采用“内容本体文件化，系统事实数据库化”的混合存储策略。当前种子模块已经通过内容访问层保存 Markdown，并在数据库中保存内容位置，但应用启动阶段尚未统一创建运行时内容根目录，也没有把种子、果实、营养、生成器等内容目录纳入统一装配。

该变更属于基础设施能力，位于 `src/app/config`、`src/app/bootstrap` 与 `src/content-access` 之间。它服务于所有需要读写内容本体的领域模块，但不改变领域模块自身边界。

## Goals / Non-Goals

**Goals:**

- 后端启动时根据配置初始化统一运行时内容根目录。
- 在内容根目录下创建标准子目录，覆盖种子、果实、营养、生成器、附件和临时文件。
- 数据库中的内容位置统一保存相对路径，不保存本机绝对路径。
- 内容访问适配器内部负责将相对路径解析为真实文件路径。
- 同步更新架构设计文档和后端开发规范文档。

**Non-Goals:**

- 不新增 HTTP API。
- 不新增或修改数据库表结构。
- 不实现营养库、生成器、果实等领域模块的业务能力。
- 不引入对象存储、远程文件服务或多租户文件隔离。
- 不实现文件一致性扫描、清理任务或迁移工具。

## Decisions

### 决策一：以 Content Root 作为所有内容本体的唯一运行时根目录

后端配置提供 `contentRootDir`，默认指向本地开发目录下的内容根目录。所有内容访问适配器都以该目录为根解析相对内容位置。

建议第一期目录骨架：

```text
data/
├── app.sqlite
└── content/
    ├── seeds/
    ├── fruits/
    ├── nutrients/
    │   ├── public/
    │   └── seed-scoped/
    ├── generators/
    ├── attachments/
    └── tmp/
```

替代方案是由每个模块自行配置和创建目录。该方案短期简单，但会导致路径规则分散，不利于后续替换本地文件系统或做一致性检查，因此不采用。

### 决策二：目录创建只发生在 Bootstrap 阶段

`src/app/bootstrap` 负责在应用服务装配前创建运行时目录。业务模块不创建目录，也不感知绝对路径。内容访问适配器可以基于 Bootstrap 结果装配，并在写入文件时处理文件名级别的安全校验。

替代方案是在每个 Adapter 写入时懒创建目录。该方案容错更强，但会削弱启动阶段对运行环境的显式检查，容易让装配问题延后到用户操作时才暴露，因此不作为主要策略。

### 决策三：内容位置必须是相对路径

系统事实中的内容位置统一保存相对路径，例如：

```text
seeds/seed_xxx.md
nutrients/public/nutrient_xxx.md
generators/generator_xxx/
```

真实绝对路径只存在于内容访问适配器内部。这样未来迁移到对象存储时，数据库记录仍然是稳定的逻辑位置，不绑定开发者本机路径。

替代方案是在数据库保存绝对路径。该方案能直接定位文件，但会污染系统事实，破坏本地环境迁移和未来存储替换，因此明确禁止。

### 决策四：运行时文件系统配置属于 App Config

运行时目录路径从 `src/app/config` 读取，允许通过环境变量覆盖。业务模块禁止读取环境变量或直接拼接运行时根目录。

第一期建议配置项：

- 内容根目录。
- SQLite 数据库路径。

后续如需要对象存储，可在同一配置边界扩展存储类型和连接信息。

## Risks / Trade-offs

- [Risk] 目录骨架过早固化，后续模块可能需要新增目录 → Mitigation：标准目录只定义第一期已知内容类型；新增内容类型必须更新架构文档和后端规范。
- [Risk] 相对路径被恶意构造为越界路径 → Mitigation：内容访问适配器必须拒绝 `..`、绝对路径和越过内容根目录的解析结果。
- [Risk] Bootstrap 创建目录成功但后续文件写入失败 → Mitigation：内容访问层仍需将文件写入失败映射为内容访问错误，不得静默伪造内容状态。
- [Risk] 当前种子模块已有本地适配器路径约定需要调整 → Mitigation：将其迁移为基于统一 Content Root 的相对路径解析，保持应用服务和存储端口不变。
