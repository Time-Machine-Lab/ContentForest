## Context

当前营养工作台把研究会话和营养内容绑定在一起：营养内容可以保存 `conversationId`，研究会话可以保存 `nutrientCardId`，创建卡片会话时会自动复用或绑定到卡片。这种设计曾用于“围绕单张营养内容持续研究”，但在实际 UX 中造成了三类问题：

- 会话是过程，营养内容是资产，二者绑定后用户难以理解点击营养内容到底是在看资料还是进入对话。
- 一个会话可以产出多个可沉淀营养块，隐式“合并到当前内容”会让合并目标不确定。
- 用户需要删除研究过程时，不应影响已经沉淀或仍在迭代的营养内容。

本设计遵循 `docs/design/domain/营养库领域模块设计文档.md` 中“营养内容是用户侧唯一资料概念”和 `docs/design/内容森林架构设计文档.md` 中“系统事实由数据库维护，内容本体文件化”的方向：会话与营养内容分离，只在用户执行保存或合并动作时发生显式写入。

## Goals / Non-Goals

**Goals:**

- 后端研究会话只归属种子，不再绑定营养内容。
- 营养内容不再保存会话标识，也不通过会话决定详情读取或合并目标。
- 支持删除研究会话，删除范围限定为会话、消息和未沉淀候选块。
- 可沉淀营养块保存为新草稿或合并到既有营养内容时，必须由请求显式提供目标。
- 更新顶层 `docs/api/nutrient.yaml` 与 `docs/sql/nutrient.sql`，使代码实现不再依赖旧绑定字段。

**Non-Goals:**

- 不在本后端提案中实现前端三栏布局调整。
- 不改变营养内容草稿、沉淀、归档、默认带入的生命周期语义。
- 不删除已经沉淀的正式营养内容，也不改变 Markdown 内容本体存储方式。
- 不新增复杂版本系统；合并仍可通过更新目标营养内容 Markdown 和记录轻量合并记录完成。

## Decisions

1. **会话只保留种子归属**

   研究会话保留 `seedId`、标题、创建/更新时间、消息和可沉淀块。移除新建会话时的 `nutrientCardId` 输入语义，不再根据营养内容查找或复用会话。

   备选方案是保留可选绑定但前端不展示。该方案会继续让后端拥有双重真相，并在删除会话、迁移旧数据时制造歧义，因此不采用。

2. **营养内容不再保存会话标识**

   工作台营养内容只维护自身生命周期、正文位置、沉淀关联和默认带入标识。旧 `conversationId` 字段从 API 响应中移除，存储层后续也不再写入。

   备选方案是把会话列表挂在营养内容详情中。该方案会把“资料资产”和“研究过程”重新耦合，不符合新的 UX 模型。

3. **删除会话采用硬删除过程数据**

   会话是研究过程，不是正式资产。删除会话时删除会话记录、消息记录和可沉淀块记录；已经通过保存/合并形成的营养内容不受影响。

   删除会话必须拒绝正在流式运行的请求，或者至少在应用服务层确认没有当前进程内活动流。第一版可以基于服务内活动流注册表做保护；如果服务重启后没有活动流，数据库里旧会话可删除。

4. **合并目标必须显式提供**

   可沉淀营养块保存为新草稿时不需要目标内容；合并到既有营养内容时，后端必须校验目标营养内容属于同一种子且未归档。后端不允许使用“当前选中内容”“会话绑定内容”推断目标。

   备选方案是继续让前端传当前选中内容但命名为 target。该方案可以作为 UI 实现细节，但后端契约必须表达为显式目标，避免业务歧义。

5. **顶层文档先行**

   因为本变更涉及 API 和 SQL，实施时必须先修改 `docs/api/nutrient.yaml` 与 `docs/sql/nutrient.sql`，再更新服务、存储和测试。领域文档同步删除“营养内容会话绑定”概念，补充“研究会话独立于营养内容”的约束。

## Risks / Trade-offs

- [旧数据仍包含绑定字段] → 迁移阶段允许忽略旧字段，并在 SQL 文档中标注废弃；实现完成后新 API 不再返回旧绑定信息。
- [删除会话时仍有流式请求运行] → 应用服务维护活动会话流，运行中拒绝删除；前端需要先停止流式请求。
- [前端未同步导致旧参数调用失败] → 后端可在短期内忽略 `nutrientCardId` 旧字段，但文档和类型标记为废弃；前端改造后移除调用。
- [合并到目标营养内容可能覆盖用户编辑] → 合并实现应读取目标最新 Markdown 后追加或合并，并保留合并记录；复杂冲突检测不在本期范围。

## Migration Plan

1. 更新 `docs/api/nutrient.yaml`：删除会话创建/列表中的 `nutrientCardId` 语义，删除营养内容响应中的 `conversationId`，新增删除会话接口，明确保存/合并可沉淀块的目标参数。
2. 更新 `docs/sql/nutrient.sql`：废弃或移除会话绑定字段说明，并调整索引定义。
3. 更新领域和规划文档：营养研究会话与营养内容解耦。
4. 修改应用服务、Controller、存储端口和适配器。
5. 更新测试，覆盖会话删除、旧绑定不再返回、合并目标显式校验。

Rollback 策略：若前端尚未完成适配，后端可短期保留旧字段解析但不写入绑定；若删除接口存在风险，可先禁用删除入口，只保留解耦后的创建/查询契约。

## Open Questions

- 旧数据库字段是立即移除，还是先在 SQL 文档中标记 deprecated 并在后续迁移统一清理？建议本次文档声明目标结构，代码实现尽量兼容现有 SQLite 初始化。
- 会话删除是否需要软删除？当前建议硬删除，因为会话属于过程数据；若未来需要审计，可另行设计审计日志。
