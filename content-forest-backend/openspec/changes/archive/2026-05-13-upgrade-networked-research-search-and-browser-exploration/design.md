## Context

内容森林第二期正在把营养库从静态资料存储升级为可持续补充、引用和沉淀的种子级营养工作台。现有联网研究模块已经具备 Provider Router、Query Planner、Browser Provider 和结果归一化雏形，但最新走查发现当前 Browser Provider 被用作“打开 Bing 搜索页并读取快照”的工具。这种方式会让搜索引擎页面本身成为瓶颈：Bing 触发 Cloudflare 验证后，系统拿到的是验证页快照，而不是小红书案例。

用户提出的方向是把搜索框架拆成两层：初步搜索通过各种搜索 API 做第一次数据发现；当需要进一步数据、明确平台观察或复杂任务时，再使用 Browser Action 直接访问目标平台并执行站内搜索和页面探索。该方向符合 `docs/design/内容森林Agent架构设计文档.md` 中“联网数据获取必须通过 Provider Router 且 Tool 只返回参考资料”的原则，也符合 `docs/内容森林第二期开发规划文档.md` 中“营养库活化”的目标。

## Goals / Non-Goals

**Goals:**
- 将联网研究明确设计为“初步搜索层 + 深入探索层”的双层管线。
- 初步搜索层通过真实 Search Provider 获取候选线索，避免默认依赖搜索引擎网页 UI。
- 深入探索层通过 Browser Action 直接访问目标平台，支持站内搜索、候选内容打开、详情读取和受限状态识别。
- 以小红书作为第一版平台策略样板，验证“打开平台 -> 观察布局 -> 站内搜索 -> 读取候选 -> 打开详情”的流程。
- 结构化表达验证码、登录墙、布局变化、无结果、Provider 不可用等状态，不把受限页面当作有效研究结果。
- 查询规划器需要区分“搜索关键词”和“用户任务指令”，减少查询词污染。
- 联网研究结果仍只作为营养汲取 Agent 的参考资料和候选营养来源，不直接写营养库或反馈快照。
- 同步更新第二期开发规划、Agent 架构设计和必要的顶层架构文档。

**Non-Goals:**
- 不绕过验证码、登录墙、平台风控或服务条款。
- 不承诺稳定批量抓取小红书、抖音、X 等平台全量爆款数据。
- 不把 Browser Action 做成通用自主网页 Agent 或重型平台情报系统。
- 不在本次变更中实现用户级平台账号授权体系。
- 不自动把搜索结果写入正式营养库、公共营养库或数据库事实。
- 不改变当前营养卡片生命周期和手动沉淀规则。

## Decisions

### Decision 1: 初步搜索和深入探索分层

联网研究不再把“搜索”和“浏览器观察”混成一个 Provider 行为。初步搜索层负责广度，适合调用 Brave、Tavily、SerpApi 等 API，返回候选链接、标题、摘要和来源。深入探索层负责深度，适合在明确目标平台或候选链接后使用 Browser Action 打开页面、站内搜索、滚动、读取详情和提取可见互动数据。

备选方案是继续让 Browser Provider 打开搜索引擎页面。该方案实现简单，但容易受搜索引擎验证码、地区策略、页面布局和反自动化机制影响，并且拿到的是搜索页语境，不是平台内语境。

### Decision 2: Search Provider 不默认依赖网页 UI

初步搜索层应使用 API 型 Provider 作为主路径。Provider Router 可以按配置选择 Brave、Tavily、SerpApi 或后续其他搜索服务。未配置搜索 API 时，系统应返回明确的 Provider 不可用状态，或在策略允许时进入平台探索，而不是回退到固定 Bing 页面。

该设计让搜索源可替换，也便于后续按成本、质量、平台覆盖范围调整 Provider 优先级。

### Decision 3: Browser Action 使用平台策略

深入探索层不应只有一个“打开 URL + snapshot”的通用脚本。系统应引入 Platform Browser Strategy，用于表达不同平台的探索方式：

- 平台入口：例如小红书首页或搜索页。
- 搜索行为：如何找到搜索框、输入关键词、提交搜索。
- 结果读取：如何从快照中识别候选卡片、标题、作者、链接或互动信息。
- 详情读取：如何打开候选内容并提取正文、评论、互动数据或封面语境。
- 受限识别：如何判断登录墙、验证码、空结果或布局变化。

第一版可以只实现小红书策略，并保留 X、YouTube、抖音等策略接口。

### Decision 4: 结果分层为候选线索和已观察案例

搜索 API 返回的内容只能称为候选线索，不能直接称为已验证案例。只有 Browser Action 打开目标链接或平台详情页并读取到可用内容后，结果才可升级为已观察案例。若同时读取到标题、链接、正文摘要和可见互动数据，则可作为更高可信度的案例输入给营养汲取 Agent。

这种分层可以降低“搜索摘要误导 Agent”的风险，也方便前端或日志展示结果来源质量。

### Decision 5: 受限页面是失败状态，不是研究结果

Provider 必须识别常见受限页面信号，例如“请验证您是真人”、Cloudflare 安全质询、登录后查看、访问频率过高、页面不可用、空结果等。命中受限信号时，Provider 应返回结构化失败或受限状态，并携带可理解原因。系统不得把验证码页、登录页或错误页作为研究结果传入 LLM。

### Decision 6: Query Planner 输出双用途计划

Query Planner 需要输出适合初步搜索的查询词，也需要输出适合平台站内搜索的关键词。它必须从用户自然语言中剥离任务指令，例如“找几篇”“保留案例”“梳理核心”“给我 5 到 10 篇”，保留平台、内容对象、题材、人群、内容形式和案例意图。

### Decision 7: 顶层文档同步是本次变更的一部分

本次不是普通 bug fix，而是联网研究架构方向调整。实现前应同步修改 `docs/内容森林第二期开发规划文档.md`、`docs/design/内容森林Agent架构设计文档.md`，必要时修改 `docs/design/内容森林架构设计文档.md`。如果实现中需要对前端返回更细的研究阶段、结果可信度或受限状态，则必须先更新 `docs/api/nutrient.yaml`。

## Risks / Trade-offs

- [Risk] 平台页面结构变化导致 Browser Strategy 失效 -> 通过结构化 `layout_changed` 状态、策略级测试和日志快照降低排查成本。
- [Risk] 小红书等平台需要登录或触发验证码 -> 返回 `restricted_by_login` 或 `restricted_by_captcha`，不尝试绕过。
- [Risk] Search API 成本或 Key 缺失导致初步搜索不可用 -> Provider Router 返回明确不可用原因，并允许策略性降级到平台探索。
- [Risk] Browser Action 比 API 搜索慢且并发成本高 -> 仅在结果不足、用户明确要求深挖、或平台策略要求时启用，并继续使用并发池限制。
- [Risk] Agent 将候选线索误当事实 -> 结果中显式区分候选线索、已观察案例和受限状态。
- [Risk] 平台策略逐渐膨胀为重型情报系统 -> 本次只做轻量策略框架和小红书样板，不做独立平台情报工作区。

## Migration Plan

1. 先更新顶层文档和 OpenSpec 契约，明确双层联网研究架构。
2. 调整 Query Planner，输出清洗后的搜索关键词和平台站内关键词。
3. 调整 Provider Router，支持初步搜索层与深入探索层的分阶段执行、触发条件和失败降级。
4. 接入至少一个 API 型 Search Provider 或提供清晰配置边界。
5. 改造 Browser Provider 为 Browser Action Runtime + Platform Browser Strategy，并实现小红书第一版策略。
6. 增加受限页面识别和结果分层归一化。
7. 更新营养汲取 Skill 的提示上下文，使 Agent 能理解候选线索、已观察案例和受限状态。
8. 使用 fake provider / fake browser strategy 覆盖单元测试，避免测试依赖真实平台。

回滚方式：保留 Provider Router 抽象，若新策略不稳定，可通过配置关闭深入探索层，仅保留初步搜索 Provider 或返回可理解的 Provider 不可用状态。

## Open Questions

- 第一版 Search Provider 选择 Brave、Tavily 还是 SerpApi 作为默认推荐？
- 小红书策略第一版是否需要支持登录态 Profile，还是只做未登录可访问范围内的探索？
- 深入探索触发条件是否允许用户在输入中显式选择，例如“只用站内搜索”或“不要打开浏览器”？
- 已观察案例是否需要在前端展示来源质量标签，还是仅供 Agent 内部使用？
