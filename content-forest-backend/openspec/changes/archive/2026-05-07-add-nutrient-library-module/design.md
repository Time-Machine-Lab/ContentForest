## Context

营养库模块是内容森林第一期枝化生长的资料来源。根据 `docs/design/domain/营养库领域模块设计文档.md`，营养库负责组织公共营养和种子专属营养，将平台指南、爆款案例、产品资料、话术、KOL 分析等外部或人工整理资料保存为可被枝化生长引用的 Markdown 内容。

当前后端已经形成了种子、生成器、果实、枝化生长、基因汲取等模块的基本分层模式：业务模块通过应用服务承载领域能力，通过存储端口维护系统事实，通过内容访问层读写 Markdown 或 Skill 文件本体。营养库模块应沿用该模式，并严格遵守“文件为内容本体，数据库为系统事实”的架构原则。

## Goals / Non-Goals

**Goals:**

- 提供营养库创建、查看、编辑名称与描述、归档和回档能力。
- 支持公共营养库与种子专属营养库两类作用域。
- 创建种子专属营养库时必须绑定一个明确种子。
- 支持在营养库下新增、查看、编辑、归档和回档 Markdown 营养内容。
- 将营养内容 Markdown 正文保存到运行时内容目录，将归属、作用域、状态、内容位置等系统事实保存到数据库。
- 提供按种子查询可引用营养内容的内部能力，供枝化生长和 Agent 只读 Tool 使用。
- 补齐 `docs/api/nutrient.yaml` 与 `docs/sql/nutrient.sql` 顶层契约文档。

**Non-Goals:**

- 不做向量检索、语义检索或知识库问答。
- 不做自动网页采集、文件上传解析或富文本编辑。
- 不判断营养内容是否有效。
- 不调用 Agent，也不决定营养内容如何被 Agent 使用。
- 不把基因汲取确认结果写入营养库。
- 不做跨用户共享市场、评分、付费或推荐。

## Decisions

### 决策一：营养库与营养内容作为同一模块内的两个核心对象

营养库负责作用域和资料集合组织，营养内容负责具体 Markdown 正文。后端模块同时提供库管理和内容管理能力，避免前端或枝化生长模块直接理解底层存储规则。

替代方案是只做营养内容、把“库”降级为标签。该方案无法清晰表达公共作用域和种子专属作用域，也不利于未来把营养库升级为知识库容器，因此不采用。

### 决策二：作用域创建后不可变

营养库创建时必须确定公共作用域或种子专属作用域。公共营养库不能改成种子专属营养库；种子专属营养库不能改绑其他种子。名称和描述允许编辑。

替代方案是允许修改作用域和归属种子。该方案会让既有枝化生长引用和权限校验变复杂，也可能让历史引用语义发生变化，因此不采用。

### 决策三：描述字段非必填

营养库支持描述字段，但描述不是创建营养库的必填条件。用户可以先快速创建资料容器，后续再补充说明。

替代方案是强制填写描述。该方案会提高创建摩擦，和第一期轻量资料库定位不匹配。

### 决策四：第一期营养内容只支持 Markdown 手写或复制粘贴

营养内容正文以 Markdown 文本创建和编辑。第一期不支持文件上传、网页采集、富文本编辑或自动解析。

替代方案是直接支持文件上传和网页导入。该方案会引入格式解析、附件管理和安全处理复杂度，偏离第一期“资料可组织、可引用”的核心目标。

### 决策五：营养内容正文文件化，meta 数据库化

营养内容 Markdown 正文通过内容访问层保存到 `data/content/nutrients/` 下。营养库归属、作用域、归档状态、内容位置和索引信息由数据库维护，不写入 Markdown。

替代方案是在 Markdown frontmatter 中保存归属和状态。该方案会形成双事实来源，并违反架构文档中 Markdown 不保存数据库维护 meta 的约束，因此不采用。

### 决策六：引用查询由营养库模块统一提供

枝化生长和 Agent 只读 Tool 不应直接拼装营养库查询规则。营养库模块需要提供“给定种子，返回可引用营养”的内部能力，并过滤已归档营养库、已归档营养内容和作用域不匹配内容。

替代方案是由枝化生长模块自行筛选。该方案会让作用域规则分散，后续修改营养库规则时容易产生不一致。

## Risks / Trade-offs

- [Risk] 营养内容文件保存成功但数据库写入失败导致孤儿文件 → Mitigation：新增营养内容时使用内容访问层统一写入，并在系统事实写入失败时清理已创建的 Markdown 文件。
- [Risk] 第一版不支持上传文件会降低资料导入效率 → Mitigation：明确第一期只支持 Markdown 手写/粘贴，后续可在同一模块下扩展上传能力。
- [Risk] 种子专属营养库需要选择归属种子，前端可能缺少足够好的选择体验 → Mitigation：后端只提供明确绑定能力，前端通过种子列表选择归属种子。
- [Risk] 不做复杂检索会影响大量资料时的引用效率 → Mitigation：第一期提供基础列表和关键字过滤能力即可，后续再扩展搜索端口或向量检索。
- [Risk] 作用域不可变会让创建错误的库难以改绑 → Mitigation：允许编辑名称和描述；若作用域或归属错误，用户可归档旧库并创建新库。
