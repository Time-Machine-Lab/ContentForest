## Context

内容森林第一期工作区是围绕单个种子的内容树操作界面。前端进入工作区后，需要先加载种子根节点、果实节点、父子关系、节点生长状态、最近失败输入提示、可用生成器、可引用营养内容和可引用基因洞察。当前这些信息分散在种子、果实、枝化生长、生成器、营养库和基因库等领域中。

现有后端已经具备主要领域能力，但 `workspace` 模块仍为空。根据 `docs/design/workbench/内容森林工作区业务设计文档.md` 和 `docs/design/workbench/内容森林工作区前端设计文档.md`，工作区需要的是页面级业务聚合，而不是新的领域事实源。根据本次讨论，节点拖拽位置不需要持久化，因此工作区不需要自有数据库表。

## Goals / Non-Goals

**Goals:**

- 为工作区首屏加载提供统一的后端快照读取能力。
- 将种子、果实、枝化生长、生成器、营养库和基因库等领域信息组合成前端可直接消费的工作区数据。
- 保持工作区模块轻量化，只做读聚合和页面编排，不持久化工作区私有状态。
- 明确工作区 API 契约必须落到 `docs/api/workspace.yaml`，由单一工作区 Controller 对应。
- 保证归档种子的工作区只读语义能被前端感知。

**Non-Goals:**

- 不新增工作区数据库表，不新增 `docs/sql/workspace.sql`。
- 不保存节点拖拽位置、画布视口、布局坐标或前端交互态。
- 不实现果实选择、淘汰、恢复、枝化生长、发布验证、数据回流或基因汲取等突变能力。
- 不把果实 Markdown 正文、种子 Markdown 正文或发布记录全量塞入工作区首屏快照。
- 不替代现有领域模块的接口和业务规则。

## Decisions

### 决策一：工作区作为读聚合模块，而不是领域事实源

工作区模块负责面向页面组合现有领域事实。它可以读取种子、果实、生长状态、生成器、营养内容和基因洞察，但不拥有这些对象的生命周期，也不定义新的核心业务状态。

替代方案是让前端直接调用多个领域接口自行拼装工作区。该方案短期可行，但会让前端承担跨领域编排，导致首屏加载、错误处理和只读状态判断分散，不利于后续工作区交互复杂化。

### 决策二：首期提供工作区快照，不提供画布状态持久化

工作区快照用于支持首屏加载和关键操作后的刷新。节点拖拽、视口缩放、临时展开等交互状态属于前端运行时体验，第一期不保存到后端。

替代方案是新增工作区布局存储。该方案会引入新的表、布局冲突和后续迁移问题，但当前用户明确不需要节点位置持久化，因此不采纳。

### 决策三：树结构由后端聚合，节点正文按需读取

工作区快照应返回构建树所需的节点摘要和父子关系；果实或种子的完整 Markdown 正文仍通过各自领域详情接口按需读取。这样既能让前端快速渲染画布，也避免首屏响应过大。

替代方案是在快照中包含所有节点正文。该方案会让接口简单一些，但内容树变大后会拖慢加载，也会把详情抽屉的按需加载能力变成冗余。

### 决策四：资源选择器由工作区聚合，资源事实仍归原领域

悬浮输入框需要展示可用生成器、可引用营养内容和可引用基因洞察。工作区模块可以提供聚合读取入口，但生成器启停、营养内容归档、基因库沉淀等事实仍归各领域维护。

替代方案是让前端分别调用资源接口。该方案对简单页面足够，但工作区输入框需要统一处理资源引用和枝化详情，聚合入口能减少前端编排复杂度。

### 决策五：突变操作继续走领域接口

工作区页面中的选择、淘汰、恢复、枝化生长、发布、反馈录入和基因汲取都不通过工作区模块代理。工作区模块只提供刷新用的读模型，避免成为“全能页面后端”。

替代方案是提供工作区统一操作接口。该方案表面上让前端调用更集中，但会让工作区模块侵入各领域规则，破坏第一期已经形成的领域边界。

## Risks / Trade-offs

- [Risk] 工作区快照可能随着领域增多而膨胀 -> Mitigation：首期只包含画布渲染和输入框必需摘要，详情内容继续按需读取。
- [Risk] 聚合层递归读取果实时可能出现性能问题 -> Mitigation：第一期数据规模较小，先保持实现简单；实现时需要防御异常循环和过深树结构。
- [Risk] 工作区模块可能被继续塞入突变能力 -> Mitigation：在 proposal、spec 和 tasks 中明确工作区只读聚合边界，突变仍归各领域接口。
- [Risk] 归档种子只读语义在不同入口表现不一致 -> Mitigation：工作区快照必须显式表达只读状态，前端据此禁用枝化输入和突变入口。
