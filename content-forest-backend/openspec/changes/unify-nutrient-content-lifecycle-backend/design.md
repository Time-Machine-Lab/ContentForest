## Context

营养库领域已经具备公共营养库、种子专属营养库、营养内容、营养工作台卡片、研究会话和缺口建议等能力。问题在于用户侧概念开始分裂：正式资料叫“营养内容”，工作台资料叫“营养卡片”，用户会困惑它们是不是两种对象，以及沉淀、归档、删除分别作用在哪里。

本设计基于 `docs/design/domain/营养库领域模块设计文档.md`、`docs/api/nutrient.yaml` 和 `docs/sql/nutrient.sql`。后端仍可以保留现有 `nutrient_cards` 表作为实现承载，但顶层语言应收束为“营养内容的工作台生命周期形态”。

## Goals / Non-Goals

**Goals:**

- 统一营养领域语言：用户侧只理解“营养内容”，状态为草稿、已沉淀、已归档。
- 打开种子营养工作台时自动确保默认种子专属营养库存在，降低用户前置配置成本。
- 支持草稿态营养内容硬删除，满足用户对草稿资料的自然预期。
- 沉淀草稿时允许后端自动选择默认种子专属营养库，避免前端强迫用户选库。
- 将“常驻营养”用户侧表达修正为“默认带入”，保留 `defaultForGrowth` 作为系统契约。

**Non-Goals:**

- 不重构营养库底层表结构为全新统一表。
- 不引入向量检索、自动网页采集或复杂知识库能力。
- 不允许沉淀态或归档态营养内容硬删除。
- 不改动 Agent 研究、联网搜索、枝化生长引用解析的核心策略。

## Decisions

1. 保留 `nutrient_cards` 作为内部承载，统一领域解释。
   - 决策：`nutrient_cards` 继续承载工作台生命周期、研究会话、默认带入和沉淀关联，但文档与 API 描述将其解释为“营养内容的工作台形态”。
   - 理由：避免为概念统一做高风险数据迁移，同时让用户侧语言变简单。
   - 替代方案：合并 `nutrient_cards` 与 `nutrient_contents` 表。该方案迁移成本高，且会把草稿、研究会话等工作台语义污染到正式营养内容表。

2. 新增默认种子专属营养库的幂等确保能力。
   - 决策：后端提供种子维度的 ensure 能力；如果当前种子没有可用默认专属营养库，则创建一个默认库；如果已存在，则直接返回。
   - 理由：前端不应猜测默认库名称或重复创建规则，后端才能保证并发与一致性。
   - 替代方案：由前端查询库列表后自行创建。该方案容易出现竞态和重复默认库。

3. 沉淀草稿时 `libraryId` 可选。
   - 决策：沉淀请求未提供目标库时，后端自动确保并使用当前种子的默认专属营养库；提供目标库时仍校验必须是当前种子的未归档专属库。
   - 理由：默认路径降低用户成本，高级路径仍保留明确归档到指定专属库的能力。

4. 草稿可以硬删除，正式资料仍只归档。
   - 决策：只有 `unsettled` 状态的工作台营养内容允许硬删除；`settled` 和 `archived` 返回业务错误。
   - 理由：草稿是试验性内容，删除符合用户预期；沉淀后的资料进入知识资产，需要保留历史和引用稳定性。
   - 处理边界：删除草稿时应清理其内容本体；已存在的研究会话可保留为历史会话，但不得继续绑定到已删除草稿作为可操作对象。

5. “默认带入”只影响默认选择，不代表强制引用。
   - 决策：后端继续维护 `defaultForGrowth`，可引用营养查询返回该标记；真正是否引用由前端本次枝化生长提交决定。
   - 理由：用户需要可控性，不应因为标记为默认带入而被后端强制塞入所有生长任务。

## Risks / Trade-offs

- [Risk] 内部 API 路径仍包含 `nutrient-cards`，与用户侧“营养内容”语言不完全一致。→ Mitigation: 顶层文档与前端文案统一为营养内容，后端路径作为兼容契约保留，后续大版本再评估 API 命名迁移。
- [Risk] 默认库自动创建可能造成多个默认库。→ Mitigation: ensure 能力必须基于种子维度做幂等查询与创建，必要时通过名称或内部标识约束唯一默认库。
- [Risk] 删除草稿会影响已绑定会话入口。→ Mitigation: 删除只移除草稿可操作对象，会话历史仍可按会话查看；前端不再从该会话定位到已删除草稿。
- [Risk] 沉淀请求兼容新旧调用方式。→ Mitigation: 保留传入 `libraryId` 的老路径，新增不传时自动默认库兜底。
