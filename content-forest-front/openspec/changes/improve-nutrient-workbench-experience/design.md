## Context

营养工作台是用户为某个种子持续补充平台知识、表达规律和可复用营养的研究空间。当前实现已经接入营养卡片、营养汲取建议、研究会话、消息提交和可沉淀营养块接口，但体验仍偏“功能罗列”：卡片详情占据对话区、消息提交反馈滞后、按钮状态相似、可沉淀块与普通消息区分不够。

前端模式下不修改 `docs/api/nutrient.yaml` 与 `docs/sql/*.sql`，因此本次体验升级必须使用现有接口：创建/查看研究会话、提交研究消息、查询可沉淀块、创建/更新/沉淀/归档营养卡片、采纳/忽略缺口建议。

## Goals / Non-Goals

**Goals:**

- 将营养工作台打磨成符合 Quiet Command Workspace 的高密度研究界面。
- 让用户提交消息后立即看到自己的输入和 Agent 研究中反馈。
- 清晰区分卡片上下文、用户消息、Agent 消息、可沉淀营养块和建议卡片。
- 让主要操作按钮具备清晰主次、拟物按压感、加载态、禁用态和危险态。
- 保持现有接口契约不变，降低改造风险。

**Non-Goals:**

- 不实现 SSE 或 token 级流式输出。
- 不修改后端接口、数据库或 Agent 执行逻辑。
- 不引入新的 UI 框架或图标库。
- 不重做营养库页面整体信息架构。

## Decisions

### 1. 先做乐观消息，不等待 SSE

用户提交后，前端立即追加本地用户消息，并展示 Agent 研究中的占位消息；后端返回后用真实消息和可沉淀块替换/补齐。

理由：这能最快解决“回车后没反应”的体感问题，同时不依赖后端协议升级。

替代方案：等待后端 SSE 完成后再改前端。这个方案体验改善会被后端改造阻塞，不适合当前页面打磨。

### 2. 卡片详情改为轻量上下文区

选中卡片时，中间区顶部只展示轻量上下文条和核心动作，完整 Markdown 内容通过展开区查看。

理由：营养研究的主任务是持续对话和沉淀，卡片详情不应长期压缩消息区。

替代方案：保留当前大卡片预览。实现成本低，但对话空间不足，尤其在小屏幕下体验较差。

### 3. 可沉淀营养块采用“成果卡”视觉

可沉淀营养块不作为普通 Agent 消息的一部分展示，而是使用明确的成果卡样式，突出标题、正文摘要和操作区。

理由：它是从对话中可落地的资产，应该比普通聊天内容更接近“待处理结果”。

替代方案：内嵌在 Agent Markdown 中。这样实现简单，但用户难以判断哪些内容可以沉淀。

### 4. 按钮体系按工作流分层

发送、新会话、保留为新卡片等为主要行动；查看、刷新、合并为次要行动；归档、忽略为弱危险或低优先行动。所有按钮都需要 hover、active、disabled、loading 状态。

理由：营养工作台按钮数量较多，必须通过状态和主次减少误操作。

替代方案：保持统一按钮样式。视觉简单，但会让用户无法快速判断下一步。

## Risks / Trade-offs

- [Risk] 乐观消息与后端返回消息重复 → 使用临时本地消息标识，后端返回后按提交结果替换或合并。
- [Risk] 本地占位消息在失败后残留 → 失败时把占位改为失败状态，并恢复输入与重试入口。
- [Risk] 视觉改造过度导致与主工作台风格不一致 → 继续遵循 `docs/spec/DESIGN.md` 的暗色、高密度、低装饰规则。
- [Risk] 三栏信息密度过高 → 保留窄屏 tabs，并控制卡片详情高度，优先保证对话区可用。
