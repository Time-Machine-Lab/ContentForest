## Context

内容森林前端工作区已经在右侧详情面板展示种子和果实详情，并承载物竞天择、发布记录、数据回流和枝化生长入口。用户提出的“果实评测员 / 预测图”更适合作为果实详情中的工作台工具面板：点击任意果实即可查看评测判断、强点、风险、预期区间和核心赌注，而不是在发布前另开一个阶段。

前端宪法要求所有 API 和数据结构严格来自顶层 `docs/api/*.yaml` 与 `docs/sql/*.sql`。因此本变更依赖后端 change `add-content-experiment-calibration` 先落地 `docs/api/content-experiment-calibration.yaml`，前端不自行定义预测图字段或接口。

## Goals / Non-Goals

**Goals:**

- 在果实详情中新增预测图入口和面板。
- 读取当前果实最新预测图，并支持生成/刷新。
- 以紧凑、可扫读的方式展示评测判断、预期区间、强点、风险、核心赌注、反事实场景、推荐观察指标、置信度和盲评状态。
- 在空态、生成中、失败、低置信度和只读工作区中保持清晰可用。
- 不改变节点选择、物竞天择、发布记录、数据回流、枝化生长输入框等既有工作流。

**Non-Goals:**

- 不实现预测快照冻结、校准复盘、基因证据交付或评分画像管理。
- 不在前端模拟平台算法或虚拟用户。
- 不从 Markdown 正文解析预测图。
- 不新增本地持久化状态或修改后端 API/SQL 文档。

## Decisions

1. **预测图放在果实详情面板内，而不是发布前专属步骤**

   用户希望点击果实即可看到预测图，不需要区分“选中”和“发布前”。因此 UI 以果实详情中的预测图视图承载能力，候选、已选择、已淘汰果实都可以查看。

   替代方案是在发布记录创建前展示预测判断。该方案会把评测能力绑定到发布流程，削弱它辅助物竞天择的价值。

2. **预测图详情按需读取，不进入工作区快照**

   预测图结构较丰富，且只在用户查看某个果实时需要。前端点击果实后单独调用预测图 API，避免扩大 `GET /api/seeds/{seedId}/workspace` 首屏快照。

   替代方案是在每个工作区节点上返回预测图摘要。该方案会增加首屏负担，且 MVP 还不确定预测图摘要是否需要在节点卡片上展示。

3. **生成/刷新使用同一个按钮状态模型**

   如果没有预测图，按钮表达生成；如果已有预测图，按钮表达刷新。前端仍调用后端同一个生成/刷新契约，避免 UI 按历史状态分裂复杂流程。

   替代方案是生成和刷新拆成两个入口。MVP 阶段用户价值一致，拆分会让面板更重。

4. **低置信度是显性状态，不做红色错误**

   低置信度表示样本或锚点不足，不是系统失败。UI 应以提示、弱标签或说明行表达边界，而不是错误态。

   替代方案是将低置信度当作 warning toast。该方案容易让用户误解为接口异常。

## Risks / Trade-offs

- [后端契约尚未落地] -> 前端 tasks 标记依赖后端 `docs/api/content-experiment-calibration.yaml`，契约未完成前只做组件骨架或类型占位。
- [预测图面板挤压果实正文阅读] -> 使用详情内 tab/分区切换，保持正文与预测图都可访问。
- [用户把预测当保证] -> UI 必须展示置信度、置信原因和盲评边界，不使用确定性结果文案。
- [刷新覆盖用户理解] -> 刷新完成后展示最新生成时间；历史对比不在 MVP 中做。
- [只读工作区误发写操作] -> 只读时禁用生成/刷新，但允许读取已有预测图。

## Open Questions

- 预测图入口最终采用详情面板 tab、分段控件还是二级工具按钮，需要在实现时结合现有组件结构决定。
- 是否在果实节点卡片上展示极简预测标签，本次先不做，避免节点视觉过载。
