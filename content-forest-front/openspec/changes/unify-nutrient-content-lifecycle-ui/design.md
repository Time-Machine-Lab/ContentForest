## Context

营养工作台已经承载用户投喂、Agent 研究、缺口建议、持续对话和资料沉淀。但前端文案仍大量使用“营养卡片”，而正式资料又叫“营养内容/营养资料”，导致用户以为两者是不同对象。用户期望是简单的：资料先是草稿，觉得有用就沉淀，不用了就归档，草稿写错了可以删。

本设计只调整前端交互语言和状态操作，不私自修改 API 或 SQL。默认专属营养库确保、草稿删除、沉淀请求可省略目标库等能力由后端变更 `unify-nutrient-content-lifecycle-backend` 提供。

## Goals / Non-Goals

**Goals:**

- 前端用户侧统一显示“营养内容”，不再把“营养卡片”作为独立对象教育用户。
- 将状态标签映射为草稿、已沉淀、已归档。
- 打开工作台时自动准备默认种子专属营养库，避免用户先建库再整理资料。
- 草稿态提供删除入口；沉淀态提供归档、默认带入、编辑；归档态提供查看和回档。
- 将“常驻营养”改为“默认带入”，并在枝化生长引用区默认选中但允许移除。

**Non-Goals:**

- 不改动后端 API 文档或 SQL 文档。
- 不在前端实现默认库创建规则。
- 不引入新的营养信息架构页面。
- 不改变 Agent 研究会话的核心执行方式。

## Decisions

1. 用户侧统一称为“营养内容”。
   - 决策：列表、详情、按钮、空状态、建议采纳和可沉淀块操作都使用“营养内容”。
   - 理由：减少用户理解成本，避免“卡片”和“资料”二分。
   - 替代方案：保留卡片概念并加强解释。该方案会继续增加认知负担。

2. 状态标签改为更自然的生命周期语言。
   - 决策：`unsettled` 显示为“草稿”，`settled` 显示为“已沉淀”，`archived` 显示为“已归档”。
   - 理由：草稿可以删除、沉淀代表进入正式资料、归档代表暂时不用，这套语言更贴近用户预期。

3. 默认专属库由后端确保。
   - 决策：工作台打开时调用后端 ensure 能力；失败时展示局部错误和重试，不在前端自行创建库。
   - 理由：前端不应该承担幂等、并发和默认库唯一性规则。

4. 删除只暴露给草稿态。
   - 决策：草稿态显示删除或丢弃入口；已沉淀和已归档不显示删除。
   - 理由：符合用户“草稿可丢弃、正式资料需留痕”的心理模型。

5. “默认带入”作为枝化生长前的可移除默认选择。
   - 决策：已沉淀且标记默认带入的营养内容在悬浮输入框引用区默认出现，用户可叉掉。
   - 理由：它是减负机制，不应剥夺本次生成的控制权。

## Risks / Trade-offs

- [Risk] API 路径仍叫 `nutrient-cards`，开发时容易把文案带回 UI。→ Mitigation: 前端类型层可保留内部命名，但 UI copy 和组件标题必须统一为营养内容。
- [Risk] 后端 ensure 能力未完成时工作台体验不完整。→ Mitigation: 将其作为明确后端依赖，前端提供局部错误和重试，不做临时伪实现。
- [Risk] 用户删除草稿后仍能看到历史会话。→ Mitigation: 会话作为研究历史保留，但删除后的草稿不再显示为可沉淀或可引用对象。
