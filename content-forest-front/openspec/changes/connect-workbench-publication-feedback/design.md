## Context

工作区果实详情面板目前已经展示物竞天择状态、正文、基因标签和记录，但底部的“发布器 / 监控器 / 发布记录 / 数据反馈”仍是禁用按钮。后端发布验证模块已经通过 `docs/api/publication.yaml` 提供人工发布记录接口；数据回流后端由独立提案 `add-feedback-module` 补齐 `docs/api/feedback.yaml` 后再完整接入。

前端需要遵守“发布记录是数据回流入口”的领域边界：发布器作用于已选择果实，数据回流作用于具体发布记录，监控器挂载在发布记录上而不是果实上。

## Goals / Non-Goals

**Goals:**

- 在工作区果实详情中接入人工发布记录创建、列表查看和编辑。
- 只允许已选择果实创建发布记录，候选和已淘汰果实只展示不可发布提示。
- 在发布记录列表中呈现数据回流入口：挂载人为监控器、追加反馈快照、查看反馈历史。
- 在 `docs/api/feedback.yaml` 可用后，通过前端 feedback API 客户端接入后端反馈能力。
- 保持工作区现有内容树、节点详情、枝化生长、基因汲取入口和只读约束不被破坏。

**Non-Goals:**

- 不新增发布器管理页面。
- 不实现自动发布或自动监控。
- 不修改 API 或 SQL 契约。
- 不把数据反馈直接挂到果实上。
- 不对表现数据做固定指标枚举，不判断内容成败。

## Decisions

### 决策一：发布记录交互放在果实详情面板内

工作区用户在浏览某个果实时最自然的下一步是发布验证，因此发布记录入口应留在右侧果实详情面板，而不是新增独立页面。创建发布记录使用轻量弹窗或抽屉，减少离开内容树的上下文成本。

备选方案是新增发布记录页面。该方案更适合批量管理，但第一期工作区闭环更强调单个果实的操作连续性，因此不采用。

### 决策二：已选择果实才显示可执行发布动作

前端遵循后端规则和果实领域语义：只有已选择果实可发布。候选或已淘汰果实可以查看详情，但发布按钮不可执行，并应给出简短说明。

备选方案是让用户点击后由后端报错。该方案实现简单但体验差，前端应提前表达领域约束。

### 决策三：发布记录列表是数据回流的容器

监控器与反馈快照不直接出现在果实一级，而是在每条发布记录下展开。这样可以表达同一果实多平台、多次发布分别回流数据的事实。

备选方案是在果实详情底部直接放“监控器 / 数据反馈”按钮。该方案会隐藏数据来源，容易让用户误以为反馈属于果实整体，不符合领域设计。

### 决策四：发布验证可先落地，反馈接入以契约为门槛

由于 `docs/api/publication.yaml` 已存在，发布记录能力可以直接实现。反馈能力依赖后端 `add-feedback-module` 生成 `docs/api/feedback.yaml`；在契约未存在前，前端任务只能保留入口设计或标记依赖，不得私自发明接口。

备选方案是在前端先做 mock 数据回流。该方案可用于视觉预览，但本次目标是对接真实闭环，因此不作为主实现。

## Risks / Trade-offs

- [工作区详情面板承载内容过多] → 使用分区、折叠列表或二级面板，将发布记录和反馈历史作为可展开内容。
- [发布记录接口可用但反馈接口尚未可用] → 先完成发布验证接入，反馈相关任务显式依赖后端契约。
- [自由结构表现数据前端表单难设计] → 第一版采用键值对或 JSON/文本混合输入，避免固定平台指标。
- [只读工作区误触写操作] → 所有创建、编辑、挂载和快照提交必须尊重 `workspaceReadOnly`。
