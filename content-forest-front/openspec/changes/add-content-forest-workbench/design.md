## Context

前端项目当前已经具备 Nuxt 4、Vue 3、TypeScript 的基础结构，但主应用外壳、种子库页面、种子 API 访问层和可复用 UI 组件尚未落地。本次变更的目标是先完成内容森林主工作台与种子模块前端闭环，为后续内容树画布、枝化生长、果实、生成器和营养库模块提供稳定的页面承载方式。

本次设计必须遵循 `DESIGN.md` 的 Quiet Command Workspace 风格，并遵循 `docs/spec/前端开发规范文档.md` 中的前端分层约束。种子数据只消费 `docs/api/seed.yaml` 已定义的接口，数据结构参考 `docs/sql/seed.sql`，不新增前端私有系统事实。

## Goals / Non-Goals

**Goals:**

- 建立内容森林主工作台外壳，包括紧凑左侧导航、顶部命令区、主内容区和右侧详情区承载方式。
- 实现种子库卡片网格，让用户可以查看未归档种子和已归档种子。
- 实现居中 Command Modal 创建种子。
- 实现右侧详情面板查看、编辑、归档和回档种子。
- 建立种子 API 访问封装，所有种子操作都通过 `docs/api/seed.yaml` 中的接口完成。
- 在前端状态中只保存交互状态、加载状态、错误状态和当前选择，不把前端缓存当作系统事实。

**Non-Goals:**

- 不实现内容树画布、枝化生长、果实详情和生长任务。
- 不实现生成器上传、营养库上传或资源引用。
- 不实现种子搜索接口、排序接口、果实数量统计或最近生长快照，因为当前 API 契约未提供这些能力。
- 不修改后端接口、SQL 文档或数据库结构。
- 不引入 SSR 作为本期核心目标；第一期工作台以客户端高交互体验为主。

## Decisions

### Decision 1: 使用应用级工作台外壳承载所有一阶段页面

工作台外壳放在 Nuxt layout 层，负责全局导航、顶部命令入口和页面内容容器。种子库、生成器、营养库和后续工作区页面作为页面内容挂载在该外壳中。

选择该方案的原因是主工作台属于产品级体验，不应由种子库页面单独拥有。这样后续生成器、营养库和内容树工作区可以复用同一套导航与视觉结构，避免每个页面重复实现主框架。

备选方案是每个页面自行实现导航和顶部栏，但会造成视觉不一致，也会让后续命令面板、快捷键和全局状态难以统一。

### Decision 2: 归档入口属于种子库内部视图

左侧导航不设置“已归档”入口。种子库页面内部提供未归档/已归档切换，并分别调用 `GET /api/seeds` 与 `GET /api/seeds/archived`。

选择该方案的原因是归档是种子集合的筛选属性，不是内容森林工作台的顶级模块。把归档放在种子库内部，可以保持主导航克制，也符合用户刚刚确认的产品边界。

备选方案是把已归档作为全局导航项，但会抬高归档的层级，并让生成器、营养库等模块未来也容易出现类似的全局筛选膨胀。

### Decision 3: 种子库使用卡片网格 + 右侧详情面板

种子库默认采用卡片网格。卡片只展示 `SeedSummary` 能支撑的信息：标题、归档状态、更新时间等。点击卡片后再调用 `GET /api/seeds/{seedId}` 读取详情，并在右侧面板展示和编辑 Markdown 正文。

选择该方案的原因是种子本质上是“灵感标本”，卡片网格比传统表格更适合表达创作对象，同时右侧详情面板能让用户快速浏览与编辑，不打断列表上下文。

备选方案是列表 + 右侧预览，效率更高但气质偏传统后台；卡片 + 弹窗详情则会让用户频繁丢失列表上下文。

### Decision 4: 新建种子使用居中 Command Modal

新建种子通过居中 Command Modal 完成，提交前在前端校验标题和 Markdown 正文均非空，再调用 `POST /api/seeds`。创建成功后关闭弹窗、刷新当前种子集合，并选中新建种子。

选择该方案的原因是创建种子是高频、命令式动作，居中弹窗更贴近 Raycast 风格，也比进入独立创建页更轻。

备选方案是右侧抽屉创建，但右侧面板已经用于查看和编辑当前种子，创建动作若复用抽屉会混淆“当前对象编辑”和“新对象创建”。

### Decision 5: API 封装与页面组件解耦

所有种子接口调用集中到种子模块的数据访问封装中。页面组件只调用模块暴露的组合逻辑或服务函数，不直接拼接 URL。接口类型以 `docs/api/seed.yaml` 的 schema 为依据。

选择该方案的原因是后续后端实现或 API 客户端生成方式变化时，可以尽量限制影响范围。页面只关心用户交互，不直接承载接口细节。

备选方案是在页面组件中直接请求接口，短期更快，但会破坏前端开发规范，也会让错误处理和状态维护分散。

### Decision 6: Markdown 展示与编辑分开处理

详情面板需要支持 Markdown 正文查看和编辑。查看态通过统一 Markdown 组件渲染，编辑态使用文本编辑控件提交完整 Markdown。保存时调用 `PATCH /api/seeds/{seedId}`。

选择该方案的原因是 Markdown 是种子内容本体，必须有统一渲染入口，避免不同页面渲染差异和安全策略不一致。

备选方案是始终显示 textarea，开发最简单但体验粗糙，也不利于后续复用到果实、营养内容等 Markdown 场景。

## Risks / Trade-offs

- [Risk] 当前种子 API 不提供搜索、排序、果实数量和最近生长信息 → 本次只实现本地视图切换和基于返回列表的基础展示，所有超出契约的数据不进入正式实现。
- [Risk] 生成器、营养库导航会先于功能实现出现 → 可先提供占位路由或禁用态入口，但不得伪造已完成功能。
- [Risk] Command Modal 与右侧详情面板同时存在时可能造成焦点混乱 → 弹窗打开时锁定焦点，关闭后回到触发按钮或新建种子卡片。
- [Risk] Markdown 渲染存在安全风险 → 通过统一 Markdown 组件集中处理渲染和安全策略。
- [Risk] 后端接口尚未运行或契约变动 → 前端实现以 `docs/api/seed.yaml` 为准；发现缺口时在任务中标记后端契约依赖。

## Migration Plan

本次为新增前端能力，无历史数据迁移。实现时先建立工作台外壳和种子 API 封装，再接入种子库页面与交互。若需要回滚，可移除新增页面、模块和路由，不影响后端与顶层文档。

## Open Questions

- 种子库后续是否需要真实搜索和排序接口，等待后续产品阶段再进入后端契约设计。
- 进入工作区入口在本次是否只跳转到占位页，还是等待内容树画布变更统一实现；本次不阻塞种子库闭环。
