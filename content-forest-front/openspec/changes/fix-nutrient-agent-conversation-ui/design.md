## Context

营养工作台目前已经有三栏布局、研究会话、乐观用户消息、SSE 读取和可沉淀营养块展示。但用户感受到的问题集中在真实对话体验：输入后不是完整流式输出，thought 块一直处理中且不可折叠，工具调用不可见，运行中无法暂停，新建会话刷新后消失，局部空白态破坏工作台连续性。

该变更依赖后端 `fix-nutrient-agent-conversation-backend` 提供更完整的 SSE 事件和种子级会话列表接口。前端只消费 `docs/api/nutrient.yaml` 定义的契约，不自行推断数据库事实。

## Goals / Non-Goals

**Goals:**
- 将营养研究中间区改造成类似主流 AI 对话产品的完整流式体验。
- 按事件类型渲染 thought、tool、assistant message 和可沉淀营养。
- 运行中提供暂停按钮，停止当前流式请求。
- 支持种子级会话列表、刷新恢复和新会话切换。
- 用更精致的工作台视觉语言重做输入框、按钮、消息块、工具块、思考块和成果卡片。

**Non-Goals:**
- 不在前端本地持久化会话事实。
- 不展示模型未公开的私有思维链。
- 不实现复杂多会话并行运行；本期一个工作台实例一次只处理一个活动流。
- 不修改营养内容生命周期的后端规则。

## Decisions

1. 使用前端流式消息状态机管理事件。
   - 将当前会话拆为消息区块：user、assistant、thought、tool、depositable。
   - SSE 事件按类型追加到对应区块，最终 `done` 事件负责确认系统事实。
   - 这样比把所有 progress 写进同一个 assistant message 更稳定。

2. thought 独立为可折叠块。
   - thought 块默认可收起，运行中显示轻量状态，收到完成/错误/取消后结束处理中状态。
   - 不把 thought 当作普通 assistant 正文，避免混淆“思考”和“回答”。

3. 工具调用使用时间线卡片展示。
   - 工具开始、进行中、完成、失败展示为紧凑事件块。
   - 工具块只显示工具名称、状态、摘要和耗时等用户可理解信息。
   - 该设计贴近 OpenClaw/Codex 类 Agent UI，但不暴露工程日志。

4. 发送按钮在运行中切换为暂停按钮。
   - 提交时创建 `AbortController`。
   - 用户暂停时 abort 当前请求，并将当前 assistant 占位切换为取消/已停止状态。
   - 取消后允许用户重试或继续输入下一条消息。

5. 会话恢复以服务端列表为准。
   - 打开工作台时先加载种子级会话列表，优先恢复最近会话或当前卡片绑定会话。
   - 新建会话成功后立即进入会话列表，不依赖浏览器临时状态。
   - 切换卡片时仍优先使用卡片绑定会话。

## Risks / Trade-offs

- [后端事件契约未完成] → 前端在提案中明确依赖后端变更，实施时先对齐 `docs/api/nutrient.yaml`。
- [流式事件乱序或重复] → 前端以 event type 和本地临时 id 合并，最终以后端持久化消息替换临时内容。
- [UI 信息量过大] → thought/tool 默认紧凑展示，可展开查看详情，普通回复和可沉淀营养保持主视觉权重。
- [暂停无法真正中止下游] → 前端以用户操作反馈为准，后端完成取消传播后再保证资源释放。
