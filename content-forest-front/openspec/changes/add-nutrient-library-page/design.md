## Context

前端当前已经具备内容森林主工作台外壳、种子库页面、生成器页面和工作区 mock 体验。营养库领域文档已经明确：营养库负责组织公共营养和种子专属营养，营养内容正文使用 Markdown，系统事实由数据库维护，已归档库和内容不可被新的枝化生长引用。

本次前端变更需要把 `/nutrients` 从占位或不可用入口升级为可操作的营养库管理页面。页面应遵守 `docs/spec/DESIGN.md` 的 Quiet Command Workspace 风格，并严格依赖后端 `docs/api/nutrient.yaml` 与 `docs/sql/nutrient.sql` 契约。若后端契约尚未落地，前端实现阶段应先等待或同步补齐该依赖，不应自行发明系统事实字段。

## Goals / Non-Goals

**Goals:**

- 提供营养库管理页面，支持公共营养库和种子专属营养库视图。
- 支持创建公共营养库，支持创建种子专属营养库并选择归属种子。
- 支持查看营养库详情，并编辑营养库名称和可选描述。
- 支持营养库归档和回档，并在界面上清晰表达归档后的只读或不可引用语义。
- 支持在营养库下新增、查看、编辑、归档和回档 Markdown 营养内容。
- 为后续工作区 `@营养` 引用预留按种子查询可引用营养内容的数据访问封装。
- 对齐后端契约，不在前端状态中创造作用域、归属、归档状态或内容位置等系统事实。

**Non-Goals:**

- 不实现向量检索、语义搜索、知识库问答或自动网页采集。
- 不支持文件上传、附件解析、富文本编辑或网页导入。
- 不实现营养内容有效性判断，也不调用 Agent。
- 不把基因库内容或基因汲取结果混入营养库页面。
- 不提供营养库独立管理页面之外的工作区 `@营养` 完整交互，本次只预留数据访问能力。
- 不修改后端实现、SQL 或 OpenAPI 文档，但必须在任务中显式标记后端契约依赖。

## Decisions

### Decision 1: `/nutrients` 采用管理型双栏工作台布局

页面左侧展示营养库集合，右侧展示当前营养库详情和营养内容列表。公共营养库与种子专属营养库通过页面内视图切换或分组呈现，不拆成两个全局导航入口。

选择该方案的原因是营养库是一个工作台模块，公共和专属只是作用域差异，不应抬高为两个顶级页面。双栏布局可以让用户在浏览库列表时快速查看内容，符合当前种子库和生成器页面的管理型体验。

备选方案是表格列表加独立详情页，但会打断资料维护流程，也不符合第一期紧凑工作台风格。

### Decision 2: 创建种子专属营养库时复用种子列表选择能力

创建弹窗中先选择营养库作用域。公共作用域只需要名称和可选描述；种子专属作用域还需要选择归属种子。归属种子创建后不可修改，编辑入口只允许修改名称和描述。

选择该方案的原因是后端领域约束已经确认作用域与归属创建后不可变。前端应在表单层提前表达该约束，减少用户误操作。

备选方案是允许在编辑时改绑种子，但会违反领域约束，并让历史引用语义不稳定。

### Decision 3: 营养内容编辑使用 Markdown 查看态与编辑态

营养内容详情中使用统一 Markdown 渲染组件展示正文，编辑时切换为文本编辑控件，提交完整 Markdown 正文。第一期只支持手写或复制粘贴，不提供上传按钮。

选择该方案的原因是营养内容本体就是 Markdown 文本，和种子、果实的内容展示方式保持一致，可以复用统一渲染和安全策略。

备选方案是引入富文本编辑器或文件上传，会增加第一期复杂度，并偏离“轻量资料库”的目标。

### Decision 4: 归档资源仍可查看，但不可作为新引用候选

页面需要支持未归档和已归档视图。已归档营养库和已归档营养内容仍可查看和回档，但应在交互上弱化，并避免出现在可引用营养查询结果中。

选择该方案的原因是用户明确要求不删除，只归档；同时领域规则要求归档资源不可被枝化生长引用。前端展示和数据访问层都应体现该边界。

备选方案是归档后完全隐藏，会让用户难以恢复；删除则会引入更多数据一致性风险。

### Decision 5: 可引用营养查询封装独立于页面展示列表

页面管理列表用于维护营养库与营养内容；工作区 `@营养` 需要的是“给定种子，返回可引用营养内容”。前端应为后者预留独立数据访问函数，严格调用 `docs/api/nutrient.yaml` 中的可引用营养查询接口。

选择该方案的原因是管理列表与引用候选的筛选语义不同。把两者分开可以避免工作区误用已归档内容或其他种子的专属内容。

备选方案是在工作区直接复用页面列表筛选，会把作用域规则散落到多个组件中，后续容易不一致。

## Risks / Trade-offs

- [Risk] 后端 `docs/api/nutrient.yaml` 和 `docs/sql/nutrient.sql` 尚未落地时前端字段不稳定 → Mitigation：任务中先校验契约存在；实现只使用契约字段，缺口回到后端提案补齐。
- [Risk] 第一版只支持 Markdown 粘贴会让大量资料录入效率较低 → Mitigation：页面保留轻量创建路径，后续通过单独提案扩展上传和导入能力。
- [Risk] 种子专属营养库创建时需要选择种子，列表过多时选择成本增加 → Mitigation：第一期复用现有种子列表能力，可先提供基础筛选，复杂搜索后续扩展。
- [Risk] 已归档库和内容都可查看，状态层级可能让用户困惑 → Mitigation：在列表、详情和操作按钮中保持一致的归档状态标识，并只提供回档操作。
- [Risk] 页面管理列表和工作区引用查询语义不同 → Mitigation：前端数据访问层将管理接口和可引用查询接口分开封装，组件不得自行拼接筛选规则。

## Migration Plan

本次为新增前端能力，无数据迁移。实现时先对齐后端营养库 API 和 SQL 契约，再建立前端数据访问封装，最后落地 `/nutrients` 页面。若需要回滚，可以恢复营养库占位页并移除新增营养库前端模块，不影响种子库、生成器和工作区。

## Open Questions

- 后端是否会在 `docs/api/nutrient.yaml` 中提供按名称或关键词筛选能力；若没有，第一期前端只做本地基础筛选或不做搜索。
- 工作区 `@营养` 引用的完整交互属于后续工作区真实接入阶段，本变更只提供可复用的数据访问封装。
