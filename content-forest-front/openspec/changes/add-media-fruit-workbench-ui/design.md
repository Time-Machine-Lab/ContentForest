## Context

当前工作区的果实详情以 Markdown 为主体，枝化输入框已经支持营养、基因和临时营养卡片引用，并支持失败任务输入恢复。多媒体能力需要在此基础上扩展：媒体资源是独立资产，果实展示媒体挂载，枝化输入提交媒体引用和用途说明。

前端不应假设上传视频后 Agent 就能理解视频；是否能处理媒体由后端和 Agent 工具能力决定。

## Goals / Non-Goals

**Goals:**

- 展示果实挂载的图片和视频资源。
- 支持用户上传图片/视频为媒体资源。
- 支持枝化生长引用媒体资源并填写用途说明。
- 提交生长任务时包含 mediaRefs，并与 nutrientRefs、temporaryNutrientCardRefs、geneRefs 共存。
- 失败时保持输入和已上传媒体可感知。
- 最近失败输入恢复时恢复仍可访问的媒体引用。

**Non-Goals:**

- 不制作纯 HTML preview。
- 不做图片编辑、视频剪辑、抽帧或转写 UI。
- 不承诺前端能判断 Agent 是否理解视频。
- 不将媒体直接内嵌为 Markdown 系统 meta。

## Decisions

### Decision 1: 媒体展示在果实详情资产区

果实正文仍是 Markdown。媒体资源展示在独立资产区，图片用画廊，视频用播放器或可打开的预览入口。

### Decision 2: 上传后先获得 Media Asset

用户在枝化输入框上传文件后，前端先调用媒体上传接口获得 mediaAssetId，再在生长任务中提交 mediaRefs。

上传成功后的媒体资源进入当前工作区资源引用模型，与营养、基因、临时营养卡片使用同一套引用标签、移除和提交映射机制。

### Decision 3: 媒体用途是引用的一部分

每个 mediaRef 必须携带用途说明。第一版使用语义化选项和可选备注，而不是复杂参数。

用途说明不等同于前端判断 Agent 能力；它只是告诉后端和 Agent 本轮希望如何使用该资源。

### Decision 4: 能力不足要提示

如果后端返回某类媒体当前不能被 Agent 理解，前端应提示“可引用/展示，但不保证理解”，不阻塞上传和展示。

### Decision 5: 失败输入恢复保守处理媒体

最近失败任务恢复时，前端应尝试恢复失败输入中的媒体引用；如果某个媒体资源已不可访问，则展示可理解提示并跳过该引用，不阻塞其他输入恢复。

理由：媒体文件可能被存储策略、权限或后端校验影响，恢复流程不能因此破坏用户上次输入。

## Risks / Trade-offs

- [Risk] 上传控件让枝化输入框变复杂 → Mitigation: 媒体引用放入枝化详情或资源浮层，不挤占正文输入。
- [Risk] 大文件上传体验差 → Mitigation: 第一版展示上传中、失败和大小限制提示。
- [Risk] 用户混淆媒体展示与 Agent 处理能力 → Mitigation: 用途选择和能力提示分开展示。
- [Risk] 媒体引用破坏现有资源标签逻辑 → Mitigation: 媒体作为新资源类型接入现有引用模型，而不是另做一套独立状态。
