## Context

营养工作台当前采用“左侧营养内容、中间 Agent、右侧建议”的三栏结构。随着会话恢复、流式输出、可沉淀营养块和营养内容生命周期能力逐步加入，原结构出现了职责混合：营养内容既是资产又像会话入口，建议常驻占据右栏但使用频率不一定高，可沉淀营养块的合并目标依赖“当前选中内容”这种隐式状态。

本次前端设计将工作台改为更接近真实生产力工具的信息架构：左侧负责研究过程，右侧负责资产库，中间负责当前对象的主工作流。视觉方向遵循 `ui-ux-pro-max`：克制、清晰、按钮状态明确，参考 Mobbin/Refero 的真实产品信息架构，并吸收 SiteInspire/Godly/CSS Design Awards/The FWA 中的高级视觉层级，但不做营销页式装饰。

重要约束：前端不得修改 API 和 SQL。当前 `docs/api/nutrient.yaml` 仍保留会话与营养内容关联字段，且缺少删除会话接口；实现阶段必须等待后端契约更新，或将对应按钮置为依赖状态。

## Goals / Non-Goals

**Goals:**

- 在 UI 上明确区分“研究会话”和“营养内容资产”。
- 让会话可以作为独立对象被创建、切换、删除。
- 让营养内容可以作为独立资产被搜索、筛选、查看、沉淀、归档和删除。
- 让汲取建议成为轻量消息入口，而不是常驻栏位。
- 让可沉淀营养块的保存和合并目标显式化，避免依赖当前选中内容。
- 在桌面和窄屏中保持主要操作可见、可理解、不卡死布局。

**Non-Goals:**

- 不在前端实现本地伪删除或伪解耦。
- 不修改 `docs/api/*.yaml` 或 `docs/sql/*.sql`。
- 不改变 Agent 研究算法、SSE 事件语义和营养内容后端生命周期规则。
- 不新增外部 UI 框架或复杂动画依赖。

## Decisions

1. 左栏改为研究会话栏。
   - 会话是研究过程，应作为用户进入对话的主入口。
   - 新会话入口只创建前端草稿态，不立即调用后端；发送第一条消息时再创建真实会话。
   - 备选方案：保留顶部横向会话条。放弃原因是横向条在长标题和多会话时容易压缩主工作区，并且像浏览器标签页，不适合承载删除、状态和更新时间。

2. 右栏改为营养内容栏。
   - 营养内容是沉淀资产，应支持搜索、筛选、状态化操作。
   - 搜索和筛选第一期只做前端本地过滤，避免扩大 API 范围。
   - 备选方案：营养内容继续放左栏。放弃原因是左栏更适合作为“当前工作流入口”，而会话是研究工作流的更自然入口。

3. 中间区使用多功能主工作区。
   - 当前选中会话时显示对话流。
   - 当前选中营养内容时显示资产详情。
   - 当前选择新会话时显示空白对话态。
   - header 负责表达当前模式和全局轻操作，例如汲取建议消息入口。

4. 汲取建议迁移到 header 消息按钮。
   - 建议是“提醒/待处理事项”，不是三栏常驻工作对象。
   - 浮层内展示建议卡片，支持采纳和忽略。
   - 采纳后不直接创建会话，而是切到新会话草稿态并填入输入框。

5. 可沉淀营养块必须显式选择目标。
   - 保存为新草稿保持原能力。
   - 合并时打开目标选择器，用户选择某个营养内容后再合并。
   - 如果后端暂时只支持合并到当前选中卡片，则前端任务必须标注依赖后端更新，不能继续使用隐式“当前内容”文案。

6. 删除会话是前后端共同能力。
   - 前端只负责入口、确认和状态刷新。
   - 真正删除必须依赖后端 API。
   - 删除会话不得影响已沉淀营养内容。

7. 视觉系统采用“Quiet Command Workspace”。
   - 三栏都应是工作台，不做漂浮卡片堆叠。
   - 按钮使用熟悉的操作语义：加号用于新会话，搜索框用于资产过滤，删除使用危险态。
   - hover、focus、disabled、loading 必须可感知。
   - reduced-motion 下禁用非必要动画。

## Risks / Trade-offs

- [后端契约未完成] → 将删除会话、会话内容解耦、可沉淀块目标合并标为依赖后端更新，前端实现前必须确认 `docs/api/nutrient.yaml`。
- [多模式主工作区让用户迷路] → header 必须显示当前模式、当前对象标题和主要动作；左右栏选中态必须稳定。
- [右侧营养内容本地过滤在数据量大时不足] → 第一版保持本地过滤，后续再升级为后端搜索。
- [删除操作误伤研究过程] → 删除会话必须有确认；删除营养内容和删除会话文案必须明确互不影响。
- [视觉过度设计影响效率] → 设计参考高级站点的层级和节奏，但以真实产品界面为主，不使用纯装饰背景和大面积展示型视觉。

## Migration Plan

1. 等待后端/API 明确会话与营养内容解耦、删除会话、目标合并能力。
2. 前端先更新 API/types，并保留对旧字段的兼容读取但不再作为 UI 主路径展示。
3. 重构 `NutrientWorkbenchDialog.vue` 的布局状态：会话栏、主工作区、营养内容栏。
4. 迁移汲取建议到 header 浮层。
5. 替换可沉淀营养块操作为显式目标模式。
6. 更新静态测试、API 测试、类型检查和桌面/窄屏视觉验证。

## Open Questions

- 会话删除后，后端是否会返回被解除关联的旧营养内容数据，还是前端只刷新会话列表即可？
- 营养内容“删除”是否仅允许草稿删除，还是已归档也允许物理删除？
- 可沉淀营养块合并到目标内容时，后端是否提供专用 merge endpoint，还是继续由前端读取目标 markdown 后调用 update？
