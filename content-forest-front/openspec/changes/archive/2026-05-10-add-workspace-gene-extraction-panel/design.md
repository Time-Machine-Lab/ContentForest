## Context

工作区是用户围绕某个种子查看内容树、选择/淘汰果实、发起枝化生长和复盘经验的主现场。基因汲取如果分散到果实详情、发布记录、数据反馈等页面，会让用户难以理解“当前到底有什么经验值得沉淀”。后端将通过工作区快照提供统一的基因汲取中心数据，前端需要把它呈现为工作区内的统一提示和操作组件。

本变更只设计前端展示与操作编排。基因汲取提醒、任务、建议、正式基因经验仍由后端 Gene API 管理，工作区快照负责提供首屏可展示的摘要数据。

## Goals / Non-Goals

**Goals:**

- 在种子工作区增加统一基因汲取提示与操作组件。
- 组件完全消费 `GET /api/seeds/{seedId}/workspace` 返回的基因汲取中心数据。
- 将汲取提醒、待确认建议、基因库入口和经验摘要收纳在同一区域。
- 从统一组件中发起汲取、忽略提醒、查看/编辑/确认/放弃建议。
- 保持发布操作和果实详情的职责纯净，不新增分散汲取入口。

**Non-Goals:**

- 不新增全局基因库侧边栏页签。
- 不由前端根据果实状态或发布状态自行判断是否提示汲取。
- 不在发布操作成功后自动触发基因汲取。
- 不替代枝化生长输入框中的 `@基因库` 引用能力。

## Decisions

### Decision 1: 工作区提供一个统一基因汲取组件

组件可以以右上角提示区、可收纳浮层或工作区侧向轻面板形式存在，但入口必须唯一，且属于工作区层级。它展示后端返回的提醒、建议和基因库摘要，并承载后续操作。

Alternative considered: 在果实详情中增加“汲取基因”按钮。该方式会使入口跟随节点详情分散出现，且容易让前端按节点状态自行推断提醒。

### Decision 2: 前端只消费后端快照，不做提醒推断

前端不基于 selectionState、publication 状态或反馈数据自行计算提醒。只要后端基因汲取中心没有返回待处理项，组件就展示空态或低优先级摘要。

Alternative considered: 前端在选择/淘汰果实后即时本地生成提醒。该方式会让前端拥有领域判断，且与后端提醒状态容易不一致。

### Decision 3: 发布操作不进入基因汲取流程

发布器仍只负责发布验证。发布成功或编辑发布记录后，前端不打开基因汲取组件、不自动创建提醒、不自动发起任务。

Alternative considered: 发布后弹出“是否汲取基因”。第一期人工发布和人工反馈数据强度不足，该交互会把发布验证和经验沉淀混在一起。

### Decision 4: 基因库入口保留种子上下文

不新增全局基因库页签。基因库入口应从种子卡片或工作区当前种子上下文进入，工作区内的统一组件可以展示当前种子基因库摘要和入口。

Alternative considered: 建立全局基因库页面作为主导航。当前基因库是种子级能力，全局入口会弱化种子上下文，也容易让用户误以为基因可跨种子通用。

## Risks / Trade-offs

- [Risk] 统一组件承载内容过多影响画布空间 → Mitigation: 默认展示摘要和最高优先级提示，详情通过展开层或抽屉处理。
- [Risk] 后端字段未就绪导致前端难以联调 → Mitigation: 以 `docs/api/workspace.yaml` 为准建立类型，开发时提供空态兼容。
- [Risk] 用户找不到基因库入口 → Mitigation: 在种子卡片和工作区当前种子上下文提供入口，但不放入全局导航。
- [Risk] 汲取操作与枝化输入框中的 `@基因库` 概念混淆 → Mitigation: 统一组件负责沉淀经验，枝化输入框负责引用已沉淀经验。

## Migration Plan

1. 等待后端变更更新 `docs/api/workspace.yaml` 的基因汲取中心契约。
2. 更新前端工作区快照类型和 gene API 调用封装。
3. 在工作区页面加入统一基因汲取组件，并接入快照数据。
4. 移除或避免新增果实详情、发布记录中的分散汲取入口。
5. 补充工作区交互测试，覆盖提醒展示、发起汲取、忽略提醒、建议处理和发布不触发汲取。

## Open Questions
