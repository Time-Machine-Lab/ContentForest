## Context

当前前端已经具备内容森林工作台外壳和种子库相关页面，`/generators` 仍处于占位状态。后端生成器模块已经通过 `docs/api/generator.yaml` 暴露生成器导入、列表、详情、启用、停用、重新上传和可选生成器查询契约，并通过 `docs/sql/generator.sql` 明确生成器系统事实边界。

本变更只负责前端生成器管理体验。页面参考 `docs/design/previews/generator-module-preview.html`，采用 Quiet Command Workspace 风格：左侧全局导航、顶部命令区、主列表、右侧详情面板。生成器页面定位为资源管理台，不是生成器市场，也不是枝化生长入口。

## Goals / Non-Goals

**Goals:**

- 将 `/generators` 做成可用的生成器管理页面。
- 支持查看生成器列表，并在页面内部按启用、停用、全部进行筛选。
- 支持搜索生成器名称和描述。
- 支持选择生成器并查看详情，包括 Skill Markdown、文件条目、内容位置和系统状态。
- 支持导入生成器、重新上传 Skill、启用和停用生成器。
- 所有 API 调用严格对齐 `docs/api/generator.yaml`。
- 视觉和交互遵守 `docs/spec/DESIGN.md`，保持紧凑、暗色、工具化。

**Non-Goals:**

- 不在生成器管理页触发枝化生长。
- 不展示“用于枝化生长”主按钮；生成器选择属于后续工作区生长输入框。
- 不在线编辑 Skill Markdown 或 Skill 文件。
- 不实现生成器市场、评分、付费、推荐、模板商店等能力。
- 不修改 `docs/api/*.yaml` 或 `docs/sql/*.sql`。
- 不定义后端数据结构，也不绕过既有 API 契约。

## Decisions

### 1. 页面采用“列表 + 详情面板”结构

生成器管理是高频资源浏览和状态管理场景，列表加详情面板能减少页面跳转，并与种子库和工作台外壳保持一致。主列表承载名称、描述、状态和快捷操作；右侧详情展示更完整的 Skill 信息与系统事实。

替代方案是传统表格页。该方案信息密度高，但会让页面更像后台管理系统，不符合内容森林的工作台气质，也不利于展示 Skill Markdown 和文件条目。

### 2. 导入生成器从页面主操作进入

页面顶部保留“导入生成器”作为主操作入口。导入流程应使用 Modal、Drawer 或同等级覆盖层承载 zip 选择、名称、描述、校验结果和提交操作，而不是常驻占据列表顶部。

这样能让页面默认态聚焦在已导入生成器管理上，也避免导入表单在没有导入意图时造成视觉负担。

### 3. 启用、停用、重新上传是管理动作

启用和停用用于控制生成器是否进入后续枝化生长可选列表。重新上传用于替换 Skill 本体，但保持生成器作为同一个受管理资源存在。前端在操作成功后必须以接口返回结果更新当前列表和详情，不基于乐观假设伪造系统事实。

### 4. 生成器页面不承担枝化生长选择

虽然后端提供 `GET /api/generators/selectable`，但该接口主要服务工作区生长输入框。生成器管理页可通过启用状态让用户理解哪些生成器可被后续选择，但不应在详情中提供“用于枝化生长”按钮。

### 5. Skill 内容只展示，不编辑

`GeneratorDetail.skillMarkdown` 和 `GeneratorDetail.entries` 用于详情展示。第一期不做在线编辑，用户需要修改 Skill 时使用重新上传能力。这样符合生成器 Skill 作为外部独立文件夹的边界，也避免前端引入复杂文件编辑器。

## Risks / Trade-offs

- [Risk] `POST /api/generators` 和 `POST /api/generators/{generatorId}/reupload` 当前契约使用 `zipBase64`，大文件会带来前端内存和传输压力。→ Mitigation：第一期按契约实现，并在 UI 中限制或提示合理文件大小；后续如需 multipart 上传，由后端先更新 API 契约。
- [Risk] 列表接口只返回摘要，详情需要额外请求。→ Mitigation：选择列表项时再请求详情，保持列表轻量；详情失败时不影响列表浏览。
- [Risk] 启停操作失败时用户可能误以为状态已改变。→ Mitigation：写操作进行中禁用对应按钮，失败时保留原状态并展示局部错误反馈。
- [Risk] 停用生成器仍可查看，容易和“不可用于新生长”混淆。→ Mitigation：用状态标签表达停用，并在管理动作中只提供重新启用，不提供删除。
