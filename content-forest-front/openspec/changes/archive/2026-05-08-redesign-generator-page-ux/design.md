## Context

`/generators` 当前已经接入生成器管理接口，并通过列表、右侧详情、导入弹窗、重传弹窗完成基础管理能力。问题主要集中在前端体验层：

- 导入 Skill zip 的入口弱，上传控件样式原生且不支持拖拽。
- 字段缺失时反馈不够精准，用户不知道应该补齐哪个元素。
- 生成器列表不够适合扫描，名称、描述、状态、位置、更新时间和操作之间缺少清晰层级。
- 详情区域展示能力存在，但信息结构偏平，文件条目、Markdown 和系统事实的阅读效率不高。

本次设计参考 `docs/design/previews/generator-page-redesign-preview.html` 的三栏结构，但视觉落地必须符合 `docs/spec/DESIGN.md` 的 Quiet Command Workspace：深色优先、紧凑、专业、低噪声、避免营销式 hero 和过重装饰。

## Goals / Non-Goals

**Goals:**

- 将生成器页面重构为「Skill 导入区 / 生成器浏览区 / Skill 详情区」的工作台结构。
- 让导入生成器支持拖拽上传、点击上传、文件选中反馈和字段级缺失提示。
- 提高生成器浏览效率，让用户可以快速搜索、筛选、选择、重传、启用或停用生成器。
- 优化详情区信息层级，让 Skill Markdown、文件条目和系统事实更容易查看。
- 保持所有数据读取与写入严格对齐 `docs/api/generator.yaml` 既有接口。

**Non-Goals:**

- 不新增、修改或删除任何后端 API。
- 不修改 `docs/sql/generator.sql` 或任何数据库结构。
- 不实现生成器市场、评分、推荐、运行测试、枝化生长入口或 Skill 在线编辑。
- 不改变生成器领域语义：生成器仍然是 Skill 文件夹，页面只负责上传、浏览、启停和重传。

## Decisions

### 1. 使用三栏工作台结构，而不是继续强化弹窗导入

页面主体采用：

- 左侧：`Skill Drop Lab`，承载导入生成器的主要流程。
- 中间：生成器浏览区，承载搜索、状态筛选和生成器卡片。
- 右侧：`Skill Inspector`，承载选中生成器详情和管理动作。

这样导入能力常驻页面，不再隐藏在单一弹窗里；用户进入生成器页面即可知道如何导入、如何浏览、如何查看详情。

替代方案是继续保留顶部按钮 + 弹窗导入，但这不能解决“导入不方便、缺少即时提示”的核心痛点。

### 2. 拖拽上传作为前端交互增强，不改变接口契约

拖拽上传只改变文件选择方式。前端仍将 zip 文件转换为 base64，并调用：

- `POST /api/generators`
- `POST /api/generators/{generatorId}/reupload`

请求体仍使用 `name`、`description`、`zipBase64` 或 `zipBase64`，不新增上传协议。

### 3. 浏览区使用卡片矩阵，但保留紧凑工具属性

生成器浏览区从传统行列表调整为卡片矩阵或紧凑卡片列表。卡片必须展示：

- 名称
- 描述
- 启用/停用状态
- 内容位置或更新时间等既有信息
- 详情、重传、启停等管理入口

卡片样式需要借鉴 preview 的空间组织，但最终颜色、圆角和密度必须回归当前设计系统，避免过度拟物、过大圆角、强渐变或装饰性插画。

### 4. 详情区按阅读任务分层

详情区不再只作为字段堆叠，而按用户阅读目的分区：

- Skill 概览：名称、描述、启停状态。
- Skill Markdown：使用既有 MarkdownViewer。
- 文件条目：展示 `GeneratorDetail.entries`。
- 系统事实：展示 `contentLocation`、`enableState`、`updatedAt`、`disabledAt`。
- 管理动作：重传、启用、停用。

### 5. 字段级错误优先于全局错误

导入表单在本地提交前校验：

- 缺少名称
- 缺少描述
- 缺少 zip 文件

每个缺失项在对应区域展示清晰提示。接口失败仍使用页面既有异常反馈组件，但不得清空用户已填写内容。

## Risks / Trade-offs

- **三栏布局在窄屏空间不足** → 使用响应式布局降级为单列或双列，详情区可下移，保证导入和浏览仍可用。
- **卡片矩阵可能降低超长列表密度** → 保留搜索、状态筛选和紧凑卡片尺寸；后续如生成器数量很大，可再引入虚拟列表。
- **拖拽上传增加状态复杂度** → 将文件读取、拖拽状态和字段错误限制在导入组件内部，不扩散到 API client。
- **视觉过度偏离设计系统** → 以 `docs/spec/DESIGN.md` 为最终约束，preview 只作为结构和体验参考，不直接照搬高饱和色彩和大装饰。
