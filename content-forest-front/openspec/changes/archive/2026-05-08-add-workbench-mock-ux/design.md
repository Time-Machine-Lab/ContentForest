## Context

前端当前已经具备种子库页面和进入 `app/pages/seeds/[seedId]/workspace.vue` 的路由入口，但工作区页面仍是占位状态。用户已经通过 `docs/design/previews/content-forest-workbench-proposal-preview.html` 确认了一版工作区交互方向：深色全屏画布、内容树节点、真实父子连接线、右侧详情面板、Codex 风格底部枝化输入框和 mock 生长反馈。

本次变更只实现 UI/UX 和前端 mock 交互，不接入真实后端接口、不提交真实生长任务、不创建真实果实。Mock 数据需要参考 `docs/api/seed.yaml`、`docs/api/fruit.yaml` 和对应 SQL 中已有的字段语义，尤其是种子根节点、果实父节点引用、果实物竞天择状态和 Markdown 内容边界。

## Goals / Non-Goals

**Goals:**

- 将种子工作区从占位页升级为可体验的 mock 工作区页面。
- 保持从种子库进入工作区的现有入口和路由语义。
- 用 mock 数据构建种子、果实、父子边、发布反馈、基因标签和引用资源。
- 实现内容树画布拖拽浏览、节点拖拽、动态 SVG 连接线和节点状态视觉。
- 实现右侧详情面板，展示种子/果实 Markdown、基因标签、发布反馈和物竞天择操作。
- 实现 Codex 风格底部枝化生长输入框，并按果实状态控制显示规则。
- 实现 mock 生长动画和新果实追加，用于演示未来枝化生长闭环。

**Non-Goals:**

- 不调用 `docs/api` 中的真实接口，不新增前端 API 封装。
- 不实现真实 Agent、真实生成器、真实营养库检索或真实基因库读取。
- 不实现真实生长任务状态、发布器、监控器或数据回流接口。
- 不修改种子库现有 API 流程，不修改后端、SQL 或 OpenAPI 文档。
- 不引入大型画布或节点编辑器依赖；本次优先用 Vue 状态、CSS 和 SVG 完成原型级交互。

## Decisions

### Decision 1: 工作区页面直接承载 mock 原型体验

本次优先改造 `app/pages/seeds/[seedId]/workspace.vue`，让用户从种子卡片进入后直接看到可交互内容树。页面内部可以组合拆分到 `src/modules/workspace`、`src/modules/fruit`、`src/modules/growth` 等模块，但用户可感知入口保持现有路由。

选择该方案的原因是当前用户要验证的是工作区整体 UX，而不是单个组件库。先把完整路径跑通，可以更快暴露画布、详情面板和输入框之间的交互问题。

备选方案是先拆组件再组页面，但在原型阶段容易过早抽象，增加迭代成本。

### Decision 2: Mock 数据遵循后端契约语义，但不伪装成接口响应

工作区 mock 数据在前端本地维护，结构上参考 `SeedRootNode`、`FruitSummary`、`FruitDetail`、`ParentNodeRef` 和 `selectionState` 的语义。Mock 数据只作为本次演示状态，不进入 API 层，不命名为真实系统事实来源。

选择该方案的原因是提案必须尊重现有 API/SQL 语义，同时本次明确不与后端对接。把 mock 数据放在工作区模块内部，可以避免未来接真实接口时误用 mock 状态。

备选方案是新增 mock API 层，但会让后续替换真实接口时边界变复杂，也容易让组件误以为已经有可用后端契约。

### Decision 3: 内容树连接线由父子关系实时计算

内容树节点与边分开维护：节点保存位置和展示状态，边保存父节点与子节点引用。SVG path 根据当前 DOM 或节点位置计算，并在节点拖拽、新果实生成、视图重置后更新。

选择该方案的原因是连接线必须表达真实树结构，不能是静态背景。动态计算可以保证拖拽节点后树边仍然正确连接。

备选方案是使用静态 SVG 或 CSS 背景，视觉成本低但会误导用户；使用完整图形库则当前阶段成本过高。

### Decision 4: 节点点击与节点拖拽区分处理

节点支持点击选中和拖拽移动。交互上通过 pointer 位移阈值区分点击与拖拽：轻点打开详情，移动超过阈值则进入拖拽，不触发选中切换。

选择该方案的原因是画布类产品中节点拖拽和详情查看都高频，必须避免拖动结束误触详情。

备选方案是只允许画布拖拽、不允许节点拖拽，但会削弱工作区探索感。

### Decision 5: 枝化输入框只属于已选择果实

只有 `selectionState = selected` 的果实被选中时，底部枝化输入框才出现。候选果实、已淘汰果实和种子节点只展示详情与对应状态操作，不展示输入框。

选择该方案的原因是当前用户明确要求第一版只允许已选择果实进入枝化生长操作区。这样能把“物竞天择之后再继续繁衍”的产品逻辑表达清楚。

备选方案是种子和所有果实都可生长，但会扩大第一期交互范围，也和当前用户确认的 UI 规则冲突。

### Decision 6: 物竞天择按钮按状态动态渲染

候选果实显示“选择”和“淘汰”；已选择果实不显示按钮，只展示进入发布验证与数据回流的状态提示；已淘汰果实只显示“恢复”。这些操作只更新前端 mock 状态。

选择该方案的原因是三个按钮同时平铺会制造错误暗示，例如已选择果实还可以继续选择。动态渲染更符合领域状态机。

备选方案是始终显示三个按钮但禁用不适用项，视觉更稳定但信息噪声更大。

### Decision 7: 枝化生长输入框采用 Codex 风格命令面板

输入框采用大块圆角暗色输入面板：顶部展示只读生长源、生成器、果实数量和突变率；中间承载意图输入与 `@` 引用高亮；底部承载添加入口、引用 chip、枝化详情按钮和发送按钮。生成器和果实数量本次不可选择、不可编辑。

选择该方案的原因是枝化生长是工作区的核心命令面，参考 Codex 输入框可以让“输入意图 + 引用上下文 + 发起动作”的心智更直接。

备选方案是传统表单式参数区，但会削弱命令输入的连续性。

### Decision 8: 枝化详情以局部小面板展示引用资源

点击“枝化生长详情”在输入框附近弹出小面板，展示生成器、果实数量、突变率、引用营养库和引用基因库。该面板是当前输入框的上下文详情，不跳转页面、不打开全屏弹窗。

选择该方案的原因是用户需要快速确认引用和参数，小面板能保持工作区上下文。

备选方案是右侧详情面板切换 tab，但会和当前选中果实详情争夺空间。

## Risks / Trade-offs

- [Risk] Mock 交互可能让用户误以为已接入真实生成 → 在实现和文案中保持“预览/mock”边界，避免调用 API 或写入真实系统事实。
- [Risk] 纯手写 SVG/拖拽逻辑后续可能不适合复杂树 → 本次作为第一期原型实现，后续如果需要缩放、自动布局、成百上千节点，再评估图形库。
- [Risk] 输入框仅对已选择果实显示会限制从种子直接生长 → 这是本次用户确认的交互规则，后续如恢复种子生长入口，需要单独提案调整。
- [Risk] DOM 位置计算连接线在响应式布局中可能出现偏差 → 实现时在窗口尺寸变化、节点拖拽和新节点生成后统一刷新连接线。
- [Risk] Preview 样式直接照搬可能与现有组件样式冲突 → 实现时以 `docs/spec/DESIGN.md` 为准，把样式收敛进工作区模块，避免全局污染。

## Migration Plan

本次无数据迁移。实现时先保留种子库入口与路由，替换工作区占位内容。若需要回滚，可以恢复 `workspace.vue` 占位页并移除新增工作区模块文件，不影响种子库与后端接口。

## Open Questions

- 后续真实接口接入时，内容树一次性读取接口或果实子树查询接口尚未在 `docs/api` 中定义，需要后端领域提案补齐。
- 后续是否恢复“种子节点也可枝化生长”的交互，需要基于第一版体验再确认。
