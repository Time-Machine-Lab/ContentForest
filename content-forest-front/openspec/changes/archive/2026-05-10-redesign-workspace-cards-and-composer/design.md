## Context

种子工作区页面已经具备内容树画布、节点卡片、右侧详情和底部枝化生长输入框。当前问题集中在两个局部 UI 面：

- 节点卡片需要更高级、更克制，同时保留种子、果实、候选、已选择、已淘汰、生成中、失败等原有概念和逻辑。
- 底部输入框中的生成器/果实下拉、`@` 引用资源候选浮层和已引用资源标签需要提升可用性，尤其是下拉过长、候选项与输入内容重叠、已引用资源无法移除。

本次变更以 `docs/design/previews/workspace-card-input-redesign-preview.html` 为视觉参考，只作用于 `app/pages/seeds/[seedId]/workspace.vue` 中的节点卡片与底部枝化输入框相关模板、样式和本地交互状态。数据仍来自 `docs/api/workspace.yaml` 的 `GET /api/seeds/{seedId}/workspace`，提交仍使用 `docs/api/growth.yaml` 的 `POST /api/growth-tasks`，不新增数据模型或接口。

## Goals / Non-Goals

**Goals:**

- 将节点卡片重构为更符合 Quiet Command Workspace 的暗色、紧凑、清晰状态表达。
- 保留卡片现有业务逻辑和概念区分，不改变节点可见性、选中、拖拽、状态计算、生成中占位和失败恢复逻辑。
- 将生成器和果实数量下拉改为受控高度的浮层菜单，使选项过多时仍可滚动选择。
- 将 `@` 引用资源候选浮层与正文输入区分层展示，避免候选项遮挡或覆盖输入内容。
- 让已引用资源标签具备明确的移除按钮，移除后同步影响后续 Growth API payload 中的 `nutrientRefs` 与 `geneRefs`。
- 保持键盘焦点、hover/focus 状态和 reduced-motion 兼容。

**Non-Goals:**

- 不修改工作区顶部工具栏、右侧详情面板、画布背景、树布局算法、连接线或拖拽/缩放行为。
- 不修改种子、果实、自然选择、枝化生长任务的后端接口。
- 不修改 `docs/sql/*.sql` 中的数据库结构。
- 不改变哪些节点可以显示输入框、哪些节点可以发起枝化生长的规则。
- 不新增全局设计系统组件库，除非实现阶段发现已有局部代码需要轻量提取以降低重复。

## Decisions

### 决策 1：卡片只重构视觉层，不改节点状态模型

节点仍使用现有 `TreeNode`、`nodeClasses`、`nodeStyle`、`nodeStateLabel`、`selectedNodeId` 和 `status`/`selectionState` 逻辑。实现阶段只调整节点模板结构和 CSS token，使不同状态通过边框、局部色彩、状态 chip、透明度和克制动画表达。

替代方案是重新抽取独立节点组件并重写状态映射；当前页面仍是单页工作区实现，拆分会扩大变更面，不符合“只改卡片和输入框”的约束。

### 决策 2：下拉菜单使用向上展开的 bounded menu

生成器和果实数量菜单应从输入框控制区向上展开，设置最大高度和内部滚动，保持在底部输入框附近，不进入右侧详情面板，不因选项过多撑出视口。菜单项保留当前选择态，生成器项可展示名称与描述，果实数量项保持紧凑。

替代方案是使用原生 `select` 或全屏命令面板；原生选择器不符合现有精细视觉方向，全屏命令面板会改变当前输入框工作流。

### 决策 3：引用候选浮层独立于输入正文和引用标签

`@` 触发的资源候选浮层应作为输入框上方的独立 overlay 展示，具备固定宽度约束、最大高度和滚动，不与 textarea、已选引用 chip 或 footer 动作区重叠。候选项继续来自 `WorkspaceSnapshot.resources.nutrients` 与 `WorkspaceSnapshot.resources.geneInsights` 过滤结果。

替代方案是把候选项内嵌到输入区内部；这会继续挤压正文输入空间，也更容易出现重叠问题。

### 决策 4：引用资源移除只更新前端本地引用集合

已选引用资源 chip 增加移除按钮，点击后从 `referencedResources` 中删除对应资源。提交枝化生长时继续从 `referencedResources` 生成 `nutrientRefs` 与 `geneRefs`，因此无需新增 API 字段。移除行为应在恢复失败输入后同样可用。

替代方案是在输入文本中保留特殊 token 并通过解析 token 删除引用；当前实现已经把引用资源与正文输入分离为结构化集合，继续维护结构化集合更稳。

## Risks / Trade-offs

- 菜单 overlay 层级与现有详情面板或画布节点冲突 → 通过局部 z-index 分层和输入框容器定位约束，验证菜单不会被节点卡片或详情面板遮挡。
- 引用 chip 增加移除按钮后误触 → 按钮目标保持足够点击面积，并使用明确的 hover/focus 状态。
- 卡片样式提升后信息密度下降 → 节点仍只展示类型、标题、状态和必要生成中反馈，完整内容继续放在右侧详情。
- 视觉动效过强影响阅读 → 动效仅用于 hover/focus 和生成中状态，并遵守 `prefers-reduced-motion`。
- 单文件页面继续增大 → 本次优先控制范围；若实现中样式或交互显著膨胀，可在不改变行为的前提下提取局部纯 UI helper，但不做跨模块重构。
