## Why

营养库后端能力将提供公共营养库、种子专属营养库和 Markdown 营养内容管理，前端需要提供可操作的营养库页面，让用户能够创建资料、维护资料，并在后续工作区枝化生长中通过 `@营养` 引用。当前前端尚无营养库页面，无法承接第一期“上传/维护营养资料并供枝化生长引用”的流程。

## What Changes

- 新增营养库管理页面：展示公共营养库和种子专属营养库，并支持切换视图。
- 新增营养库创建交互：支持创建公共营养库；支持创建种子专属营养库并选择归属种子；名称必填，描述非必填。
- 新增营养库详情与编辑：展示营养库名称、描述、作用域、归属种子、归档状态，并允许编辑名称和描述。
- 新增营养内容管理：在营养库中新增 Markdown 营养内容、查看详情、编辑正文、归档和回档。
- 新增状态反馈：覆盖列表加载、详情加载、创建失败、编辑失败、归档失败、空状态和只读/归档提示。
- 为工作区 `@营养` 引用预留前端数据访问能力：按当前种子查询可引用营养内容，过滤已归档和作用域不匹配资源。
- 保持页面边界：不做向量检索、知识库问答、自动网页采集、跨用户共享市场，不把基因库经验混入营养库页面。

## Capabilities

### New Capabilities

- `nutrient-library-page`: 定义前端营养库页面的公共/种子专属视图、营养库创建编辑、营养内容 Markdown 管理、归档回档、可引用营养查询和接口联动行为。

### Modified Capabilities

无。

## Impact

- 影响前端页面：新增或升级 `/nutrients` 为可用的营养库管理页面。
- 影响前端接口调用：严格使用后端新增的 `docs/api/nutrient.yaml` 接口契约；如果契约尚未存在，前端实现需等待后端提案落地该顶层 API 文档。
- 影响前端数据展示：展示字段必须来自 `docs/api/nutrient.yaml` 的营养库与营养内容响应契约，并参考 `docs/sql/nutrient.sql` 中的系统事实边界。
- 影响种子数据依赖：创建种子专属营养库时需要选择归属种子，前端可复用现有种子列表能力。
- 影响工作区：后续工作区悬浮输入框的 `@营养` 引用可复用本变更提供的可引用营养数据访问能力。
- 影响设计系统：页面必须遵守 `docs/spec/DESIGN.md` 的 Quiet Command Workspace 风格，采用紧凑暗色工作台和管理型信息布局。
