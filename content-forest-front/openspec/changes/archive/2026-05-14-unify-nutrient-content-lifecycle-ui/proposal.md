## Why

营养工作台当前把“营养卡片”和“营养资料/营养内容”分开展示，用户会误以为这是两类对象，并且在没有专属营养库时不知道草稿沉淀到哪里。前端需要把交互语言统一为“营养内容”，让用户只关注资料的草稿、沉淀、归档和默认带入状态。

## What Changes

- 营养工作台用户侧文案统一：将“营养卡片”改为“营养内容”，将“未沉淀”改为“草稿”，将“常驻营养”改为“默认带入”。
- 打开种子营养工作台时，前端调用后端默认专属营养库确保能力；若后端尚未完成，按后端依赖处理，不在前端私自拼接库创建规则。
- 营养内容列表按生命周期展示：草稿、已沉淀、已归档；不同状态显示对应操作。
- 草稿态营养内容允许删除；沉淀态只允许归档、默认带入、编辑；归档态只允许查看和回档。
- 采纳缺口建议或可沉淀营养块时，用户看到的是“保存为草稿营养内容”，而不是“创建营养卡片”。
- 枝化生长默认引用区域显示“默认带入”的已沉淀营养内容，用户可以在本次生成前移除。

## Capabilities

### New Capabilities

- None

### Modified Capabilities

- `nutrient-library-page`: 统一营养工作台的用户侧概念、状态展示、默认专属库入口和草稿删除交互。

## Impact

- 页面与组件：营养库页面、种子营养工作台 Dialog、营养内容列表/详情/会话面板、枝化生长引用选择入口
- API 依赖：依赖后端在 `docs/api/nutrient.yaml` 中补齐默认专属库确保接口、草稿删除接口，以及沉淀接口的可选目标库语义
- 数据结构：前端不得修改 SQL；继续按 `docs/sql/nutrient.sql` 读取现有状态值并映射为用户侧标签
- 测试：更新营养工作台交互测试、文案断言和状态操作可用性测试
