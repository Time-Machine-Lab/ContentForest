## Context

工作区页面目前承载了内容树画布、节点卡片、右侧详情、底部枝化输入框和生长任务状态。现有实现已经接入后端工作区快照和生长任务接口，但节点视觉样式经过多轮试验后叠加较多，显得花哨；树布局仍以 depth 分层横排为主，不能稳定呈现“由种子向上生长”的树形结构。

本变更只处理前端展示和运行时布局，不修改 API、SQL 或后端行为。节点位置继续只存在于前端运行时，不持久化。

## Goals / Non-Goals

**Goals:**

- 将节点视觉收敛为简洁拟物风格，保留可感知但克制的生长动效。
- 在节点卡片上只保留当前状态标签，减少信息噪声。
- 让种子与果实有明显但统一的视觉差异。
- 将生长状态组件整合进工作区 header，作为 header 的原生信息区。
- 提供一键整理能力，将当前节点重新排布为自底向上、结构清晰、间距合理的树。
- 让生成中占位节点使用同一套树布局思想落位，避免与既有节点挤压。

**Non-Goals:**

- 不新增节点位置持久化能力。
- 不修改 `GET /api/seeds/{seedId}/workspace`、`POST /api/growth-tasks` 等接口契约。
- 不引入第三方树布局库，优先使用本页面内的轻量布局算法。
- 不重做右侧详情面板、底部枝化输入框或后端 Agent 流程。

## Decisions

### 1. 节点视觉采用“简化拟物”而不是复杂 Liquid Frost

节点将保留拟物感，但不继续使用大面积半透明高光、复杂纹理和多层状态装饰。种子节点可以偏圆润、厚实，表达根节点和灵感源；果实节点更像内容产物卡片，结构更轻。果实状态通过边缘色、状态点、轻微透明度和唯一状态标签表达。

替代方案是继续沿用 Liquid Frost 或完整重做为扁平 SaaS 卡片。前者过于花哨，后者又削弱“内容森林”的概念感，因此选择中间路线。

### 2. 节点卡片只展示当前状态标签

节点底部只保留一个状态标签，例如“根节点”“候选”“已选择”“已淘汰”“生长中”“最近失败”。基因标签、内容路径等信息继续放在右侧详情面板，不再占用节点卡片视觉空间。

这样可以让画布承担结构浏览职责，详情面板承担信息阅读职责。

### 3. 生长状态组件进入工作区 header

原画布左下角的生长状态条会并入顶部 header。header 需要同时容纳种子标题、当前同步/生长状态、进度条和操作按钮，形成一个统一的画布控制区。

这避免状态组件悬浮在画布中遮挡内容，也让“整理树形/适应视图”等操作和当前树状态处于同一个信息层级。

### 4. 树布局采用轻量 tidy tree 思路

布局算法应从父子边关系构建树：

- 以种子 rootNodeId 为根。
- 递归计算每个节点的子树宽度。
- 叶子节点按最小水平间距占位。
- 父节点水平居中于子节点集合。
- depth 决定纵向层级，自底向上展示。
- 整棵树计算完成后统一平移到画布中心。

用户拖拽节点后，拖拽位置仍可临时覆盖自动布局；点击“整理树形”时应清除运行时拖拽坐标影响，重新应用 tidy tree 坐标。

### 5. 生成中占位节点参与运行时树布局

发起枝化生长后，前端生成的占位果实不应只用简单 spread 放在父节点上方。占位节点应作为父节点的临时 children 加入运行时布局输入，再通过同一套树布局计算位置。

如果为了实现简化，可以在添加占位节点后对当前完整树执行一次运行时布局；不需要做复杂局部布局。

## Risks / Trade-offs

- [Risk] 自动整理树形会覆盖用户手动拖拽后的视觉位置 → Mitigation：只在用户点击整理按钮或快照初次加载时应用；普通拖拽仍保留运行时自由调整。
- [Risk] 树很宽时一键居中后节点仍超出首屏 → Mitigation：根据树边界计算合适缩放比例，并限制最小/最大缩放。
- [Risk] 删除节点额外标签后节点信息减少 → Mitigation：状态留在节点，基因和详情继续在右侧详情面板展示。
- [Risk] 复杂树的 tidy tree 算法可能出现交叉或过宽 → Mitigation：第一期优先保证父子层级清晰和兄弟子树间距，后续再考虑高级布局优化。
