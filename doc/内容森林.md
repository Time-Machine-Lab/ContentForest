# 内容森林 (Content Forest) - AI 内容进化引擎

> "以进化论为算法，以 AI 为工蚁，构建自动化的内容生态森林。"

# 一，项目简介

## 1. 项目愿景 (Vision)

**利用代码杠杆撬动媒体杠杆，打造零边际成本的自动化内容工厂。**

内容森林不仅是一个内容生成工具，更是一个**基于遗传算法的内容自进化系统**。它模拟自然界的演化过程，通过“生成-投放-反馈-迭代”的闭环，让内容在市场（流媒体平台）的真实反馈中不断自我优化，最终筛选出最具传播力与生命力的“超级物种”。

## 2. 核心逻辑 (Core Logic)

该项目建立在以下第一性原理之上：

1. **进化论 (Evolution)**：适者生存。只有经过市场（用户注意力）检验的内容才是好内容。
2. **复利效应 (Compounding)**：每一次迭代都基于上一次的成功经验，内容的质量与匹配度呈指数级上升。
3. **代码杠杆 (Code Leverage)**：AI Agent 将内容生产的边际成本降至接近零，实现无限规模化。
4. **媒体杠杆 (Media Leverage)**：产出的内容（帖子、视频）本身就是资产，可以在网络上无限分发，无需许可。

## 3. 系统架构：五步循环法 (The 5-Step Loop)

整个内容孵化的过程就像是一棵树的生长，往复循环，螺旋上升。

### Phase 1: 播种 (Genesis) - 注入专属知识

- **定义**：一切的起点。这不是具体的文案，而是你的**核心意图**与**元数据**。
- **输入**：
  - **种子 (Seed)**：你的产品卖点、品牌价值观、原始创意、独特洞察。
  - **基因 (Gene)**：目标受众画像、平台调性（小红书/TikTok/推特）、情感基调。
- **关键点**：这是系统中唯一需要人类深度参与的部分。**你的判断力决定了种子的质量。**

### Phase 2: 生长 (Growth) - Agent 规模化生产

- **定义**：AI Agent 作为园丁，基于种子生成多样化的内容变体。
- **动作**：
  - **裂变 (Fission)**：同一个核心观点，生成 10 种不同的标题风格（震惊体、情绪体、干货体...）。
  - **跨模态 (Cross-Modal)**：将文字脚本转化为短视频脚本、长图文案、播客大纲。
  - **变异 (Mutation)**：引入 **10% 的随机性**。允许 AI 尝试非常规的表达方式，防止系统陷入局部最优解，探索未知的流量蓝海。

### Phase 3: 投放 (Harvest) - 市场验证

- **定义**：将果实（实际内容）投入市场，检验其生存能力。
- **动作**：多平台自动化发布（小红书、抖音、TikTok、推特等）。
- **关键点**：覆盖不同生态位，测试内容在不同土壤（平台算法）下的表现。

### Phase 4: 监控 (Feedback) - 数据回流

- **定义**：收集平台反馈，这是市场的**客观真理**。
- **指标**：
  - **显性指标**：浏览量、点赞数、评论数、转发数。
  - **隐性指标**：完播率、停留时长、主页点击率。
- **动作**：将非结构化的用户反馈转化为结构化的数据评分。
- **情感分析 (Sentiment Analysis)**：监控评论区的情绪。有时候“黑红”也是红，但需要区分恶评和争议。

### Phase 5: 迭代 (Evolution) - 算法优化判断力

- **定义**：基于数据反馈，修正生长策略，完成闭环。
- **逻辑**：
  - **优胜劣汰**：淘汰数据差的内容方向（剪枝）。
  - **基因提取 (Gene Extraction)**：分析爆款内容的共同特征（如：特定的关键词、某种剪辑节奏、某种配色），将其固化为新的“优良基因”，存入**基因库 (Gene Bank)**。
  - **基因重组 (Crossover)**：将不同平台的优良基因进行“杂交”（例如小红书的文案结构 + TikTok 的视频节奏），生成新的内容变体。
  - **参数调整**：告诉 Agent：“这类标题在小红书更受欢迎，下次多生成这种风格。”

## 4. 关键增强模块 (Enhancement Modules)

为了增加系统的反脆弱性与资产沉淀能力，集成以下核心模块：

1. **基因库 (Gene Bank)**：建立一个爆款特征数据库。记录历史上所有高转化内容的特征，供新种子参考。这是系统的核心资产之一。
2. **A/B 测试自动化**：系统自动进行对照实验，比如同一视频配两个不同封面，看哪个点击率高，自动保留胜者。
3. **资产沉淀 (Asset Accumulation)**：系统的最终产物不只是粉丝，而是一套 **“高转化 Prompt 库”和“爆款特征数据库”**。

# 二，项目架构

## 2.1 DDD 统一语言 (Domain-Driven Design)

为了确保系统各模块之间的清晰协作，避免概念歧义，我们定义以下核心术语：


| 术语           | 英文                  | 定义                       | 说明                                                 |
| ------------ | ------------------- | ------------------------ | -------------------------------------------------- |
| **种子库**      | Seed Repository     | 创意的源头，存放用户的想法、项目、idea、创意 | 以 Markdown 格式存储，包含产品卖点、品牌价值观、原始创意、独特洞察等            |
| **营养库**      | Nutrient Repository | 存放种子孵化所需的内容、素材、规则等知识经验沉淀 | 分为三层：平台营养库、领域营养库、种子营养库                             |
| **├─ 平台营养库** | Platform Knowledge  | 特定平台的通用知识                | 如小红书的用户特征、爆款格式、标签策略；抖音的算法偏好、完播率优化等                 |
| **├─ 领域营养库** | Domain Knowledge    | 垂直领域的专业知识（跨平台通用）         | 如 AI 工具领域的专业术语、技术趋势；健身领域的训练方法、营养知识等                |
| **└─ 种子营养库** | Seed Knowledge      | 当前种子的专属素材和历史数据           | 包含该种子的历史生成记录、高转化果实特征、用户偏好分析、竞品参考等                  |
| **生成器**      | Generator           | 将种子/果实转化为果实的执行组件            | 由**外部元数据**（平台展示信息）+ **生成核心**（Skill 文件夹）组成。只关注内容生成，不感知营养库和 MCP，具备可移植性。一个平台可有多个生成器 |
| **生成核心**    | Generator Skill     | 生成器内部的 Skill 文件夹              | 定义生成逻辑、生成方法和输出内容类型。单独取出放到其他 Agent 或项目中同样可运行 |
| **生成器 Agent** | Generator Agent    | 编排整个生成流程的决策层               | MVP 阶段为 AI IDE（Cursor/Trae）内置 Agent；最终阶段为平台自研 Agent。流程逻辑一致，唯一变量是 Agent 的宿主环境 |
| **我的生成器**  | My Generators       | 用户已安装的生成器私有副本             | 存储在 `/cf/data/{userId}/generators/` 下，可被用户迭代优化 |
| **生成器市场**  | Generator Market    | 生成器的分享与获取平台                 | MVP 支持免费导入/分享；预留 `price`、`rating`、`install_count` 字段，为付费市场扩展铺路 |
| **果实载荷**    | Fruit Payload       | 果实中由生成器自定义的内容部分         | 结构完全由生成器决定，系统只存储不解析。不同平台、不同生成器的载荷结构可以完全不同 |
| **果实展示契约** | Fruit Preview Contract | 生成器必须填写的最小展示信息        | 包含 `title`、`summary`、`contentType`，供系统果实池卡片展示使用。是系统与生成器之间的松耦合契约 |
| **生成**       | Generation          | 使用生成器 + 营养库将种子发育为果实的过程   | 包含：选择生成器 → 加载营养库 → LLM 决策 → 多果实生成 → 记录日志           |
| **果实**       | Fruit               | 生成后的可发布内容                | 包含标题、正文、标签、关联资源等。生命周期：生成 → Pick Up → 投放 → 发布/淘汰    |
| **果实池**      | Fruit Pool          | 存放所有生成果实的容器              | 包含已 Pick、待观察、已淘汰的果实。未被 Pick 的果实不删除，可用于后续分析或 A/B 测试 |
| **Pick Up**  | Pick Up             | 用户从多个果实中选择优质内容的过程        | 在"生成"和"投放"之间加入人类判断力，被 Pick 的果实特征会反哺到种子营养库          |
| **基因突变**     | Mutation            | 引入随机性和反常识探索，避免陷入局部最优解    | 由 LLM 自适应决策，包含三类策略：风格突变、要素重组、反常识探索                 |
| **生成日志**     | Generation Log      | 记录每次生成的关键信息              | 包含生成时间、使用的生成器、突变信息、营养库使用情况、输出果实列表等                 |
| **果实成长数据**   | Fruit Growth Data   | 果实在平台上的表现数据              | 包含点赞、收藏、播放量、评论等指标，以 JSON 格式存储。可包含平台特有指标（如完播率、想要数）  |
| **数据监控器**    | Data Monitor        | 收集果实成长数据的组件              | 可以是程序自动采集（主动监控器），也可以是用户手动填写（被动监控器）                 |
| **营养汲取**     | Nutrient Extraction | 从果实或外部内容中提取优质信息的过程       | 用户手动触发，分析高转化/低转化果实，提取成功因素或失败教训，更新营养库和生成器 Skills    |
| **迭代**       | Iteration           | 基于果实进行再次生成的过程            | 本质上就是"生成"，但输入是果实而非种子。果实产出后可以像种子一样继续生成新果实           |
| **迭代树**      | Iteration Tree      | 种子及其衍生果实的完整进化路径          | 种子 → 果实 → 果实的果实 → ...，形成树状结构，记录内容的进化历程             |
| **汲取日志**     | Extraction Log      | 记录每次营养汲取的详细信息            | 包含汲取时间、来源果实、提取的特征、更新的营养库内容                         |


**核心价值**：

- **营养库三层结构**：避免每次生成都"从零开始"，将成功经验固化为可复用的知识资产，支持跨种子、跨平台的知识迁移
- **多果实生成 + Pick Up**：在自动化和人类判断力之间找到平衡，降低完全自动化的风险
- **基因突变机制**：探索未知的流量蓝海，防止内容同质化
- **迭代树机制**：果实可以像种子一样继续生成，形成进化路径，记录内容的完整演化历程

## 2.2 核心流程一：从种子到果实

### 流程概览

```
种子库 → 生成器（加载营养库 + 突变决策）→ 多果实生成 → 用户 Pick Up → 进入后续流程
```

### 详细步骤

#### Step 1: 种子选择

用户从种子库中选择一个种子，明确生成意图：

- 目标平台（小红书/抖音/推特等）
- 内容类型（图文/视频/长图等）
- 生成数量（建议 3-5 个果实）

#### Step 2: 生成器初始化

系统根据用户选择，加载对应的生成器，并自动加载三层营养库：

- 平台营养库：该平台的规则和偏好
- 领域营养库：该种子所属领域的专业知识
- 种子营养库：该种子的历史数据和成功经验

#### Step 3: 突变决策（LLM 自适应）

生成器的 Agent 分析种子的历史表现，决定是否需要突变：

- **不突变**：如果历史数据良好，继续沿用成功模式
- **突变**：如果需要探索新方向，选择突变策略（详见 2.3 节）

#### Step 4: 多果实生成

生成器一次性生成 3-5 个差异化的果实：

- **果实 A（保守型）**：严格遵循营养库的成功模式，风险最低
- **果实 B（平衡型）**：70% 遵循成功模式 + 30% 创新尝试
- **果实 C（激进型）**：引入突变机制，探索未知方向
- **果实 D/E（随机型）**：LLM 自由发挥，增加多样性

**差异化策略**：

- 标题风格不同（震惊体/干货体/情绪体/故事体）
- 排版风格不同（极简风/密集风/图文混排）
- 情感基调不同（理性/感性/幽默/严肃）

#### Step 5: 用户 Pick Up

系统将生成的果实展示给用户，用户可以：

- **Pick 1 个或多个**：选中的果实进入后续流程（投放、监控）
- **标记"待观察"**：暂时不确定，保留在果实池
- **标记"淘汰"**：明确不符合预期，但不删除（用于后续分析）

**Pick Up 的价值**：

- 在"生成"和"投放"之间加入人类判断力
- 降低完全自动化的风险
- 被 Pick 的果实特征会反哺到种子营养库，优化后续生成

#### Step 6: 果实池管理

- **已 Pick 的果实**：进入投放流程（第三章内容）
- **未 Pick 的果实**：
  - 保留在果实池，不直接删除
  - 如果后续 Pick 的果实数据不佳，可以回来"捞"这些备选方案
  - 也可以用于 A/B 测试对照组

## 2.3 基因突变机制 (Mutation Mechanism)

### 设计理念

**核心思想**：避免系统陷入"局部最优解"，通过引入随机性和反常识探索，发现未知的流量蓝海。

**决策主体**：LLM（Agent）根据种子历史表现和营养库数据，自主判断是否需要突变，以及选择哪种突变策略。

**触发逻辑**：

- **不突变**：前 5 次生成效果都不错（转化率 > 阈值）
- **轻度突变**：最近 2-3 次生成效果平平（转化率在平均水平）
- **强制突变**：连续 3 次生成数据差（转化率 < 阈值）

---

### 三类突变策略

#### 策略 1：风格突变 (Style Mutation)

**适用场景**：同一种子已生成 3+ 次，需要探索新的表达方向。

**LLM 指令示例**：

- "将这个种子用**完全相反的情绪**重新表达"
  - 理性 → 感性
  - 严肃 → 幽默
  - 高冷 → 接地气
- "用**不同圈层的黑话**改写"
  - 科技圈术语 → 饭圈用语
  - 学术表达 → 网络梗
- "改变内容密度"
  - 长文案压缩为**极简风格**（只保留核心观点）
  - 极简风格扩展为**故事化叙事**（增加场景和细节）

**示例**：

- 原始种子："AI 工具可以提升工作效率"
- 风格突变后：
  - 感性版："深夜加班时，是 AI 陪我熬过了那些崩溃瞬间"
  - 幽默版："老板：你怎么这么快？我：因为我有 AI 打工人"
  - 极简版："AI = 效率 × 10"

---

#### 策略 2：要素重组 (Element Remix)

**适用场景**：营养库中有多个高转化案例，尝试"杂交"出新物种。

**LLM 指令示例**：

- "从营养库中随机抽取 2 个爆款案例的**核心特征**，进行组合"
  - 案例 A 的标题结构 + 案例 B 的视觉符号
  - 案例 C 的情感钩子 + 案例 D 的排版风格
- "将**不同平台的优势**融合"
  - 小红书的排版美学 + 推特的简洁表达
  - 抖音的视频节奏 + B 站的知识密度

**示例**：

- 营养库中的爆款案例：
  - 案例 A："3 个 AI 工具，让你下班早 2 小时"（数字 + 利益点）
  - 案例 B："我用 AI 做了一件很酷的事"（第一人称 + 悬念）
- 要素重组后：
  - "我用 3 个 AI 工具，每天早下班 2 小时"（融合两者优势）

---

#### 策略 3：反常识探索 (Anti-Logic Probe)

**适用场景**：常规策略效果平平，需要突破性尝试。

**LLM 指令示例**：

- "故意违反一条平台规则"
  - 小红书通常要精美排版 → 这次试试纯文字无排版
  - 抖音通常要快节奏 → 这次试试慢镜头 + 留白
- "使用**反直觉的标题**"
  - 大家都说"必看" → 你说"别看"
  - 大家都说"干货" → 你说"废话"
- "引入**10% 的随机噪音**"
  - 在科技文案里突然提到"奶茶"
  - 在严肃话题里插入一个无厘头的比喻

**示例**：

- 常规标题："5 个 AI 工具，你一定要知道！"
- 反常识标题："这 5 个 AI 工具，我劝你别用"（反向吸引好奇心）

---

### 突变记录与分析

每次突变都会记录在生成日志中：

- `mutation_applied`：是否应用突变（true/false）
- `mutation_type`：突变类型（style/remix/anti-logic）
- `mutation_params`：突变参数（如：randomness=0.15）

**后续分析**：

- 统计哪种突变策略的成功率更高
- 分析突变后的果实是否带来数据提升
- 将成功的突变模式固化到营养库

## 2.4 核心流程二：数据收集与迭代

### 流程概览

```
果实发布 → 数据监控 → 营养汲取 → 迭代生成 → 新果实 → ...（循环往复）
```

这个流程形成了**数据闭环**，让内容在市场反馈中不断自我优化，实现"进化论"和"复利效应"。

---

### 详细步骤

#### Step 1: 数据监控

当果实被 Pick Up 并发布到平台后，需要收集其在平台上的表现数据。

**监控方式**：

**方式 1：主动监控器（Active Monitor）**

- **定义**：一段程序，定时调用平台 API，自动采集果实的成长数据
- **适用场景**：平台提供 API 接口，且需要实时监控数据变化
- **配置项**：
  - 采集频率（如：每小时/每天）
  - 停止条件（如：发布 7 天后停止监控）
  - 监控指标（点赞、收藏、评论、分享、浏览量等）
- **输出格式**：JSON 格式的果实成长数据

**方式 2：被动监控器（Passive Monitor）**

- **定义**：用户手动填写或导入果实的成长数据
- **适用场景**：平台不提供 API，或用户只需定期更新数据
- **操作方式**：
  - 用户在平台查看数据
  - 手动填写到系统中（JSON 格式）
  - 或导入平台导出的数据文件

**果实成长数据格式**：

系统提供**默认格式**，用户也可以根据需求**自定义格式**。

**默认格式示例**：

```json
{
```

#### Step 2: 营养汲取

基于果实的成长数据或外部优质内容，提取有价值的信息，更新营养库和生成器 Skills。

**汲取触发**：

- 用户手动触发（当前阶段）
- 未来可扩展：基于规则自动触发（如：果实数据达到阈值时自动汲取）

**汲取来源**：

**来源 1：内部果实**

- **高转化果实**：提取成功因素（正向汲取）
  - 例如：标题结构、关键词、视觉风格、发布时间、情感基调
- **低转化果实**：提取失败教训（负向汲取）
  - 例如：避免某些表达方式、避免某些话题、避免某些发布时段

**来源 2：外部优质内容**

- **竞品/爆款内容**：用户手动导入
- **操作方式**：
  - 用户将外部内容整理成 Markdown 格式
  - 存入对应的营养库（平台营养库/领域营养库/种子营养库）
- **内容示例**：
  - 竞品的爆款帖子分析
  - 行业热点趋势报告
  - 平台算法规则更新

**汲取策略**：


| 汲取类型     | 分析对象  | 提取内容                      | 更新目标                          |
| -------- | ----- | ------------------------- | ----------------------------- |
| **正向汲取** | 高转化果实 | 成功因素（标题结构、关键词、视觉风格、发布时间等） | 营养库（固化为知识）、生成器 Skills（优化生成能力） |
| **负向汲取** | 低转化果实 | 失败教训（避免的表达方式、话题、时段等）      | 营养库（记录禁忌）、生成器 Skills（规避风险）    |
| **外部汲取** | 竞品/爆款 | 行业趋势、平台规则、创新玩法            | 平台营养库、领域营养库                   |


**汲取输出**：

1. **更新营养库**：
  - 将提取的特征写入对应的营养库文件（Markdown 格式）
  - 例如：在 `种子营养库/ai-tools-seed.md` 中新增"高转化标题模式"章节
2. **优化生成器 Skills**：
  - 根据汲取结果，调整生成器的 Prompt 或参数
  - 例如：告诉 Agent"这类标题在小红书更受欢迎，下次多生成这种风格"
3. **生成汲取日志**：
  - 记录汲取时间、来源果实、提取的特征、更新的内容
  - 用于后续分析和追溯

**汲取日志示例**：

```json
{
  "extraction_id": "extract_001",
  "extracted_at": "2026-03-08T14:00:00Z",
  "source": {
    "type": "internal_fruit",
    "fruit_id": "fruit_001",
    "performance": "high"
  },
  "extracted_features": [
    "标题使用数字 + 利益点结构",
    "发布时间在晚上 8-10 点",
    "配图使用暖色调"
  ],
  "updated_targets": [
    "营养库/种子营养库/ai-tools-seed.md",
    "生成器/xiaohongshu-post-generator/skills/title-generation.md"
  ]
}
```

#### Step 3: 迭代生成

基于果实的成长数据和汲取的经验，对果实进行再次生成，产出优化版本。

**核心理念**：

- **迭代本质 = 生成**：迭代就是把果实当作"种子"，再次执行生成流程
- **果实 ≠ 种子**：果实不会放入种子库，但可以作为生成的输入
- **迭代树**：种子 → 果实 A → 果实 A1/A2/A3 → 果实 A1-1/A1-2...，形成树状结构

**迭代触发方式**：

**方式 1：手动迭代**

- **触发条件**：用户主动选择某个果实进行迭代
- **适用场景**：
  - 用户发现某个果实表现不错，想基于它生成更多版本
  - 用户想尝试不同的风格或平台
- **操作流程**：
  1. 用户选择果实 A
  2. 指定迭代意图（延续风格 or 突变探索）
  3. 系统将果实 A 作为输入，调用生成器
  4. 生成 3-5 个新果实（A1/A2/A3...）

**方式 2：自动迭代**

- **触发条件**：果实的成长数据达到预设阈值
- **触发规则示例**：
  - **数据阈值触发**：点赞数 > 1000 时自动迭代
  - **趋势触发**：增长速度突然加快（如：1 小时内点赞增长 > 100）时自动迭代
  - **时间触发**：发布 24 小时后自动迭代一次
- **适用场景**：
  - 爆款果实自动扩展（趁热打铁）
  - 定期优化（持续改进）

**迭代策略**：


| 策略类型      | 定义             | 适用场景               | 示例            |
| --------- | -------------- | ------------------ | ------------- |
| **延续迭代**  | 基于原果实的风格，微调优化  | 原果实表现良好，想保持风格并优化细节 | 改标题、换封面、调整排版  |
| **突变迭代**  | 基于原果实的主题，大幅度变异 | 想探索新方向，或原果实表现平平    | 换平台、换风格、换情感基调 |
| **跨平台迭代** | 将原果实改编为其他平台的内容 | 想扩大传播范围            | 小红书图文 → 抖音短视频 |


**迭代输入**：

- 原果实的内容（标题、正文、标签、资源）
- 原果实的成长数据（用于判断哪些元素有效）
- 用户的迭代意图（延续 or 突变）
- 营养库的最新知识（汲取后的经验）

**迭代输出**：

- 3-5 个新果实（迭代版本）
- 迭代日志（记录迭代原因、策略、输入输出）

**迭代树的形成**：

```
种子：AI 工具推荐
├── 果实 A（小红书图文）
│   ├── 果实 A1（优化标题）
│   ├── 果实 A2（换封面）
│   └── 果实 A3（改为视频脚本）
│       └── 果实 A3-1（视频优化版）
├── 果实 B（推特长推文）
│   └── 果实 B1（改为抖音短视频）
└── 果实 C（知乎长文）
```

**迭代树的价值**：

- 记录内容的完整进化历程
- 分析哪条路径的转化率更高
- 为后续种子提供参考（哪种演化路径更成功）

---

### 数据闭环的完整流程

```
1. 种子 → 生成器 → 果实 A/B/C
2. 用户 Pick Up 果实 A
3. 果实 A 发布到平台
4. 数据监控器收集果实 A 的成长数据
5. 用户触发营养汲取，提取果实 A 的成功因素
6. 更新营养库和生成器 Skills
7. 基于果实 A 进行迭代生成 → 果实 A1/A2/A3
8. 重复步骤 2-7，形成迭代树
```

**关键设计**：

- 每次迭代都基于**市场反馈数据**，而非主观猜测
- 营养库不断积累**成功经验**和**失败教训**
- 生成器 Skills 不断优化，生成质量呈**指数级上升**
- 迭代树记录**完整进化路径**，为后续决策提供依据

## 2.5 数据流与日志 (Data Flow & Logging)

### 生成日志结构（概念层）

每次生成都会产生一条日志，记录关键信息：

```
生成日志 (Generation Log)
├── 基础信息
│   ├── generation_id：生成批次唯一标识
│   ├── seed_id：种子 ID
│   ├── generator：使用的生成器名称
│   ├── platform：目标平台
│   └── timestamp：生成时间
├── 突变信息
│   ├── mutation_applied：是否应用突变
│   ├── mutation_type：突变类型（如有）
│   └── mutation_reason：突变原因（如：连续 3 次数据差）
├── 营养库使用
│   └── nutrients_used：使用的营养库文件列表
└── 输出信息
    ├── fruits_count：生成的果实数量
    └── fruits_ids：生成的果实 ID 列表
```

### 数据流向

**完整数据流**：

```
阶段 1：生成阶段
种子库 → 生成器 → 营养库（读取）→ 果实生成 → 生成日志（写入）
                                    ↓
                            用户 Pick Up → 种子营养库（反馈）

阶段 2：迭代阶段
果实发布 → 数据监控器 → 果实成长数据（写入）
                          ↓
                    营养汲取（用户触发）
                          ↓
        营养库（更新）+ 生成器 Skills（优化）+ 汲取日志（写入）
                          ↓
                    迭代生成（基于果实）
                          ↓
                    新果实 → 迭代树（更新）
```

**关键设计**：

- **生成日志**：记录"做了什么"和"为什么这么做"
- **果实成长数据**：记录市场的真实反馈
- **汲取日志**：记录从数据中提取了哪些经验
- **迭代树**：记录内容的完整进化路径
- **数据闭环**：每次迭代都基于市场反馈，形成正向循环

## 2.6 前端交互流程 (Frontend User Flow)

### 核心工作流程

内容森林的前端交互遵循以下核心流程：

```
a. 用户进入工作台 (Dashboard)
   ↓
b. 访问种子库模块 (Seed Repository)
   ↓
c. 种子卡片墙展示 (Seed Card Wall)
   - 用户可在卡片墙上进行操作（编辑、删除、发布、生成等）
   ↓
d. 点击某个种子进入种子工作区 (Seed Workspace)
   ↓
e. 种子工作区 (核心工作区) - 树状图结构
   - 呈现为动效生动的树状图，展示种子的完整进化路径
   - 种子为根节点，果实为分支节点
   - 支持多代迭代：种子 → 果实 → 果实的果实 → ...
   - 核心操作全部在此完成：
     * 果实生成 (Fruit Generation)
     * 营养库选择 (Nutrient Selection)
     * 基因突变配置 (Mutation Configuration)
     * 果实 Pick Up 决策 (Fruit Selection)
     * 迭代生成 (Iteration Generation)
```

### 工作区设计理念

**种子工作区的特殊性**：

种子工作区是内容森林的**核心工作区**，与其他工作区完全不同。它采用**树状图结构**来呈现种子的完整进化过程：

- **根节点**：原始种子
- **分支节点**：生成的果实
- **子分支**：基于果实的迭代果实
- **树的深度**：代表迭代的代数

**为什么用树状图**：

1. **直观展示进化路径**：用户可以一眼看到从种子到果实的完整演化过程
2. **支持多代迭代**：树状结构天然支持无限深度的迭代
3. **视觉化决策**：用户可以比较不同分支的表现，做出更好的决策
4. **动效生动**：通过动画展示果实的生长过程，增强用户体验

**树状图示例**：

```
                    🌱 种子：AI 工具推荐
                           │

                │          │          │
              🍎A        🍎B        🍎C
            (小红书)    (推特)     (知乎)
              │          │          │
         ┌────┴────┐     │     ┌────┴────┐
         │         │     │     │         │
       🍎A1      🍎A2  🍎B1  🍎C1     🍎C2
```

### 视图 5：场景视图（Scenarios）

**核心使用场景**：

```
场景 1：从零到一 - 创建第一个爆款
┌──────────────────────────────────────────────────────┐
│ 用户创建种子 → 选择生成器 → 生成 5 个果实           │
│ → Pick Up 果实 A → 发布 → 监控数据（点赞 1200+）    │
│ → 触发汲取 → 提取成功因素 → 更新营养库              │
└──────────────────────────────────────────────────────┘

场景 2：迭代优化 - 爆款内容的二次传播
┌──────────────────────────────────────────────────────┐
│ 果实 A 表现优异 → 自动触发迭代 → 生成 3 个版本      │
│ - A1：优化标题（延续迭代）                           │
│ - A2：改为视频脚本（跨模态迭代）                     │
│ - A3：改为推特长推文（跨平台迭代）                   │
│ → Pick Up A2 → 发布抖音 → 形成迭代树                │
└──────────────────────────────────────────────────────┘

场景 3：知识沉淀 - 从失败中学习
┌──────────────────────────────────────────────────────┐
│ 果实 B 表现不佳（点赞 < 50）→ 触发汲取（负向）      │
│ → 分析失败原因：标题学术化、发布时间不佳            │
│ → 更新营养库：避免学术表达、最佳时间 8-10 点        │
│ → 下次生成自动规避这些问题                           │
└──────────────────────────────────────────────────────┘

场景 4：规模化生产 - 批量生成多平台内容
┌──────────────────────────────────────────────────────┐
│ 创建种子 → 同时选择 3 个生成器（小红书/抖音/推特）  │
│ → 并行生成 15 个果实（每平台 5 个）                 │
│ → Pick Up 各平台最优果实 → 批量发布                 │
│ → 监控所有平台数据 → 分析最优传播渠道               │
└──────────────────────────────────────────────────────┘
```

**迭代树示例**：

```
种子：AI 工具推荐
├── 果实 A（小红书图文，点赞 1200+）
│   ├── 果实 A1（优化标题）
│   ├── 果实 A2（抖音短视频，播放 8.5 万）
│   └── 果实 A3（推特长推文）
├── 果实 B（推特长推文）
│   └── 果实 B1（改为抖音短视频）
└── 果实 C（知乎长文）
```
