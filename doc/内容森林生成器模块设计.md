# 内容森林 - 生成器模块设计 (Generator Module)

> 版本：v1.0 (MVP)
> 日期：2026-03-16
> 状态：设计中

---

## 1. 核心语义与定义

### 1.1 什么是生成器 (Generator)

**生成器**是内容森林中负责将种子（或果实）转化为可发布内容的核心执行组件。它由两个部分组成：

- **外部元数据 (Metadata)**：用于平台展示与管理，包含生成器名称、作者、适用平台、描述、内容格式、版本号、创建时间、更新时间等。
- **生成核心 (Skill)**：一个标准的 Skill 文件夹，定义了该生成器如何生成内容、能生成哪些内容（图片、文字、视频脚本等）以及生成优质内容的方法论。

**关键设计原则**：

> 生成器**只关注如何生成内容数据**。它不感知、不依赖营养库，不调用 MCP，不关心数据如何落库。它的核心职责是：接收输入文本，输出结构化内容。

这意味着生成器的 Skill 部分具备完全的**可移植性**——单独取出放到其他 Agent 或其他项目中，同样可以运行。

---

### 1.2 生成器的组成结构

生成器由**两个独立关注点**组成，分属不同的维护主体：

```
┌─────────────────────────────────────────────────────────┐
│                      生成器                              │
├──────────────────────────┬──────────────────────────────┤
│   外部元数据              │   生成核心（Skill）           │
│   Generator Metadata     │   Generator Skill            │
│                          │                              │
│   由平台存储和管理         │   由用户创建和维护            │
│   与 Skill 内容解耦       │   标准 Skill 结构，无平台约束 │
│                          │                              │
│   关注「这个生成器是什么」  │   关注「怎么生成好内容」      │
└──────────────────────────┴──────────────────────────────┘
```

**关键设计原则**：用户创建一个内容生成 Skill 时，**只需关注 Skill 本身的生成规范**，无需了解平台需要什么字段，无需维护任何平台元数据文件。两者通过平台注册时建立关联。

---

#### 部分 A：外部元数据（平台维度）

元数据由**平台存储**，用户在平台注册/上传生成器时填写。Skill 内容更新，元数据不一定要变。

**元数据结构（平台侧数据模型）**：

```typescript
interface GeneratorMetadata {
  // 平台标识（平台生成）
  id: string;                    // 如 "gen_xhs_lifestyle_v1"
  createdAt: number;
  updatedAt: number;

  // 用户填写（注册时提交）
  name: string;                  // "小红书生活方式图文生成器"
  description: string;
  thumbnail?: string;
  platform: string;              // "xiaohongshu" | "douyin" | "twitter" | ...
  contentTypes: string[];        // ["image_text", "text_only"]
  domain?: string;               // "lifestyle" | "tech" | "fitness" | ...

  // 作者信息
  author: {
    id: string;
    name: string;
  };

  // 输出能力声明（告知 Agent 该生成器能产出哪些字段）
  outputCapabilities: string[];  // ["title", "body", "tags", "cover_prompt"]

  // 市场数据（平台维护，预留）
  price: number;                 // 0 = 免费，预留付费扩展
  installCount: number;          // 安装次数，用户安装时自动累加
  rating: number | null;         // 评分，暂不实现
}
```

---

#### 部分 B：生成核心 Skill（用户维度）

Skill 就是一个**标准的 Skill 文件夹**，没有任何平台特有的约束。用户只需按照 Skill 规范编写，不需要额外的 `generator.yaml` 或其他平台配置文件。

Skill 的文件结构由用户自由决定，下面是一个推荐的参考结构（非强制）：

```
xiaohongshu-lifestyle-post/    ← Skill 目录，命名自由
├── SKILL.md                   ← 主 Skill 文件（必须），描述整体生成逻辑
└── references/                ← 参考资料（可选）
    ├── title-patterns.md      ← 标题写法参考
    ├── body-patterns.md       ← 正文结构参考
    └── tag-strategy.md        ← 标签策略参考
```

**SKILL.md 的核心内容**（用户自由发挥，以下为建议包含的要素）：

- 该生成器擅长什么、适合什么场景
- 如何接收输入（种子/果实内容的哪些部分）
- 生成几个果实、每个果实的差异化策略
- 必须输出的 `preview` 字段（title + summary + contentType）
- 自定义 `payload` 的结构说明
- 优质内容的方法论（平台规律、爆款模式等）

**与平台的关联**：用户将 Skill 文件夹上传到生成器市场时，平台为其创建一条 `GeneratorMetadata` 记录，通过 `generatorId` 与 Skill 建立关联。两者独立演进，互不耦合。

---

### 1.3 什么是生成器 Agent (Generator Agent)

**生成器 Agent** 是调度和编排整个生成过程的决策层。它负责：

- 接收用户输入（种子/果实 + 生成器选择 + 营养库挂载 + 突变策略）
- 通过 MCP 获取所需数据（种子内容、营养库内容）
- 加载并执行生成器的 Skills
- 通过 MCP 保存生成结果并记录日志

生成器 Agent 本身不是生成器，它是**使用生成器的协调者**。

**MVP 阶段与最终阶段的唯一区别**：


| 阶段      | Agent 来源                    | 交互入口            |
| ------- | --------------------------- | --------------- |
| **MVP** | AI IDE（Cursor/Trae）内置 Agent | 用户在 IDE 聊天框输入指令 |
| **最终版** | 平台自研 Agent                  | 用户在平台 Web 页面操作  |


流程编排逻辑完全一致，唯一变量是 Agent 的宿主环境。

## 2. 生成器与用户的关系

### 2.1 生成器的来源

```
生成器来源
├── 平台生成器（Platform Generator）  ← 平台官方提供，存储在云端，由平台维护
└── 用户生成器（User Generator）       ← 用户自己创建或从平台安装
    └── 我的生成器（My Generators）    ← 安装到本地后成为用户私有副本
```

### 2.2 生命周期流程

```
1. [发布]  用户/平台 → 创建生成器 → 上传至生成器市场（MVP 阶段支持导入）
2. [安装]  其他用户 → 从市场安装 → 复制到本地用户目录 → 变为【我的生成器】
3. [使用]  用户在生成时选择【我的生成器】或【平台生成器】
             └─ 选择平台生成器时自动安装并转为【我的生成器】
4. [迭代]  用户在营养汲取阶段，基于果实表现，使用 skill-creator 工具优化【我的生成器】
```

### 2.3 生成器存储路径

```
# 用户私有生成器（我的生成器）——本地文件系统
/cf/data/{userId}/generators/{generatorId}/
    └── {generatorName}
        ├── SKILL.md
        └── ...

# 平台生成器——云端维护，不在本地文件系统
# 通过 Generator Market API 拉取，安装时复制到用户目录
```

### 2.4 生成器迭代（Skill 优化）

生成器迭代的本质是对 Skill 文件的优化。在**营养汲取**阶段，当用户分析了高/低转化果实后，可以：

- 手动编辑生成器的 `skills/*.md` 文件，调整 Prompt
- 使用 AI IDE 中的 skill-creator 工具，让 Agent 辅助优化 Skill

## 4. 生成器 Agent 工作流程

### 4.1 完整生成流程

```
用户在 AI IDE 输入生成指令
    ↓
生成器 Agent（AI IDE）
    ↓
[Step 1] 通过 MCP 获取种子/果实内容
    → MCP Tool: get_seed(seedId) 或 get_fruit(fruitId)
    ↓
[Step 2] 通过 MCP 获取挂载的营养库内容（可选）
    → MCP Tool: get_nutrients(nutrientPaths[])
    ↓
[Step 3] 加载生成器 Skill
    → 读取 /cf/data/{userId}/generators/{generatorId}/skills/SKILL.md
    ↓
[Step 4] 组装完整 Prompt Context
    → 种子/果实内容 + 营养库内容 + 突变策略指令
    ↓
[Step 5] 执行生成（使用生成器 Skill）
    → Agent 调用 LLM，遵循生成器 Skill 定义，产出多个内容草稿
    ↓
[Step 6] 通过 MCP 保存生成结果并记录日志
    → MCP Tool: write_generation_log(log)（果实保存由果实模块 MCP 负责）
    ↓
展示结果，等待用户 Pick Up
    ↓
展示结果，等待用户 Pick Up
```

### 4.2 生成器 Agent 的编排 Skill

MVP 阶段需要实现一个 `**generation-orchestrator**` Skill，它告诉 AI IDE Agent 如何完成整个生成流程。该 Skill 存放在项目的 `.cursor/skills/` 或 `skills/` 目录中。

**Skill 核心内容**：

```markdown
# generation-orchestrator

## 职责
编排完整的果实生成流程。当用户发出生成指令时，按以下步骤执行。

## 输入要求
用户必须提供：
- 种子 ID 或果实 ID（生成基础）
- 生成器 ID 或生成器路径（使用哪个生成器）

用户可选提供：
- 营养库路径列表（挂载哪些营养库）
- 突变策略（style / remix / anti-logic，不指定则由 Agent 判断）
- 生成数量（默认 4 个）

## 执行步骤
1. 调用 MCP get_seed 或 get_fruit 获取输入内容
2. 如有营养库路径，调用 MCP get_nutrients 获取营养内容
3. 读取生成器 Skill 文件（SKILL.md）
4. 判断突变策略（用户指定优先，否则根据历史数据自主判断）
5. 组装 Prompt，调用 LLM 生成 N 个差异化果实
6. 每个果实必须包含 preview（title + summary + contentType）和 payload
7. 调用 MCP save_fruits 保存所有果实
8. 调用 MCP write_generation_log 记录本次生成日志
9. 向用户展示生成结果，提示进行 Pick Up

## 可用 MCP Tools
- get_seed(seedId)
- get_fruit(fruitId)
- get_nutrients(paths[])
- save_fruits(fruits[])
- write_generation_log(log)
- pick_fruit(fruitId)
```

### 4.3 生成输入汇总


| 输入项     | 是否必选 | 提供方               | 说明           |
| ------- | ---- | ----------------- | ------------ |
| 种子 / 果实 | 必选   | 用户                | 生成的基础内容来源    |
| 生成器     | 必选   | 用户                | 决定生成内容的类型和风格 |
| 营养库     | 可选   | 用户选择 / Agent 按需加载 | 提供领域知识和历史经验  |
| 突变策略    | 可选   | 用户指定 / Agent 自主判断 | 控制生成的探索程度    |
| 生成数量    | 可选   | 用户（默认 4 个）        | 一次生成多少个内容草稿  |

## 7. 数据存储设计

### 7.1 文件系统结构（新增部分）

```
/cf/data/{userId}/
├── seeds/                          ← 种子库（已有）
├── nutrients/                      ← 营养库（已有）
├── generators/                     ← 生成器库（新增）
│   └── {generatorId}/
│       └── {generatorName}/        ← 生成核心 Skill 文件夹（用户维护）
│           ├── SKILL.md            ← 主 Skill 文件
│           └── references/         ← 参考资料（可选）
└── logs/
    └── generation/                 ← 生成日志（新增）
        └── {generation_id}.json
```

**说明**：生成器元数据（`GeneratorMetadata`）由**平台存储**，不在本地文件系统中。用户目录下的 `generators/{generatorId}/` 只存放 Skill 文件夹。MVP 阶段平台元数据存储在 Redis 中，SaaS 阶段迁移至云端数据库。果实库由**果实模块**负责定义和实现，不在本模块范围内。

### 7.3 Redis Key 设计（新增部分）

| Key 模式 | 类型 | 说明 |
| :--- | :--- | :--- |
| `cf:gen:{genId}:meta` | Hash | 生成器元数据（市场侧，平台维护） |
| `cf:market:gens` | ZSet | 市场生成器列表（score = updatedAt） |
| `cf:u:{userId}:gens` | ZSet | 用户已安装生成器列表（score = installedAt） |
| `cf:u:{userId}:gen:{genId}:install` | Hash | 用户安装记录（installedAt 等） |
| `cf:u:{userId}:genlog:{logId}` | String (JSON) | 生成日志 |

## 5. 核心功能逻辑

### 5.1 市场生成器列表 (Market Generator List)

**场景**：用户访问生成器市场，浏览平台上所有可用的生成器。支持分页，若携带 `userId` 则返回数据中标注当前用户是否已安装。

**API 定义**:
- **URL**: `GET /api/generators/market`
- **Headers**: `X-User-Id: {userId}`（可选，不传则不返回安装状态）
- **Query**: `page=1&size=20&platform=xiaohongshu&domain=lifestyle`
- **Response**:
  
```json
  {
    "code": 0,
    "data": {
      "list": [
        {
          "id": "gen_xhs_lifestyle_v1",
          "name": "小红书生活方式图文生成器",
          "description": "专为小红书打造...",
          "platform": "xiaohongshu",
          "contentTypes": ["image_text", "text_only"],
          "author": { "id": "user_001", "name": "森林园丁" },
          "installCount": 128,
          "price": 0,
          "isInstalled": true
        }
      ],
      "total": 56,
      "page": 1,
      "size": 20
    }
  }
```

  
*注：`isInstalled` 字段仅在请求携带 `X-User-Id` 时出现，否则该字段不返回。*

**处理流程**：

1. **分页查询**：从 Redis ZSet `cf:market:gens` 中按 score（updatedAt）分页取出生成器 ID 列表。
2. **批量获取元数据**：Pipeline 批量读取 `cf:gen:{genId}:meta`。
3. **平台/领域筛选**：如有 `platform` / `domain` 参数，在结果集上过滤。
4. **安装状态标注**（仅当携带 `userId`）：批量检查 `cf:u:{userId}:gens` ZSet 中是否包含对应 `genId`，追加 `isInstalled` 字段。
5. **组装响应**：返回分页数据。

---

### 5.2 我的生成器列表 (My Generator List)

**场景**：用户查看自己已安装的所有生成器，支持分页。

**API 定义**:

- **URL**: `GET /api/generators/mine`
- **Headers**: `X-User-Id: {userId}`（必须）
- **Query**: `page=1&size=20`
- **Response**:
```json
  {
    "code": 0,
    "data": {
      "list": [
        {
          "id": "gen_xhs_lifestyle_v1",
          "name": "小红书生活方式图文生成器",
          "platform": "xiaohongshu",
          "contentTypes": ["image_text"],
          "author": { "id": "user_001", "name": "森林园丁" },
          "installCount": 128,
          "price": 0,
          "installedAt": 1741680000000,
          "skillPath": "/cf/data/local_admin/generators/gen_xhs_lifestyle_v1/skills/"
        }
      ],
      "total": 3,
      "page": 1,
      "size": 20
    }
  }
```

**处理流程**：

1. **校验 userId**：`X-User-Id` 必须存在，否则返回 401。
2. **分页查询**：从 Redis ZSet `cf:u:{userId}:gens`（score = installedAt）中分页取出 `genId` 列表。
3. **批量获取元数据**：Pipeline 批量读取 `cf:gen:{genId}:meta`，合并 `installedAt` 和本地 `skillPath`。
4. **组装响应**：返回分页数据。

### 5.3 安装生成器 (Install Generator)

**场景**：用户在生成器市场选中一个生成器，点击「安装」，将其添加到「我的生成器」并下载 Skill 文件到本地目录。

**API 定义**:

- **URL**: `POST /api/generators/{generatorId}/install`
- **Headers**: `X-User-Id: {userId}`（必须）
- **Body**: 无
- **Response**:
  ```json
  {
    "code": 0,
    "data": {
      "generatorId": "gen_xhs_lifestyle_v1",
      "skillPath": "/cf/data/local_admin/generators/gen_xhs_lifestyle_v1/skills/",
      "installedAt": 1741680000000
    }
  }
  ```

**处理流程**：

1. **校验 userId**：`X-User-Id` 必须存在，否则返回 401。
2. **重复安装检查**：检查 `cf:u:{userId}:gens` ZSet 中是否已存在该 `generatorId`，若已存在返回 409（已安装）。
3. **获取生成器元数据**：读取 `cf:gen:{generatorId}:meta`，确认生成器存在。
4. **下载 Skill 文件**：
  - 从平台存储（云端/本地市场目录）拉取 Skill 文件夹。
  - 写入本地路径 `/cf/data/{userId}/generators/{generatorId}/skills/`。
5. **更新存储层**：
  - 将 `generatorId` 写入 Redis ZSet `cf:u:{userId}:gens`（score = 当前时间戳）。
  - 记录安装时间 `installedAt` 到用户安装记录 Hash `cf:u:{userId}:gen:{generatorId}:install`。
6. **更新安装计数**：将市场元数据中的 `installCount` 原子性 +1（`HINCRBY cf:gen:{generatorId}:meta installCount 1`）。
7. **返回结果**：返回本地 Skill 路径和安装时间。

### 5.4 上传生成器 (Upload Generator)

**场景**：用户将自己创作的 Skill 上传到生成器市场，填写元数据后发布供他人使用。

**API 定义**:

- **URL**: `POST /api/generators/upload`
- **Headers**: `X-User-Id: {userId}`（必须）
- **Body**: `multipart/form-data`
  ```
  metadata  (JSON string):
  {
    "name": "小红书生活方式图文生成器",
    "description": "专为小红书打造...",
    "platform": "xiaohongshu",
    "contentTypes": ["image_text", "text_only"],
    "domain": "lifestyle",
    "outputCapabilities": ["title", "body", "tags", "cover_prompt"],
    "price": 0
  }

  skillFile  (zip/tar.gz):
    打包后的 Skill 文件夹（必须包含 SKILL.md）
  ```
- **Response**:
  ```json
  {
    "code": 0,
    "data": {
      "id": "gen_xhs_lifestyle_v1",
      "name": "小红书生活方式图文生成器",
      "status": "published"
    }
  }
  ```

**处理流程**：

1. **校验 userId**：`X-User-Id` 必须存在，否则返回 401。
2. **校验元数据**：`name`、`platform`、`outputCapabilities` 为必填项，缺失返回 400。
3. **校验 Skill 文件**：
  - 解压上传文件，检查根目录是否包含 `SKILL.md`，不存在返回 400（「Skill 文件夹缺少 SKILL.md」）。
  - MVP 阶段跳过安全扫描；最终版本此处接入 Prompt Injection 关键词扫描。
4. **生成 generatorId**：规则为 `gen_{nanoid}`。
5. **存储 Skill 文件**：将解压后的 Skill 文件夹保存至平台存储目录（MVP 阶段为本地服务器目录，SaaS 阶段为云端对象存储）。
6. **写入元数据**：
  - 组装 `GeneratorMetadata`，写入 Redis Hash `cf:gen:{generatorId}:meta`。
  - 将 `generatorId` 写入市场索引 ZSet `cf:market:gens`（score = createdAt）。
7. **同步到「我的生成器」**：上传者自动安装自己的生成器，执行 5.3 安装流程（跳过文件下载步骤，直接引用服务器目录）。
8. **返回结果**：返回生成的 `generatorId` 和发布状态。

---

### 5.5 卸载生成器 (Uninstall Generator)

**场景**：用户从「我的生成器」中移除一个生成器，删除本地 Skill 文件和存储层记录。

**API 定义**:

- **URL**: `DELETE /api/generators/{generatorId}/uninstall`
- **Headers**: `X-User-Id: {userId}`（必须）
- **Response**:
  ```json
  { "code": 0, "data": { "generatorId": "gen_xhs_lifestyle_v1", "uninstalledAt": 1741680000000 } }
  ```

**处理流程**：

1. **校验 userId**：`X-User-Id` 必须存在，否则返回 401。
2. **安装记录检查**：检查 `cf:u:{userId}:gens` 中是否存在该 `generatorId`，不存在返回 404（「未安装该生成器」）。
3. **删除本地 Skill 文件**：
  - 递归删除 `/cf/data/{userId}/generators/{generatorId}/` 目录。
  - 若目录不存在（文件系统与存储层不一致），记录 warning 日志，继续执行。
4. **清理存储层**：
  - 从 Redis ZSet `cf:u:{userId}:gens` 中移除 `generatorId`。
  - 删除 Redis Hash `cf:u:{userId}:gen:{generatorId}:install`。
5. **注意**：卸载**不修改**市场元数据（`cf:gen:{generatorId}:meta`）和 `installCount`，installCount 仅在安装时增加，不因卸载减少（保持历史安装数据真实性）。
6. **返回结果**：返回卸载时间。

| ------- | -------- | ----------------------------------------- | ---- | --------------------------- |
