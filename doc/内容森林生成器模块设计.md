# 内容森林 - 生成器模块设计 (Generator Module)

> 版本：v1.0 (MVP)
> 日期：2026-03-16
> 状态：设计中

---

## 1. 核心语义与定义

### 1.1 什么是生成器 (Generator)

**生成器**是内容森林中负责将种子（或果实）转化为可发布内容的核心执行组件。它由两个部分组成：

- **外部元数据 (Metadata)**：用于平台展示与管理，包含生成器名称、作者、适用平台、描述、内容格式、版本号、创建时间、更新时间等。
- **生成核心 (Skill)**：一个标准的 Skill 文件夹，定义了该生成器如何生成内容、能生成哪些内容（图片、文字、视频脚本等）以及生成优质内容的方法论。

**关键设计原则**：

> 生成器**只关注如何生成内容数据**。它不感知、不依赖营养库，不调用 MCP，不关心数据如何落库。它的核心职责是：接收输入文本，输出结构化内容。

这意味着生成器的 Skill 部分具备完全的**可移植性**——单独取出放到其他 Agent 或其他项目中，同样可以运行。

---

### 1.2 生成器的组成结构

生成器由**两个独立关注点**组成，分属不同的维护主体：

```
┌─────────────────────────────────────────────────────────┐
│                      生成器                              │
├──────────────────────────┬──────────────────────────────┤
│   外部元数据              │   生成核心（Skill）           │
│   Generator Metadata     │   Generator Skill            │
│                          │                              │
│   由平台存储和管理         │   由用户创建和维护            │
│   与 Skill 内容解耦       │   标准 Skill 结构，无平台约束 │
│                          │                              │
│   关注「这个生成器是什么」  │   关注「怎么生成好内容」      │
└──────────────────────────┴──────────────────────────────┘
```

**关键设计原则**：用户创建一个内容生成 Skill 时，**只需关注 Skill 本身的生成规范**，无需了解平台需要什么字段，无需维护任何平台元数据文件。两者通过平台注册时建立关联。

---

#### 部分 A：外部元数据（平台维度）

元数据由**平台存储**，用户在平台注册/上传生成器时填写。Skill 内容更新，元数据不一定要变。

**元数据结构（平台侧数据模型）**：

```typescript
interface GeneratorMetadata {
  // 平台标识（平台生成）
  id: string;                    // 如 "gen_xhs_lifestyle_v1"
  createdAt: number;
  updatedAt: number;

  // 用户填写（注册时提交）
  name: string;                  // "小红书生活方式图文生成器"
  description: string;
  thumbnail?: string;
  platform: string;              // "xiaohongshu" | "douyin" | "twitter" | ...
  contentTypes: string[];        // ["image_text", "text_only"]
  domain?: string;               // "lifestyle" | "tech" | "fitness" | ...

  // 作者信息
  author: {
    id: string;
    name: string;
  };

  // 输出能力声明（告知 Agent 该生成器能产出哪些字段）
  outputCapabilities: string[];  // ["title", "body", "tags", "cover_prompt"]

  // 市场数据（平台维护，预留）
  price: number;                 // 0 = 免费，预留付费扩展
  installCount: number;          // 安装次数，用户安装时自动累加
  rating: number | null;         // 评分，暂不实现
}
```

---

#### 部分 B：生成核心 Skill（用户维度）

Skill 就是一个**标准的 Skill 文件夹**，没有任何平台特有的约束。用户只需按照 Skill 规范编写，不需要额外的 `generator.yaml` 或其他平台配置文件。

Skill 的文件结构由用户自由决定，下面是一个推荐的参考结构（非强制）：

```
xiaohongshu-lifestyle-post/    ← Skill 目录，命名自由
├── SKILL.md                   ← 主 Skill 文件（必须），描述整体生成逻辑
└── references/                ← 参考资料（可选）
    ├── title-patterns.md      ← 标题写法参考
    ├── body-patterns.md       ← 正文结构参考
    └── tag-strategy.md        ← 标签策略参考
```

**SKILL.md 的核心内容**（用户自由发挥，以下为建议包含的要素）：
- 该生成器擅长什么、适合什么场景
- 如何接收输入（种子/果实内容的哪些部分）
- 生成几个果实、每个果实的差异化策略
- 必须输出的 `preview` 字段（title + summary + contentType）
- 自定义 `payload` 的结构说明
- 优质内容的方法论（平台规律、爆款模式等）

**与平台的关联**：用户将 Skill 文件夹上传到生成器市场时，平台为其创建一条 `GeneratorMetadata` 记录，通过 `generatorId` 与 Skill 建立关联。两者独立演进，互不耦合。

---

### 1.3 什么是生成器 Agent (Generator Agent)

**生成器 Agent** 是调度和编排整个生成过程的决策层。它负责：

- 接收用户输入（种子/果实 + 生成器选择 + 营养库挂载 + 突变策略）
- 通过 MCP 获取所需数据（种子内容、营养库内容）
- 加载并执行生成器的 Skills
- 通过 MCP 保存生成结果并记录日志

生成器 Agent 本身不是生成器，它是**使用生成器的协调者**。

**MVP 阶段与最终阶段的唯一区别**：

| 阶段 | Agent 来源 | 交互入口 |
| :--- | :--- | :--- |
| **MVP** | AI IDE（Cursor/Trae）内置 Agent | 用户在 IDE 聊天框输入指令 |
| **最终版** | 平台自研 Agent | 用户在平台 Web 页面操作 |

流程编排逻辑完全一致，唯一变量是 Agent 的宿主环境。

## 2. 生成器与用户的关系

### 2.1 生成器的来源

```
生成器来源
├── 平台生成器（Platform Generator）  ← 平台官方提供，存储在云端，由平台维护
└── 用户生成器（User Generator）       ← 用户自己创建或从平台安装
    └── 我的生成器（My Generators）    ← 安装到本地后成为用户私有副本
```

### 2.2 生命周期流程

```
1. [发布]  用户/平台 → 创建生成器 → 上传至生成器市场（MVP 阶段支持导入）
2. [安装]  其他用户 → 从市场安装 → 复制到本地用户目录 → 变为【我的生成器】
3. [使用]  用户在生成时选择【我的生成器】或【平台生成器】
             └─ 选择平台生成器时自动安装并转为【我的生成器】
4. [迭代]  用户在营养汲取阶段，基于果实表现，使用 skill-creator 工具优化【我的生成器】
```

### 2.3 生成器存储路径

```
# 用户私有生成器（我的生成器）——本地文件系统
/cf/data/{userId}/generators/{generatorId}/
    ├── generator.yaml
    └── skills/
        ├── SKILL.md
        └── ...

# 平台生成器——云端维护，不在本地文件系统
# 通过 Generator Market API 拉取，安装时复制到用户目录
```

### 2.4 生成器迭代（Skill 优化）

生成器迭代的本质是对 Skill 文件的优化。在**营养汲取**阶段，当用户分析了高/低转化果实后，可以：

- 手动编辑生成器的 `skills/*.md` 文件，调整 Prompt
- 使用 AI IDE 中的 skill-creator 工具，让 Agent 辅助优化 Skill

每次迭代更新 `generator.yaml` 中的 `version` 和 `updated_at`。

## 3. 果实数据结构设计

### 3.1 设计挑战

果实的内容由生成器决定，不同平台、不同生成器产出的内容结构完全不同：

- **平台多样性**：小红书图文 vs 抖音脚本 vs 推特长文，结构差异极大
- **生成器多样性**：有的能产出封面图提示词，有的只能产出纯文字
- **平台变动性**：平台可能新增内容属性，结构需要持续演进

### 3.2 解决方案：容器格式（Container Pattern）

采用**固定容器 + 自由载荷**的设计：

- **系统约定容器格式**：用于索引、展示、状态管理，结构固定不变
- **生成器自定义载荷**：`payload` 字段完全由生成器决定，系统不解析、不约束

```typescript
// 果实容器格式（系统级约定，不随平台变化）
interface Fruit {
  // 系统标识
  id: string;
  userId: string;
  seedId: string;               // 关联的种子 ID（若基于果实迭代则为父果实的 seedId）
  parentFruitId?: string;       // 父果实 ID（迭代生成时存在）
  generatorId: string;          // 产出此果实的生成器 ID
  generatorVersion: string;     // 生成时使用的生成器版本
  createdAt: number;

  // 生命周期状态
  status: 'generated' | 'picked' | 'published' | 'rejected';

  // 突变信息
  mutation: {
    applied: boolean;
    type?: 'style' | 'remix' | 'anti-logic';
    styleLabel?: string;        // 风格标签：'conservative' | 'balanced' | 'radical' | 'random'
  };

  // 最小展示契约（生成器必须填写，用于果实池卡片展示）
  preview: {
    title: string;              // 展示标题
    summary: string;            // 一句话摘要（不超过 50 字）
    contentType: string;        // 内容类型：'text' | 'image_text' | 'video_script' | ...
  };

  // 生成器自由载荷（完全由生成器定义，系统不解析）
  payload: Record<string, any>;

  // 成长数据（发布后由监控器写入）
  metrics?: FruitMetrics;
}
```

### 3.3 payload 示例

不同生成器产出不同的 payload，系统均能存储，无需改动：

```json
// 小红书图文生成器的 payload
{
  "title": "用了这 3 个 AI 工具，我每天多出 2 小时",
  "body": "## 前言\n最近发现了几个神器...",
  "tags": ["AI工具", "效率提升", "职场干货"],
  "cover_prompt": "flat design, warm colors, woman using laptop, cozy office"
}

// 推特长推文生成器的 payload
{
  "thread": [
    "1/ 用了半年 AI 工具，我总结了 3 个真正有用的：",
    "2/ 第一个：...",
    "3/ 第二个：..."
  ],
  "hashtags": ["#AI", "#Productivity"]
}

// 抖音视频脚本生成器的 payload
{
  "hook": "你还在手动写文案？",
  "script": [
    { "timecode": "0-3s", "action": "特写镜头：电脑屏幕", "narration": "你还在..." },
    { "timecode": "3-10s", "action": "切换镜头", "narration": "其实只需要..." }
  ],
  "bgm_style": "轻快科技感"
}
```

### 3.4 preview 契约约束

生成器在产出 payload 的同时，**必须**填写 `preview` 字段。这是生成器与系统之间的最小契约：

```json
{
  "preview": {
    "title": "用了这 3 个 AI 工具，我每天多出 2 小时",
    "summary": "介绍 3 个提升效率的 AI 工具，适合职场人群",
    "contentType": "image_text"
  }
}
```

`preview` 的作用：
- 果实池列表展示（不需要解析 payload）
- 生成日志快速浏览
- 迭代树节点标签

## 4. 生成器 Agent 工作流程

### 4.1 完整生成流程

```
用户在 AI IDE 输入生成指令
    ↓
生成器 Agent（AI IDE）
    ↓
[Step 1] 通过 MCP 获取种子/果实内容
    → MCP Tool: get_seed(seedId) 或 get_fruit(fruitId)
    ↓
[Step 2] 通过 MCP 获取挂载的营养库内容（可选）
    → MCP Tool: get_nutrients(nutrientPaths[])
    ↓
[Step 3] 加载生成器 Skill
    → 读取 /cf/data/{userId}/generators/{generatorId}/skills/SKILL.md
    ↓
[Step 4] 组装完整 Prompt Context
    → 种子/果实内容 + 营养库内容 + 突变策略指令
    ↓
[Step 5] 执行生成（使用生成器 Skill）
    → Agent 调用 LLM，遵循生成器 Skill 定义，产出多个果实草稿
    → 每个果实包含 preview + payload
    ↓
[Step 6] 通过 MCP 保存果实并记录日志
    → MCP Tool: save_fruits(fruits[])
    → MCP Tool: write_generation_log(log)
    ↓
展示结果，等待用户 Pick Up
```

### 4.2 生成器 Agent 的编排 Skill

MVP 阶段需要实现一个 **`generation-orchestrator`** Skill，它告诉 AI IDE Agent 如何完成整个生成流程。该 Skill 存放在项目的 `.cursor/skills/` 或 `skills/` 目录中。

**Skill 核心内容**：

```markdown
# generation-orchestrator

## 职责
编排完整的果实生成流程。当用户发出生成指令时，按以下步骤执行。

## 输入要求
用户必须提供：
- 种子 ID 或果实 ID（生成基础）
- 生成器 ID 或生成器路径（使用哪个生成器）

用户可选提供：
- 营养库路径列表（挂载哪些营养库）
- 突变策略（style / remix / anti-logic，不指定则由 Agent 判断）
- 生成数量（默认 4 个）

## 执行步骤
1. 调用 MCP get_seed 或 get_fruit 获取输入内容
2. 如有营养库路径，调用 MCP get_nutrients 获取营养内容
3. 读取生成器 Skill 文件（SKILL.md）
4. 判断突变策略（用户指定优先，否则根据历史数据自主判断）
5. 组装 Prompt，调用 LLM 生成 N 个差异化果实
6. 每个果实必须包含 preview（title + summary + contentType）和 payload
7. 调用 MCP save_fruits 保存所有果实
8. 调用 MCP write_generation_log 记录本次生成日志
9. 向用户展示生成结果，提示进行 Pick Up

## 可用 MCP Tools
- get_seed(seedId)
- get_fruit(fruitId)
- get_nutrients(paths[])
- save_fruits(fruits[])
- write_generation_log(log)
- pick_fruit(fruitId)
```

### 4.3 生成输入汇总

| 输入项 | 是否必选 | 提供方 | 说明 |
| :--- | :--- | :--- | :--- |
| 种子 / 果实 | 必选 | 用户 | 生成的基础内容来源 |
| 生成器 | 必选 | 用户 | 决定生成内容的类型和风格 |
| 营养库 | 可选 | 用户选择 / Agent 按需加载 | 提供领域知识和历史经验 |
| 突变策略 | 可选 | 用户指定 / Agent 自主判断 | 控制生成的探索程度 |
| 生成数量 | 可选 | 用户（默认 4 个） | 一次生成多少个果实草稿 |

## 5. 生成器市场 (Generator Market)

### 5.1 设计理念

生成器市场的核心价值是**内容生产经验的共享与沉淀**。优质的生成器（高爆款率的 Skill）是真实的资产，应当能够被分享、传播、付费获取。

MVP 阶段实现：**导入 / 分享功能（免费）**
最终阶段扩展：**付费购买、评分系统、版本更新订阅**

**页面位置**：控制台侧边栏 → 生成器市场

## 7. 数据存储设计

### 7.1 文件系统结构（新增部分）

```
/cf/data/{userId}/
├── seeds/                          ← 种子库（已有）
├── nutrients/                      ← 营养库（已有）
├── generators/                     ← 生成器库（新增）
│   └── {generatorId}/
│       └── skills/                 ← 生成核心 Skill 文件夹（用户维护）
│           ├── SKILL.md            ← 主 Skill 文件
│           └── references/         ← 参考资料（可选）
├── fruits/                         ← 果实库（新增）
│   └── {YYYY}/{MM}/
│       └── {fruit_id}.md           ← Markdown + YAML Frontmatter
└── logs/
    └── generation/                 ← 生成日志（新增）
        └── {generation_id}.json
```

**说明**：生成器元数据（`GeneratorMetadata`）由**平台存储**，不在本地文件系统中。用户目录下的 `generators/{generatorId}/` 只存放 Skill 文件夹。MVP 阶段平台元数据存储在 Redis 中（`cf:u:{userId}:gen:{genId}:meta`），SaaS 阶段迁移至云端数据库。

### 7.3 Redis Key 设计（新增部分）

| Key 模式 | 类型 | 说明 |
| :--- | :--- | :--- |
| `cf:u:{userId}:gen:{genId}:meta` | Hash | 生成器元数据 |
| `cf:u:{userId}:gens` | ZSet | 用户生成器列表（score = updatedAt） |
| `cf:u:{userId}:f:{fruitId}:meta` | Hash | 果实元数据 |
| `cf:u:{userId}:s:{seedId}:fruits` | ZSet | 种子关联的果实列表（score = createdAt） |
| `cf:u:{userId}:genlog:{genId}` | String (JSON) | 生成日志 |

## 8. 安全说明（最终版本备注）

*本节为最终 SaaS 版本的安全设计备忘，MVP 阶段无需实现。*

在最终版本中，Agent 由平台提供，但 Skill 由用户上传。潜在安全风险：用户可能上传包含 Prompt Injection 的恶意 Skill，试图让平台 Agent 执行危险操作。

**防御策略**：

1. **Skill 仅为纯文本 Prompt**：禁止 Skill 中包含可执行代码（Python/JS 等），Skill 文件只能是 Markdown/YAML 格式的 Prompt 模板，从根本上消除代码注入风险。
2. **平台 Agent 最小权限**：生成器 Agent 的 MCP Tools 仅限 `save_fruits`、`get_seed`、`get_nutrients` 等只读/写果实的工具，禁止文件系统访问和外网请求工具。
3. **Skill 上架审核**：平台在生成器上架前，对 Skill 文件进行关键词扫描，拦截明显的 Prompt Injection 攻击特征（如「忽略之前的所有指令」等）。

## 9. DDD 统一语言补充

本模块新增以下核心术语，补充至项目 DDD 词汇表：

| 术语 | 英文 | 定义 | 说明 |
| :--- | :--- | :--- | :--- |
| **生成器** | Generator | 将种子/果实转化为果实的执行组件 | 由外部元数据 + 生成核心（Skill）组成。只关注内容生成，不感知营养库和 MCP |
| **生成核心** | Generator Skill | 生成器内部的 Skill 文件夹 | 定义生成逻辑、生成方法和输出内容类型，具备可移植性 |
| **生成器 Agent** | Generator Agent | 编排整个生成流程的决策层 | MVP 阶段为 AI IDE Agent，最终阶段为平台 Agent，流程逻辑一致 |
| **我的生成器** | My Generators | 用户已安装的生成器私有副本 | 存储在用户目录下，可被用户迭代优化 |
| **生成器市场** | Generator Market | 生成器的分享与获取平台 | MVP 支持免费导入/分享，预留付费扩展能力 |
| **果实载荷** | Fruit Payload | 果实中由生成器自定义的内容部分 | 结构完全由生成器决定，系统只存储不解析 |
| **果实展示契约** | Fruit Preview Contract | 生成器必须填写的最小展示信息 | 包含 title、summary、contentType，供系统展示使用 |
