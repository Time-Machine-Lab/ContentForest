# 内容森林 - 果实模块设计 (Fruit Module Design)

> 状态：MVP 阶段设计
> 日期：2026-03-19
> 最后更新：2026-03-19（前端交互评审同步）

## 1. 核心设计理念 (Core Concepts)

果实（Fruit）是内容森林中，种子经过 AI Agent 生成后的具体内容产物。果实模块的设计核心围绕**"溯源性"**、**"进化树展现"**与**"平台解耦"**展开。

*   **树状物理法则**：果实是进化树上的节点（MVP 单父节点树结构，二期扩展 DAG 嫁接重组），不设隐藏/归档状态，保留完整的繁衍记录，被淘汰的果实以视觉"枯萎"展现。
*   **读时模式 (Schema-on-read)**：系统不硬编码平台数据指标，由数据监控器提供自描述的数据反馈载荷。
*   **画布即工作台**：抛弃传统的列表与侧边栏导航，将进化树的无限画布作为唯一核心工作台。
*   **Markdown 载荷**：果实内容以 Markdown 格式存储，生成器不受平台格式约束，前端统一渲染，修剪即编辑 Markdown。

## 2. 数据结构设计 (Data Structure)

### 2.1 核心数据结构

```typescript
interface Fruit {
  // --- 基础身份信息 ---
  id: string;
  userId: string;
  seedId: string; // 所属种子

  // --- 进化溯源 ---
  // MVP 阶段使用单父节点，多父节点（DAG嫁接重组）为二期能力，当前不实装。
  parentFruitId?: string;    // 父节点（MVP单父；二期扩展为 parentFruitIds: string[]）
  generation: number;        // 代数（种子产出为1，基于果实迭代为2+）

  // --- 生产上下文快照 ---
  // 只记录影响因子，不记录冗长 Prompt
  context: {
    generatorId: string;
    generatorVersion: string;
    nutrientsUsed?: string[];   // 当时使用的营养库版本/ID
    crossFruitIds?: string[];   // 杂交参考果实 ID 列表（不影响树结构，仅作上下文记录）
  };

  // --- 极简状态机 ---
  // 状态流转规则：
  //   generated → picked（选中）| rejected（堆肥淘汰）
  //   picked    → published（发布，终态）| generated（撤回选中）
  //   rejected  → generated（撤回堆肥决策）
  //   published → 终态，不可流转
  status: 'generated' | 'picked' | 'rejected' | 'published';
  createdAt: number;
  updatedAt: number;

  // --- 基因突变记录 ---
  mutation: {
    applied: boolean;
    type?: 'style' | 'remix' | 'anti-logic';
    description?: string;
  };

  // --- 最小展示契约 ---
  // 由 Agent 生成时单独填写，不从 payload Markdown 解析，保证卡片渲染稳定性
  preview: {
    title: string;
    summary: string;
    coverUrl?: string;
    contentType: 'text' | 'image_text' | 'video_script' | 'audio';
    platform: string;  // 目标平台标识
  };

  // --- Markdown 内容载荷 ---
  // 生成器输出的完整内容，以 Markdown 格式存储。
  // 系统不解析其结构，前端统一使用 Markdown 渲染组件展示。
  // 修剪（Prune）操作即直接编辑此字段。
  payload: string; // Markdown 格式
}
```

**Payload 设计决策说明：**
- 采用纯 Markdown 字符串，取代原容器化 `Record<string, any>` 方案。
- 生成器不受系统约束，输出任意平台格式的 Markdown 内容，系统只存储不解析。
- `preview.title` / `preview.summary` 由 Agent 在生成时单独填写，作为卡片展示的稳定数据来源，不依赖解析 payload。
- Agent 营养汲取时直接读取 Markdown 全文，LLM 自行理解语义，无需结构化字段辅助。

### 2.2 数据反馈的领域解耦 (Fruit Growth Data)

数据反馈（Metrics）本质上属于"监控器领域"，它是果实发布后产生的衍生数据。为了遵循单一职责原则，将数据反馈从果实实体中剥离，作为一个独立的实体附着于果实之上。

```typescript
// 独立的成长数据实体
interface FruitGrowthData {
  fruitId: string;         // 关联的果实 ID
  platform: string;        // 冗余平台信息，方便监控器独立工作
  collectedAt: number;     // 数据最后收集/填写的更新时间
  metrics: MetricsField[]; // 自描述的数据载荷
}

// 灵活的指标载荷
interface MetricsField {
  key: string;         // 字段唯一键名 (例如: 'coin', 'retention_rate')
  value: number;       // 可量化的数据值 (例如: 150, 0.45)
  label: string;       // 页面展示名称 (例如: '投币数', '完播率')
  description: string; // 字段描述，供Agent营养汲取时理解数据意义
}
```

### 2.3 平台指标字典 (Platform Metrics Dictionary)

为了解决用户手动填写的痛点，系统引入"平台指标字典"。

1. **字典预设**：系统预设各平台的常见指标（如小红书的浏览、点赞、收藏）。
2. **按需勾选**：用户手动录入时，先从字典中勾选关心的指标，系统自动带出 `label` 和 `description`，用户只需填写 `value` 数字即可。
3. **动态组装**：前端将用户填写的数字与字典的描述组装成完整的 `MetricsField[]` 提交给后端。这保证了底层数据的自描述性，对 Agent 汲取营养极其友好。

> **后端实现说明**：平台指标字典以静态 TypeScript 常量存储于 `src/domain/platform-metrics-dict.ts`，不写入 Redis。后端通过 `GET /api/metrics/dict?platform=xiaohongshu` 接口将字典下发给前端，前端负责组装完整的 `MetricsField[]` 后提交。

### 2.4 Redis 存储键设计 (Redis Key Design)

```
果实主体:    cf:u:{userId}:fruit:{fruitId}           → Hash（存储 Fruit 所有字段）
种子果实索引: cf:u:{userId}:seed:{seedId}:fruits      → ZSet（score=createdAt，用于按时间排序查询）
成长数据:    cf:u:{userId}:fruit:{fruitId}:growth     → Hash（存储 FruitGrowthData）
```

## 3. 核心功能与流转 (Core Functions)

1. **生成 (Generate)**：由 AI IDE 中的 Agent 驱动，基于种子/父果实生成新果实，结果通过 MCP 落库。前端通过**轮询** `GET /api/seeds/{seedId}/fruits` 感知新果实到来，画布增量渲染新节点。
2. **选品比对 (Taste/Compare)**：用户将果实加入对比槽，横向平铺对比各果实的 Markdown 内容。入口为：节点 hover 时出现"加入对比"手柄，或详情抽屉内"加入对比"按钮。
3. **修剪 (Pruning)**：直接编辑果实的 `payload` Markdown 字段。不新开迭代分支，保持 MVP 极简。
4. **进化 (Evolution)**：
   - **营养汲取 (Extract)**：针对 Picked（优质）果实，提取成功经验，沉淀到对应的营养库。
   - **堆肥 (Compost)**：针对 Rejected（淘汰）果实，提取失败教训，作为负向经验沉淀到营养库。

## 4. API 接口设计 (API Interfaces)

为了保持职责单一，API 严格分为 **MCP (供 AI Agent 调用)** 和 **HTTP (供 Web UI 调用)** 两套。

### 4.1 MCP Tools (Agent 操作层)

| Tool | 说明 |
| :--- | :--- |
| `save_fruits` | 批量写入新生成果实（含 `payload` Markdown 和 `preview` 展示契约），同步更新种子的 `fruitCount` |
| `get_fruit_context` | 获取果实完整上下文（含 payload Markdown），供 Agent 迭代或营养汲取 |
| `update_fruit_status` | 变更果实状态，校验状态机流转合法性 |
| `pick_fruit` | 快捷将果实标记为 picked（Agent 自动选品场景） |
| `compost_fruit` | 将果实标记为 rejected，预留负向经验写入营养库的扩展点 |

### 4.2 HTTP RESTful API (Web 前端操作层)

| Method | URL | 说明 |
| :--- | :--- | :--- |
| `GET` | `/api/seeds/:seedId/fruits` | 查询种子下所有果实（前端轮询入口） |
| `PATCH` | `/api/fruits/:fruitId/pickup` | 手动选中果实（→ picked） |
| `PATCH` | `/api/fruits/:fruitId/publish` | 发布果实（→ published） |
| `PATCH` | `/api/fruits/:fruitId/compost` | 堆肥淘汰果实（→ rejected） |
| `PUT` | `/api/fruits/:fruitId/payload` | 原位修剪果实 Markdown 内容 |
| `GET` | `/api/fruits/:fruitId/metrics` | 获取果实成长数据 |
| `PUT` | `/api/fruits/:fruitId/metrics` | 更新果实成长数据 |
| `GET` | `/api/metrics/dict` | 获取平台指标字典（query: platform） |

## 5. 前端页面设计：无限画布工作台 (The Infinite Canvas)

**核心理念：进化树不是左侧导航，进化树就是整个工作台。**

### 5.1 技术选型

| 模块 | 选型 | 理由 |
| :--- | :--- | :--- |
| **核心画布** | `@antv/g6` | 专为大规模图数据设计，Canvas/WebGL 渲染，支持 100+ 节点流畅运行；内置树形布局算法与流动连线效果；支持自定义节点（可嵌入 Vue 组件） |
| **动效引擎** | `@vueuse/motion` | 弹簧阻尼物理动效，用于节点弹现、输入框弹出等 spring 动画 |
| **Markdown 渲染** | `@vueuse/motion` + `markdown-it` / `vue-markdown-render` | 详情抽屉内 payload 内容的渲染 |
| **样式框架** | 深度定制 `TailwindCSS` | 禁用默认色板，完全自定义设计系统 |

### 5.2 美学与配色设定（有机生态风格）

有别于常规 SaaS 工具的科技冷感，内容森林采用**有机生命感**的视觉语言，贴合"森林/生态/进化"的产品气质。

*   **背景**：墨绿黑 `#0F1712`，叠加**极细菌丝网络纹**（SVG pattern，模拟真菌菌丝），营造有机生命感而非机械网格感。
*   **连线**：带轻微抖动的有机贝塞尔曲线，模拟藤蔓生长路径，而非生硬的直线。突变节点的连线呈现断续电流感虚线，暗示基因变异。
*   **节点材质**：毛玻璃（Glassmorphism）叠加微妙内阴影，使卡片在画布上产生浮起感。
*   **主色调**：琥珀/橙金系（`#E8A045`、`#C97B2F`）作为主交互高亮色，取代冷色荧光绿，与「森林/果实」的暖色调呼应。
*   **状态配色**：
    *   **Picked（选中）**：橙金色描边 + 轻微呼吸发光动效（暖色脉冲）
    *   **Published（已发布）**：深绿描边 `#3D7A4F`，稳定无动效，代表已成熟落地
    *   **Rejected（堆肥）**：色彩饱和度降至 20%，透明度 50%，呈现枯萎蒙尘感；默认**折叠为小灰点**（见 5.5）
    *   **Generated（默认）**：中性白/浅灰描边，无特殊高亮

### 5.3 入场动画 (Growth Animation)

*   **触发**：从种子库点击某颗种子卡片进入工作台。
*   **转场**：共享元素动画平滑过渡，背景渐暗进入"实验室"模式。
*   **生长**：种子节点在画布中央落定发光 → 连线如藤蔓向四周蔓延（SVG `stroke-dasharray` 动画）→ 连线到达终点时，果实节点伴随 spring 弹簧动效依次弹现。

### 5.5 核心交互详解

#### ① 节点操作入口（触发方式明确分离）

| 操作 | 触发方式 | 效果 |
| :--- | :--- | :--- |
| 查看内容 | **单击节点** | 右侧详情抽屉滑出，节点微移左侧 |
| 生成/迭代 | Hover 时点击 **⚡ 生成/迭代** 按钮 | 节点正下方弹出输入框（弹簧动效） |
| 修剪内容 | Hover 时点击 **✂ 修剪** 按钮 | 详情抽屉打开并进入编辑模式 |
| Pick 选中 | Hover 时点击 **✓ Pick** 按钮 | 节点状态更新，橙金色描边亮起 |
| 堆肥淘汰 | Hover 时点击 **🗑 堆肥** 按钮 | 节点枯萎并折叠为小灰点 |
| 加入对比 | Hover 时点击 **⚖ 加入对比** 按钮，或详情抽屉内按钮 | 果实加入底部对比槽 |

#### ② 弹簧输入框（生成/迭代入口）

点击节点 Hover 时的 **⚡ 生成/迭代** 按钮后，在该节点正下方以**弹簧弹出动效**（spring，带弹性缓冲，非线性 ease）弹出悬浮输入框，不挤占画布空间。

**输入框结构：**

```text
┌─────────────────────────────────────────────────────────┐
│  [@营养库]  [@生成器]  [@杂交果实]          ← 顶部选项栏  │
│  点击后弹出选择面板，选中后自动插入 @ 标签               │
├─────────────────────────────────────────────────────────┤
│  @果实[别再用ChatGPT..] @生成器[小红书图文] ← @ 标签     │
│  生成两个不同风格的版本                   ← 普通文本     │
│                                                         │
│                                  [⚡ 生成]  ← 右下角    │
└─────────────────────────────────────────────────────────┘
```

**@ 标签交互规则：**
- `@` 标签整体不可手动编辑，只能整体删除（防止用户误改 ID）
- 删除某个 `@标签` 时，顶部选项栏对应项同步取消选中
- 标签与普通文本在视觉上明确区分（标签有背景色/圆角/特殊字体色）
- 输入框 placeholder 根据触发节点代际动态变化：
  - 点击**种子节点**触发 → "从这颗种子生长出什么？"
  - 点击**果实节点**触发 → "基于这个果实，继续进化..."

**生成按钮行为（MVP 阶段）：**

点击 **⚡ 生成** 后，将输入框内容组装为以下格式的提示词，自动复制到剪贴板，并在输入框上方显示 Toast 提示：

```
基于: {parentFruitId 或 seedId}
使用生成器: {generatorId}, {generatorId}...
使用营养库: {nutrientId}...
杂交参考: {crossFruitId}...
{用户输入的文本内容}
```

> Toast 提示文案：「提示词已复制 ✓ — 去 Cursor / Trae 的对话框粘贴，Agent 会自动生成果实」

前端随后开始轮询 `GET /api/seeds/{seedId}/fruits`，每隔 3 秒检查一次，检测到新果实时增量渲染至画布，并以 spring 弹现动效出现。

#### ③ 枯萎节点折叠机制

`rejected` 状态的果实默认**折叠为画布上的一个小灰点**，保留进化树完整性的同时避免杂乱。

*   **常态**：显示为小灰圆点，带连线保留树结构
*   **Hover**：展开为半透明枯萎卡片，可预览内容
*   **右键菜单**：提供"展开全部枯萎节点" / "永久折叠"选项

#### ④ 右侧详情抽屉 (Detail Drawer)

单击任意节点后，节点微移至画布偏左，右侧以毛玻璃质感抽屉滑入。抽屉采用双 Tab 设计解耦内容与数据：

```text
[ × ] 关闭            [ ⚡ 生成/迭代 ]  ← 抽屉顶部也有迭代入口
──────────────────────────────────────
🍎 别再用ChatGPT了！试试这3个AI神器
[小红书] 状态: 🟡 Picked   代数: 第2代
──────────────────────────────────────
[ 📝 内容本体 ]   [ 📊 市场反馈 ]
──────────────────────────────────────

(内容本体 Tab)
  Markdown 渲染内容展示
  [✂ 编辑 Markdown]  [⚖ 加入对比]

(市场反馈 Tab)
  最后更新: 2026-03-19 14:00  来源: 🧑‍💻 手动录入
  👀 浏览量: 12,000
  👍 点赞量: 340
  ⭐ 收藏量: 120
  [ + 添加/更新指标 ]  ← 弹出平台指标字典
  ──────────────────────────────
  [ 🧬 触发营养汲取 ]
```

**修剪（Prune）交互：**
点击"✂ 编辑 Markdown"后，抽屉内容区切换为 Markdown 编辑器（代码区 + 实时预览双栏），编辑完成后点击"保存"调用 `PUT /api/fruits/:fruitId/payload` 接口，节点的 `preview.title` 同步更新。

#### ⑤ 对比选品槽 (Tasting Dock)

*   **入口**：节点 Hover 时的"⚖ 加入对比"按钮，或详情抽屉内的"⚖ 加入对比"按钮
*   **触发展开**：加入第一个果实后，底部滑出对比槽（与详情抽屉互斥，打开对比槽时右侧抽屉自动收起）
*   **对比视图**：横向宽屏区域，平铺展示各果实的 Markdown 渲染内容（最多 3 个）
*   **槽内操作**：支持在对比视图内直接 Pick / 堆肥，无需回到画布

#### ⑥ 焦点管理与 Minimap

*   **点击节点**：画布自动平移，使目标节点居中（平滑动画，非瞬移）
*   **Minimap**：右下角常驻全局导览图，显示整棵进化树的缩略结构，可点击 Minimap 快速跳转区域
*   **缩放**：顶部导航显示当前缩放比例，支持滚轮缩放与顶部缩放控件

#### ⑦ 连线动效 (Edges)

*   **普通继承关系**：带轻微有机抖动的暖色贝塞尔曲线，模拟藤蔓生长
*   **突变节点连线**：`mutation.applied === true` 时，连线呈现断续电流感虚线，暗示基因变异
*   **流动粒子**：连线上有缓慢流动的粒子效果，代表"内容能量"的传递

## 6. 后端架构设计 (Backend Architecture)

### 6.1 模块文件结构

严格对齐现有后端分层架构（Domain → Repository → Service → API / MCP）：

```
src/
├── domain/
│   ├── fruit.ts                         # Fruit + FruitGrowthData 领域模型 + 状态机
│   └── platform-metrics-dict.ts         # 平台指标字典静态常量
├── repositories/
│   ├── fruit-repository.ts              # FruitRepository 接口
│   ├── fruit-growth-repository.ts       # FruitGrowthRepository 接口
│   ├── redis-fruit-repository.ts        # Redis 实现
│   └── redis-fruit-growth-repository.ts
├── services/
│   └── fruit-service.ts                # 业务逻辑（含 fruitCount 同步）
├── api/
│   └── fruits.ts                       # HTTP RESTful 路由（8个接口）
└── mcp/
    └── fruit-tools.ts                  # MCP Tools（5个工具）
```

### 6.2 状态机流转规则（最终确认版）

```
generated  →  picked     （用户选中）
generated  →  rejected   （用户堆肥淘汰）
picked     →  published  （用户发布，终态）
picked     →  generated  （撤回选中）
rejected   →  generated  （撤回堆肥决策）
published  →  [终态，不可流转]
```

`published` 是 `picked` 的最终表达形式，代表内容已对外发布，不回退到 `picked`。

### 6.3 关键设计决策

| 决策点 | 结论 | 理由 |
| :--- | :--- | :--- |
| 父节点字段 | MVP 用 `parentFruitId?: string`（单父） | 简化存储与前端树渲染；多父 DAG 为二期能力 |
| 杂交果实 | `context.crossFruitIds` 记录，不影响树结构 | 杂交只是生成时的上下文参考，保持树结构简洁 |
| Payload 格式 | 纯 Markdown 字符串 | 生成器解耦，前端统一渲染，修剪简单直接 |
| Preview 字段 | Agent 单独填写，不解析 payload | 保证卡片渲染的稳定数据来源 |
| GrowthData 存储 | 独立 Redis Hash，与 Fruit 主体分离 | 遵循单一职责；监控器可独立工作 |
| 平台指标字典 | 静态 TS 常量，不写 Redis | 字典为配置数据，无需持久化 |
| `save_fruits` MCP | 支持批量写入 | Agent 单次生成 3-5 个变体，批量更高效 |
| `pick_fruit` MCP | 新增此工具 | Agent 完成生成后需能自动标记 picked |
| `fruitCount` 维护 | FruitService 内部原子维护 | 调用方不负责计数，避免遗漏 |
| 前端感知生成 | 轮询（3秒间隔） | MVP 阶段简单可靠；SSE 为二期优化 |
| 画布组件 | `@antv/g6` | 100+ 节点性能保障，内置布局算法 |
| 营养库（Nutrients） | 本期只预留字段，不实装 | 营养库为独立模块，单独出提案 |
| Markdown 文件备份 | 本期只做 Redis，不实装文件备份 | MVP 优先跑通核心流程，备份为二期 |
